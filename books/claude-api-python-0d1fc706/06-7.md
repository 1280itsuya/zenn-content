## 落とし穴と対策：実運用で踏んだ7つの失敗と修正コード

自動パイプラインは「動く」と「実用に耐える」の間に深い溝がある。本章では筆者が実際に踏んだ7つの障害を、再現コードと修正パッチとともに解説する。

---

## 失敗1：タイトルと本文のドリフト

### 何が起きたか

「節約術10選」というタイトルで生成を走らせると、本文がいつの間にか「AI副業の始め方」に化けていた。Claude APIはシステムプロンプトとユーザープロンプトを分けて渡すが、長い会話履歴が蓄積すると前のターンのテーマが滲み出る。

### 修正コード

```python
def verify_content_matches_title(title: str, body: str) -> bool:
    # タイトルの主キーワードが本文に含まれるか検証
    keywords = extract_keywords(title)  # 形態素解析等で抽出
    match_count = sum(1 for kw in keywords if kw in body)
    return match_count / len(keywords) >= 0.5

def generate_article(title: str) -> str:
    body = call_claude(title)
    if not verify_content_matches_title(title, body):
        raise ValueError(f"ドリフト検出: '{title}' と本文が乖離しています")
    return body
```

生成のたびに検証を挟み、不一致なら即座に中断して次のテーマへ進む。

## 失敗2：レート制限による途中停止

### 何が起きたか

Qiitaへの投稿APIが429（Too Many Requests）を返し、50件中途中の23件目でパイプラインが全停止した。リトライ処理がなく、どこまで投稿したかも分からなくなった。

### 修正コード

```python
import time
import json
from pathlib import Path

CHECKPOINT_FILE = "data/post_checkpoint.json"

def load_checkpoint() -> set:
    if Path(CHECKPOINT_FILE).exists():
        return set(json.loads(Path(CHECKPOINT_FILE).read_text()))
    return set()

def save_checkpoint(posted_ids: set):
    Path(CHECKPOINT_FILE).write_text(json.dumps(list(posted_ids)))

def post_with_retry(article: dict, posted_ids: set, max_retries=3):
    if article["id"] in posted_ids:
        return  # 再実行時のスキップ
    for attempt in range(max_retries):
        try:
            post_to_qiita(article)
            posted_ids.add(article["id"])
            save_checkpoint(posted_ids)
            return
        except RateLimitError:
            wait = 2 ** attempt * 60  # 指数バックオフ
            print(f"429: {wait}秒待機...")
            time.sleep(wait)
    raise RuntimeError(f"投稿失敗: {article['id']}")
```

チェックポイントファイルで進捗を永続化し、再実行時は済んだ記事を自動スキップする。

## 失敗3：`load_dotenv()`の欠落による無音スキップ

### 何が起きたか

投稿スクリプトを別ディレクトリに移した後、APIキーが読まれなくなった。エラーは出ず、ログに「スキップ」とだけ出力されて100件全て投稿ゼロ。

### 修正コード

```python
# 各スクリプトの先頭に必ず記載する
from dotenv import load_dotenv
import os

load_dotenv()  # .env を明示ロード

QIITA_TOKEN = os.getenv("QIITA_TOKEN")
if not QIITA_TOKEN:
    raise EnvironmentError("QIITA_TOKEN が未設定です。.env を確認してください。")
```

環境変数が取れなければ即 `raise` する。無音スキップは最も発見しにくいバグの一形態だ。

## 失敗4：重複記事の大量生成

### 何が起きたか

タスクスケジューラが前回の実行を検知できず、同一テーマの記事が3日分・計36件生成された。生成コストも無駄だが、プラットフォームからスパム判定を受けるリスクが大きい。

### 修正コード

```python
import hashlib
import sqlite3

DB_PATH = "data/generated.db"

def get_db():
    con = sqlite3.connect(DB_PATH)
    con.execute("CREATE TABLE IF NOT EXISTS articles (hash TEXT PRIMARY KEY)")
    return con

def is_duplicate(title: str, date: str) -> bool:
    key = hashlib.md5(f"{title}:{date}".encode()).hexdigest()
    con = get_db()
    row = con.execute("SELECT 1 FROM articles WHERE hash=?", (key,)).fetchone()
    if row:
        return True
    con.execute("INSERT INTO articles VALUES (?)", (key,))
    con.commit()
    return False
```

タイトル＋日付のハッシュをSQLiteに記録し、二重生成を防ぐ。

## 失敗5：Gitプッシュの対話型ハングアップ

### 何が起きたか

Zennはリポジトリ連携で自動公開するため `git push` を自動実行していたが、GitHub認証ダイアログがTTYを待ち続け約2時間ハングした。

### 修正コード

```python
import subprocess
import os

def push_to_github(repo_path: str):
    env = os.environ.copy()
    env["GIT_TERMINAL_PROMPT"] = "0"  # 対話入力を無効化
    # PATをURLに埋め込んでHTTPS認証
    pat = os.getenv("GITHUB_PAT")
    remote = f"https://{pat}@github.com/yourname/zenn-content.git"
    subprocess.run(
        ["git", "push", remote, "main"],
        cwd=repo_path, env=env, check=True, timeout=60
    )
```

`GIT_TERMINAL_PROMPT=0` でダイアログを抑止し、PATをURLに直接埋め込む。PATは `.env` に保管し、リモートリポジトリにはコミットしないこと。

## 失敗6：オーケストレーターの無限ハングアップ

### 何が起きたか

`claude -p` でサブプロセスを呼び出す設計で、パイプ出力の読み取り待ちに入ったまま6時間以上CPU使用率ほぼ0で停止した。投稿もレポートも届かない。

### 修正コード

```python
import subprocess

result = subprocess.run(
    ["claude", "-p", prompt],
    capture_output=True, text=True,
    timeout=300,  # 5分でタイムアウト
    check=True
)
output = result.stdout
```

`timeout` を必ず設定する。タイムアウト後は `subprocess.TimeoutExpired` が上がるのでキャッチしてリトライかスキップを選択する。

## 失敗7：テンプレートタイトルのスパム化

### 何が起きたか

「【2024年最新】〇〇完全ガイド」という雛形タイトルを使い回した結果、Qiitaに掲載した記事20件のタイトルが全て同じパターンになった。タグも「未分類」ばかりで検索ゼロ。

### 修正コード

```python
import random

TITLE_TEMPLATES = [
    "{topic}でつまずいた3つのポイントと解決策",
    "{topic}を5分で理解する実装例",
    "{topic}：ハマりどころと回避パターン",
    "実務で使う{topic}チェックリスト",
]

VALID_TAGS = {
    "python": ["Python", "自動化", "スクリプト"],
    "claude": ["Claude", "LLM", "生成AI"],
    "api": ["API", "REST", "WebAPI"],
}

def build_title(topic: str) -> str:
    tmpl = random.choice(TITLE_TEMPLATES)
    return tmpl.format(topic=topic)

def pick_tags(keywords: list[str]) -> list[str]:
    tags = []
    for kw in keywords:
        tags.extend(VALID_TAGS.get(kw.lower(), []))
    return list(set(tags))[:5]  # Qiitaは5タグ上限
```

テンプレートを複数用意してランダム選択し、タグは公式カテゴリから引く。

## まとめ

| # | 失敗 | 主な修正 |
|---|------|----------|
| 1 | タイトル・本文ドリフト | 生成後にキーワード一致率を検証 |
| 2 | 429による途中停止 | チェックポイント＋指数バックオフ |
| 3 | `load_dotenv` 欠落 | 起動時に必須変数を `raise` 確認 |
| 4 | 重複記事の大量生成 | SQLiteでタイトル＋日付ハッシュ管理 |
| 5 | Git pushハングアップ | `GIT_TERMINAL_PROMPT=0`＋PAT埋め込み |
| 6 | サブプロセス無限待機 | `timeout=300` を全subprocess呼び出しに追加 |
| 7 | テンプレートタイトルスパム | 複数雛形ランダム＋正規タグリスト |

自動化の本当のコストはコード量ではなく、夜中に止まっているのに気づかない時間だ。上記パッチを初期実装に組み込むことで、無人運用の安定性が大幅に上がる。
