前章でClaude Codeとの最初の対話を体験しました。本章では、その対話の質を根本から変える2つのファイルを扱います。CLAUDE.mdと.claude/settings.jsonです。

私は記事の生成からQiita・Zenn・Dev.toへの投稿までを毎日自動で回すパイプラインをClaude Codeで運用しています。無人で動かす以上、「AIが迷わない環境」を先に作ることが生命線です。その中心にあるのが、この2つのファイルでした。

### CLAUDE.mdは「新人メンバーへの引き継ぎ書」

CLAUDE.mdは、Claude Codeが起動時に自動で読み込むプロジェクトの説明書です。私はこれを「明日から合流する新人メンバーへの引き継ぎ書」だと考えています。

新人に渡す引き継ぎ書に何を書くか、想像してみてください。プロジェクトの目的。ディレクトリの構成。守ってほしい規約。絶対にやってはいけないこと。よく使うコマンド。CLAUDE.mdに書くべき内容は、これと同じです。

逆に、新人が自分で調べれば分かることは書きません。コードを読めば分かる実装の詳細や、一般的な言語仕様の説明は不要です。Claudeはコードを読む力が高いので、「読んでも分からないこと」「読む前に知っておくべきこと」に絞ります。

### 実運用しているCLAUDE.mdの記述パターン

私の自動化プロジェクトで実際に使っている構成を、要素ごとに紹介します。

まずプロジェクト構成です。どこに何があるかを最初に教えます。

```markdown
# プロジェクト概要
記事を自動生成しQiita/Zenn/Dev.toへ投稿するパイプライン。

## 構成
- src/          生成スクリプト(チャンネル別)
- orchestrator/ 投稿・通知・スケジュール制御
- data/         生成物と実績ログ
- config/       エージェント設定(agents.json)
```

次にコーディング規約と頻出コマンドです。

```markdown
## 規約
- Python 3.11 / 型ヒント必須
- Windows環境。ファイルI/Oのencoding="utf-8"指定を省略しない
- 環境変数を使うスクリプトは先頭でload_dotenv()を呼ぶ

## よく使うコマンド
- テスト: python -m pytest tests/ -x
- 単発実行: python orchestrator/run.py --channel blog
```

そして最重要なのが禁止事項です。私は失敗のたびに、ここへ1行ずつ追加してきました。

```markdown
## 禁止事項
- 数値や実績を捏造しない。実測値のみ使う
- 記事タイトルと本文のテーマを一致させる。不一致なら公開を中止
- git pushは非対話モードで行う(GIT_TERMINAL_PROMPT=0)
```

この3行は、すべて実際の事故から生まれています。生成した記事にAIが根拠のない数値を書き込んだ事故。タイトルと本文の内容がずれた記事を量産しかけた事故。git pushが認証ダイアログ待ちで数時間ハングし、無人パイプラインが止まった事故。同じ失敗を二度させないための「傷跡の記録」がCLAUDE.mdなのです。

### settings.jsonのpermissions——自動許可の判断基準

.claude/settings.jsonでは、ツール実行の許可ルールを定義できます。私の設定の骨格はこうです。

```json
{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(python -m pytest:*)"
    ],
    "ask": [
      "Bash(git push:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Read(.env)"
    ]
  }
}
```

allowは確認なしで実行、askは毎回人間に確認、denyは実行自体を拒否します。私の判断基準は3つのルールに言語化できます。

第一に「読むだけの操作は自動許可」。git statusやテスト実行は、何度実行しても壊れるものがありません。確認ダイアログは作業のテンポを削るだけです。

第二に「外部に影響が出る操作は確認を残す」。git pushや投稿APIの実行は、失敗したときの回復コストが高いからです。人間の目を1回挟みます。ただし完全無人のパイプラインでは確認を挟めないので、公開前の検証をスクリプトやhooksで自動化した上で許可します(第5章・第6章で扱います)。

第三に「秘密情報と破壊操作はdenyで遮断」。.envにはAPIキーが入っています。AIに読ませる必要がないものは、読み取り自体を禁止するのが安全です。

### ビフォーアフター——同じ指示で出力はこう変わる

効果を体感してもらうため、実際の差を示します。指示は同じ「投稿スクリプトにリトライ処理を足して」です。

設定なしの場合。Claudeはまずプロジェクト全体の探索から始めます。どのファイルが投稿処理かを推測し、ときには既存と違うリトライ用ライブラリを新規導入し、encoding未指定のコードを書きます。Windows環境ではこれが文字化けの原因になります。動くまでに何往復もの修正指示が必要でした。

設定ありの場合。CLAUDE.mdで構成と規約を知っているので、最初からorchestrator配下の該当ファイルを開きます。既存コードのパターンに合わせた実装を提案し、規約通りのテストコマンドを実行します。permissionsでテストが自動許可されているため、確認なしで「実装→テスト→修正」まで一気に進みます。

体感として、往復回数が大きく減ります。設定に使う30分は、その日のうちに回収できる投資です。

### 書きすぎても読まれない——試行錯誤の教訓

ただし、CLAUDE.mdは「書けば書くほど良い」わけではありません。私は両方向の失敗をしました。

最初は薄すぎました。数行の概要だけ書いて放置した結果、Claudeは毎回プロジェクトを探索し直し、実行のたびに違う流儀のコードが生まれました。無人運用では、この「迷走」が翌朝の壊れた成果物として発見されます。

反動で、今度は書きすぎました。あらゆる注意事項を詰め込んだ長大なCLAUDE.mdでは、重要な禁止事項が雑多な記述に埋もれ、かえって守られなくなりました。人間の新人と同じです。100ページの引き継ぎ書は読み飛ばされます。

試行錯誤の末に落ち着いた運用ルールは次の通りです。

- 常に守るべきことだけをCLAUDE.mdに残す。目安は1〜2画面に収まる分量
- 特定の作業でしか使わない知識は別ファイルに切り出し、「この作業のときはこのファイルを読む」とパスだけ書く
- 事故が起きたら1行足す。長く参照されていない記述は消す

CLAUDE.mdは書いて終わりの文書ではなく、運用しながら剪定し続ける「生きた文書」です。

### まとめ

- CLAUDE.mdは新人への引き継ぎ書。構成・規約・頻出コマンド・禁止事項に絞る
- 禁止事項は実際の失敗から1行ずつ育てる
- permissionsの基準は「読み取りは自動、外部影響は確認、秘密と破壊は遮断」
- 薄すぎれば迷走し、厚すぎれば読まれない。剪定し続ける
