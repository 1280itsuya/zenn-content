前章までで、Claude Codeをスクリプトから呼び出す道具立てがそろいました。本章では、それを「毎朝、人が起きる前に勝手に動く仕組み」に仕上げます。使うのはWindows標準のタスクスケジューラだけです。私は記事の生成からQiita・Zenn・Dev.toへの投稿までを、この方法で毎朝の定時実行にしています。成果の話はしません。仕組みの再現性だけに集中します。

### 8.1 毎朝パイプラインの全体像

構成は次の通りです。

```
タスクスケジューラ（毎朝7:00）
  └─ run_pipeline.bat
      └─ python orchestrator.py
          ├─ [1] claude -p で記事を生成
          ├─ [2] 品質チェック（タイトルと本文の一致・文字数）
          ├─ [3] Qiita / Zenn / Dev.to へ投稿
          └─ [4] ログ保存 → 結果を通知
```

起点はバッチファイル1本です。タスクスケジューラからPythonを直接呼ばず、バッチを挟みます。環境変数の設定とログのリダイレクトをバッチ側に寄せられるからです。

```bat
@echo off
cd /d C:\work\auto-blog
set GIT_TERMINAL_PROMPT=0
python orchestrator.py >> logs\%date:~0,4%%date:~5,2%%date:~8,2%.log 2>&1
```

orchestrator.pyの骨格です。生成は第5章で扱ったclaude -pで行います。

```python
import subprocess, sys
from pathlib import Path

LOCK = Path("pipeline.lock")

def generate(theme: str) -> str:
    r = subprocess.run(
        ["claude", "-p", f"{theme}について技術記事を書いて"],
        capture_output=True, text=True, timeout=1800)
    return r.stdout

def main():
    if LOCK.exists():
        sys.exit("前回の実行が残っています")
    LOCK.touch()
    try:
        article = generate("...")
        if validate(article):          # 品質ゲート
            post_all(article)          # Qiita/Zenn/Dev.to
    finally:
        LOCK.unlink()
```

なお、WindowsのPATH上のclaudeは.cmd経由のことがあります。subprocessで見つからない場合は"claude.cmd"を指定してください。

### 8.2 タスクスケジューラへの登録手順

画面に沿って進めます。

1. Win+Rキーで「taskschd.msc」と入力して起動します。
2. 右側の操作ペインで「タスクの作成」を選びます。「基本タスクの作成」より細かく設定できます。
3. 「全般」タブ。名前を付け、「ユーザーがログオンしているかどうかにかかわらず実行する」を選びます。画面ロック中でも動きます。
4. 「トリガー」タブ。「新規」→「毎日」→開始時刻を7:00にします。
5. 「操作」タブ。「プログラム/スクリプト」にバッチのフルパス、「開始(オプション)」に作業フォルダを入れます。ここを空にすると相対パスが壊れます。
6. 「条件」タブ。ノートPCなら「コンピューターをAC電源で使用している場合のみタスクを開始する」のチェックを外します。「タスクを実行するためにスリープを解除する」にはチェックを入れます。
7. 「設定」タブ。「タスクが既に実行中の場合に適用される規則」を「新しいインスタンスを開始しない」にします。

スリープからの起動には、電源オプション側で「スリープ解除タイマーの許可」が有効である必要もあります。登録後は右クリック→「実行」で手動テストし、翌朝を待たずに動作を確かめてください。

### 8.3 無人運用の三つの罠

いずれも私が実際に踏んだものです。

**罠1: 対話プロンプト待ちのハング。** claude -pでも、許可していないツールの確認待ちで止まることがあります。第3章のsettings.jsonでpermissionsの許可リストを整え、実行時は--allowedToolsで範囲を明示します。それでもsubprocessのtimeoutは必ず付けます。パイプ待ちハングの症状と対策の詳細は、第5章の「落とし穴1」で述べた通りです。

**罠2: git pushの認証ダイアログ。** Zennの記事はGitHubへのpushで公開されます。第5章の「落とし穴2」で述べた通り、GIT_TERMINAL_PROMPT=0で対話を禁止して「即失敗」に変え、認証自体は事前にGit Credential Managerへ保存しておきます。

**罠3: 二重実行。** 前回がハングしたまま次の定時が来ると、二重に走ります。前掲のロックファイルで排他します。ただしロックが残留すると今度は永久にスキップされます。ロックにPIDを書き、そのプロセスの生存を確認して古いロックは無効化する設計が安全です。

### 8.4 失敗に気づける仕組み

無人運用の本質は「動かすこと」ではなく「止まったことに気づけること」です。三点セットで設計します。

- **ログ設計**: 日付別ファイルに集約します。開始と終了を必ず記録します。「終了ログのない日」がハングの証拠になります。
- **結果通知**: 処理の最後に、DiscordのWebhookなどへ成否を1行送ります。

```python
import urllib.request, json
def notify(msg):
    req = urllib.request.Request(WEBHOOK_URL,
        json.dumps({"content": msg}).encode(),
        {"Content-Type": "application/json"})
    urllib.request.urlopen(req)
```

- **朝の確認ルーチン**: 通知と投稿先を1分だけ見る習慣を持ちます。

ポイントは「失敗通知」ではなく「成功通知が来ないこと」を異常のシグナルにすることです。本当に壊れたときは、失敗を通知する処理まで到達しないことが多いからです。

### 8.5 プラットフォームへの配慮

自動投稿は、各サービスの利用規約の範囲で行うのが大前提です。

- 各プラットフォームの規約とAPIのレート制限を必ず確認します。
- 量産しないこと。テンプレート的な記事の連投はスパム判定と読者離れの近道です。私自身、テンプレ調のタイトルを量産してほぼ読まれなかった経験があります。
- 第6章のhooksで作った品質ゲートを投稿前に必ず挟みます。
- 同一サービスへの投稿間隔を空けます。429(レート制限)が続いたら数日休止する判断も必要です。

自動化は「書く時間を作る手段」であって、投稿数を稼ぐ手段ではありません。この一線を守ることが、無人運用を長く続ける条件です。

### まとめ

タスクスケジューラ、バッチ、claude -p。この三つだけで毎朝のAIパイプラインは組めます。難所は登録手順ではなく、ハング・認証・二重実行という無人特有の罠と、失敗に気づく仕組みの側にあります。次章では、この仕組みの応用として824ページの静的サイトを自動生成した事例を見ていきます。
