### 6.1 hooksの本質——AIの行動の前後にスクリプトを挟む

Claude Codeは自分で判断してファイルを編集し、コマンドを実行します。対話中なら人間の目が光っています。しかし第5章のようにスクリプトから呼び出し、第8章のように無人で走らせるなら、「AIを信じる」だけでは事故を防げません。

そこで使うのがhooksです。本質は一言で言えます。AIの行動の前後に、自分のスクリプトを自動で挟める仕組みです。プロンプトで「危険なコマンドは使わないで」と書くのは、あくまで"お願い"です。AIが忘れれば破られます。hooksでブロックすれば、それは破れない"ルール"になります。

主な発火タイミングを整理します。

| イベント | 発火タイミング | 主な用途 |
|---|---|---|
| PreToolUse | ツール実行の直前 | 危険操作のブロック |
| PostToolUse | ツール実行の直後 | フォーマッタ、生成物の検証 |
| UserPromptSubmit | プロンプト送信時 | 入力チェック、文脈の注入 |
| Notification | 許可待ちなどの通知時 | デスクトップ通知 |
| Stop | 応答の完了時 | 完了通知、後処理 |
| SubagentStop | サブエージェントの完了時 | 同上 |
| SessionStart | セッション開始時 | 環境情報の注入 |
| PreCompact | 会話履歴の圧縮直前 | ログの退避 |

設定は第3章で扱った`.claude/settings.json`に書きます。hookとして起動されたスクリプトは、標準入力からJSON(ツール名と引数)を受け取ります。そして終了コードで意思表示します。0なら続行です。2ならその操作をブロックし、標準エラー出力に書いた理由がAIへ伝わります。AIは理由を読んで別の方法を探します。この「exit 2で拒否できる」仕様が、品質ゲートの土台です。

### 6.2 実用例1——編集後にフォーマッタを自動実行

AIが書くコードは、インデントやクォートのスタイルが揺れがちです。編集のたびに人間が直すのは無駄なので、PostToolUseで自動整形します。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "python .claude/hooks/format.py" }
        ]
      }
    ]
  }
}
```

```python
# .claude/hooks/format.py
import json, subprocess, sys

data = json.load(sys.stdin)
path = data.get("tool_input", {}).get("file_path", "")
if path.endswith((".js", ".ts", ".json")):
    subprocess.run(f'npx prettier --write "{path}"', shell=True)
```

これでAIがファイルを保存するたびにPrettierが走ります。「整形して」と頼む必要はもうありません。

### 6.3 実用例2——危険コマンドのブロック

無人運用でいちばん怖いのは、破壊的なコマンドです。PreToolUseでBashツールを監視し、危険な文字列を含むコマンドを拒否します。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "python .claude/hooks/block_danger.py" }
        ]
      }
    ]
  }
}
```

```python
# .claude/hooks/block_danger.py
import json, sys

data = json.load(sys.stdin)
cmd = data.get("tool_input", {}).get("command", "")
banned = ["rm -rf", "git push --force", "Remove-Item -Recurse"]
for b in banned:
    if b in cmd:
        print(f"禁止コマンドを検出: {b}", file=sys.stderr)
        sys.exit(2)  # exit 2 で実行を拒否。理由はAIに伝わる
```

素朴な文字列一致ですが、無いのとは大違いです。私はこの型で、強制pushと再帰削除を常時ブロックしています。

### 6.4 実用例3——処理完了の通知

長いタスクの間、画面に張り付くのは時間の無駄です。Stopイベントで完了を知らせます。

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "powershell -c \"[console]::beep(880,500)\"" }
        ]
      }
    ]
  }
}
```

まずはビープ音で十分です。私はここをWebhook通知に置き換え、外出先のスマホでも完了が分かるようにしています。

### 6.5 実運用の検証ゲート——失敗から生まれた公開前サニタイズ

私は毎朝、記事の生成から投稿までを全自動で回しています(第8章)。運用の初期に、二つの事故が起きました。

一つ目は、タイトルと本文のドリフトです。タイトルは特定の技術の入門なのに、本文の後半がほぼ別のテーマ、という記事が量産されました。長い生成の途中で、AIが話題を見失うのです。

二つ目は、数値の捏造です。本文に、出典のない統計らしき数字がもっともらしく書き込まれていました。AIは悪意なく"それらしい数字"を作ります。気づかず公開していれば、読者を欺くところでした。

そこで公開の直前に検証ゲートを差し込みました。チェックは三つです。

- タイトルの主要キーワードが本文に一定回数以上現れるか
- 出典を示さない統計調の数値(「◯%が」など)が含まれていないか
- 必須の構成要素(見出しや免責の記述)が揃っているか

一つでも不合格なら投稿処理を中止し、記事は下書きのまま止めます。中身はただの検査スクリプトです。しかし同じことを人間の注意力でやれば、必ずいつか漏れます。「AIの出力を無条件に信用せず、機械的な関門を通す」。hooksの思想をパイプライン全体に広げたものが、この検証ゲートです。

### 6.6 増やしすぎの罠——最小構成から育てる

hooksは強力ですが、コストもあります。対象イベントのたびに毎回実行されるため、重いhookはすべての操作を遅くします。編集のたびに数十秒かかるフォーマッタを挟めば、体感速度は目に見えて落ちます。またhook自体がエラーを吐くと、原因の切り分けが一段むずかしくなります。ブロックが厳しすぎて、AIが正当な作業をできなくなることもあります。

おすすめは最小構成から始めることです。最初は「危険コマンドのブロック」と「完了通知」の二本で十分です。そして実際に事故が起きたら、その再発防止として一本ずつ足します。私の検証ゲートも、失敗が先にあって生まれました。品質ゲートは"事故駆動"で育てるのが、結局いちばん堅実です。

hooksは、AIに対する性善説をやめるための仕組みでした。次章では逆に、AIの能力を外へ広げる仕組み——MCPを扱います。
