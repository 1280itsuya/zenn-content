### 7.1 MCPは「AIに手足を増やす共通規格」

Claude Codeは、そのままでもファイルの読み書きやコマンド実行ができます。しかし、標準の手足だけでは届かない場所があります。ブラウザの操作、外部サービスのAPI、社内のデータベース。こうした「外の世界」とつなぐ仕組みがMCP(Model Context Protocol)です。

比喩で言えば、MCPは「USB-Cのような共通規格」です。パソコンにUSB-Cポートが1つあれば、キーボードも外付けディスクもカメラもつなげます。機器ごとに専用の接続方法を覚える必要はありません。同じように、Claude CodeはMCPという1つの規格を話せます。だからMCP対応の「サーバー」を差し込むだけで、新しい能力が増えるのです。

構造を図にすると次のとおりです。

```
あなた ──指示──> Claude Code(頭脳)
                     │
         ┌───────────┼────────────┐
       MCP          MCP          MCP
         │           │            │
   ファイル系     ブラウザ系    外部API系
  (資料を読む)  (Webを操作)   (GitHub等と連携)
```

ポイントは、AI本体を改造しない点です。手足(サーバー)を差し替えるだけで、できることが変わります。頭脳は同じまま、届く範囲だけが広がります。

### 7.2 MCPサーバーを追加する手順

追加は1行のコマンドで済みます。例として、ブラウザ操作用のPlaywrightサーバーを追加してみます。

```bash
claude mcp add playwright -- npx -y @playwright/mcp
```

登録済みのサーバーは一覧で確認できます。

```bash
claude mcp list
```

チームやプロジェクト単位で設定を共有したい場合は、プロジェクト直下の `.mcp.json` に書きます。実際の設定ファイル例を示します。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "C:/Users/you/docs"
      ]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

`command` が起動コマンド、`args` が引数です。`env` にはそのサーバーに渡す環境変数を書けます。`${GITHUB_TOKEN}` のように書くと、実行時に環境変数から値が展開されます。トークンそのものをファイルに書かないための書き方です(詳しくは7.5節)。

設定後にClaude Codeを起動すると、サーバーが提供するツールをAIが自分で選んで使うようになります。あなたが覚えるコマンドは増えません。「このページを開いて内容を確認して」と頼めば、ブラウザ系サーバーがあるときだけ、それが実行されます。

### 7.3 代表的なサーバーと使いどころ

MCPサーバーは数多く公開されています。まずは3系統だけ押さえれば十分です。

#### ファイル系

指定したフォルダの読み書きを提供します。作業ディレクトリの外にある資料集や、共有フォルダの原稿を扱いたいときに使います。アクセス範囲をフォルダ単位で絞れるのが利点です。

#### ブラウザ系

Playwrightなどのブラウザ自動化をAIの手足にします。「ページを開く」「フォームに入力する」「スクリーンショットを撮る」が指示ひとつでできます。APIが用意されていないWebサービスの確認や操作に向きます。

#### 外部API系

GitHub、データベース、各種SaaSなど、APIを持つサービスとの橋渡しです。Issueの起票やデータの取得を、自然言語の指示だけで行えます。

選ぶ基準は単純です。「自分が手作業でやっている操作は、どの系統か」。それに対応するサーバーを1つ入れるところから始めてください。

### 7.4 私の実運用——対話でできることは、無人運用でもできる

私は毎朝、記事の生成からQiita・Zenn・Dev.toへの投稿までを自動で走らせています(第8章)。この文脈でのMCPの価値は、次の一点に尽きます。

対話モードで使えるMCPサーバーは、`claude -p` のヘッドレス実行(第5章)でもそのまま使えます。

つまり、昼間に対話で「ブラウザで投稿後のページを開いて、表示崩れがないか見て」と試し、うまくいく手順を確認したら、その指示文をほぼそのままスクリプトに移せます。翌朝からは無人で同じ確認が走ります。「人間が対話で検証し、そのまま自動化する」という流れが、MCPによって切れ目なくつながるのです。

私のパイプラインでは、標準機能で足りる工程(記事ファイルの生成、gitでの公開)はそのままにしています。標準では届かない工程——Web上の表示確認や外部サービスとの連携——だけをMCPに任せています。何でもMCP化するのではなく、「届かない場所にだけ手足を足す」のが運用のコツです。サーバーを増やすほど、AIの判断対象も増えて動作が複雑になるからです。

### 7.5 APIトークンの管理——平文コミットは事故のもと

この節だけは飛ばさないでください。MCPで外部サービスにつなぐと、ほぼ確実にAPIトークン(サービスの合鍵)を扱うことになります。そして最悪の事故が「トークンを平文のままGitにコミットして公開する」ことです。公開リポジトリでは、漏れたトークンは短時間で第三者のスキャンに拾われると考えるべきです。私自身、トークンを平文のまま置いてしまい、慌てて無効化(revoke)した経験があります。

守るべきルールは3つです。

1. トークンは `.env` ファイルに書き、コードや設定ファイルには書かない。
2. `.env` は必ず `.gitignore` に追加する。
3. 設定ファイルからは環境変数経由で参照する。

`.env` の例です。

```bash
GITHUB_TOKEN=ghp_xxxxxxxxxxxx
```

`.mcp.json` 側は前掲のとおり `${GITHUB_TOKEN}` と書くだけです。これならリポジトリを公開しても、鍵そのものは含まれません。

もし平文でコミットしてしまったら、削除コミットでは不十分です。Gitの履歴に残り続けるためです。即座にトークン自体を無効化し、再発行してください。「鍵を取り替える」のが唯一の確実な対処です。

### 7.6 まとめ

- MCPは「AIに手足を差し込む共通規格」。本体を変えずに能力を足せます。
- 追加は `claude mcp add` の1行。チーム共有は `.mcp.json` で。
- まずはファイル系・ブラウザ系・外部API系の3系統から。
- 対話で検証した手順は、`claude -p` の無人実行でもそのまま動きます。
- トークンは `.env` に分離。平文コミットしたら即無効化・再発行。
