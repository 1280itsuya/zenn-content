本章では、私が実際に運用している開発エラー解決サイトを題材にします。実在するエラーメッセージを1件ずつ解説する静的ページを、Claude Codeとスクリプトで824ページ自動生成し、GitHub Pagesで公開した事例です。第5章のヘッドレス実行と第8章の無人運用を組み合わせた、総合演習と考えてください。

### 「エラーメッセージ1件=1ページ」というロングテール設計

開発者はエラーが出ると、エラーメッセージをそのまま検索します。「Python 入門」のような大きなキーワードは競争が激しい一方、具体的なエラー文は競合ページが少なく、検索意図も明確です。そこで方針を1つに絞りました。

- 実在するエラーメッセージ1件につき、解説ページを1枚作る
- 各ページは「原因」「最小再現コード」「解決手順」だけに絞る
- 件数を積み上げ、細く長い流入の合計で勝負する

1ページあたりの期待値は小さくても、ページ数を増やせるのが自動生成の強みです。ただし後述の通り、量が価値になるのは「1ページずつが正しい」場合だけです。

### サイト全体のアーキテクチャ

構成は4段のパイプラインです。

```text
[データ]       errors.json（実在エラーの一覧）
    ↓
[生成]         generate.py が claude -p を1件ずつ呼び出し
    ↓          Markdownで解説を出力
[静的ページ化]  build.py がHTMLテンプレートに流し込み
    ↓          docs/ 配下に .html を配置
[デプロイ]     git push → GitHub Pages が自動公開
```

データ・生成・ページ化を分離しておくと、途中で失敗しても再開できます。生成済みのMarkdownはファイルとして残るので、デザインを変えたいときも再生成は不要です。

### 工程1——入力データの用意

まず「実在するエラー」の一覧を作ります。私は自分の開発ログや公式ドキュメントで確認できたものだけを登録しました。ここを機械任せにすると、存在しないエラー文が混入します。データの入口だけは人間が守る、が原則です。

```json
[
  {
    "id": "py-modulenotfound-requests",
    "lang": "Python",
    "error": "ModuleNotFoundError: No module named 'requests'"
  },
  {
    "id": "js-cannot-read-undefined",
    "lang": "JavaScript",
    "error": "TypeError: Cannot read properties of undefined"
  }
]
```

### 工程2——生成テンプレートとclaude -p

生成には第5章のヘッドレス実行を使います。プロンプトはテンプレート化し、書式と禁止事項を固定します。

```python
import json, subprocess, pathlib

PROMPT = """あなたは技術記事の執筆者です。
次のエラーの解説をMarkdownで書いてください。
言語: {lang}
エラー: {error}
構成: 原因 / 最小再現コード / 解決手順（番号付き）
禁止: 確認できないバージョン番号や統計値を書かない。
不確かな点は「環境により異なります」と書く。"""

errors = json.loads(pathlib.Path("errors.json").read_text(encoding="utf-8"))
for e in errors:
    out = pathlib.Path(f"articles/{e['id']}.md")
    if out.exists():
        continue  # 生成済みはスキップ＝再実行に強くする
    r = subprocess.run(
        ["claude", "-p", PROMPT.format(**e)],
        capture_output=True, text=True, encoding="utf-8")
    out.write_text(r.stdout, encoding="utf-8")
```

ポイントは2つです。生成済みファイルをスキップして何度でも再実行できるようにすること。そして「書いてよいこと」だけでなく「書かないこと」を明示することです。

### 工程3——静的ページ化

MarkdownをHTMLテンプレートに流し込みます。凝ったフレームワークは不要です。

```python
import markdown, pathlib

TPL = pathlib.Path("template.html").read_text(encoding="utf-8")
for md in pathlib.Path("articles").glob("*.md"):
    body = markdown.markdown(md.read_text(encoding="utf-8"))
    html = TPL.replace("{{body}}", body)
    pathlib.Path(f"docs/{md.stem}.html").write_text(html, encoding="utf-8")
```

あわせて、全ページへのリンクを持つ索引ページとsitemap.xmlも同じ要領で生成します。静的ファイルだけなので、サーバー費用はかかりません。

### 工程4——GitHub Pagesへ自動デプロイ

リポジトリ設定で docs/ を公開対象にすれば、pushするだけで公開されます。

```powershell
git add docs articles
git commit -m "add error pages"
$env:GIT_TERMINAL_PROMPT = "0"
git push origin main
```

GIT_TERMINAL_PROMPT=0 の意図は第5章・第8章で述べた通りです。無言のハングを「即失敗」に変え、ログで気づけるようにします。

### 大量生成の品質担保——私が踏んだ教訓

824ページ分を回して痛感したのは、品質チェックも自動化しないと量産は破綻する、という点です。

#### 重複コンテンツの排除

似たエラーからは似た解説が生成されます。私はエラー文を正規化（小文字化・ファイルパス除去）してハッシュ化し、同一ハッシュは登録段階で弾きました。生成後も、本文の先頭部分が既存ページとほぼ一致するものを検出して除外します。重複ページは読者にもサイト評価にも害しかありません。

#### 事実でない記述の検知

AIは存在しないコマンドオプションや根拠のない数値を、自然な文章で書いてしまいます。対策は2段構えです。まずプロンプトで数値やバージョンの断定を禁止します。その上で、公開前スクリプトで「%」「倍」など統計めいた表現を含むページを機械的に抽出し、目視確認に回します。数値捏造の事故と検証ゲートの考え方は第6章で述べた通りで、大量生成でも公開前のサニタイズは必須です。

#### 公開前チェックリスト

最低限、次をスクリプトで検査してから公開します。

- タイトルのエラー文が本文にも含まれるか（題名と中身のずれの検知）
- 本文が規定の文字数以上か、見出し構成が揃っているか
- コードブロックが少なくとも1つあるか
- 空ファイルや生成失敗の断片が混ざっていないか

### 縮小版チュートリアル——まず10ページから

読者はいきなり824ページを目指す必要はありません。次の手順なら週末に完走できます。

1. 自分が過去に遭遇したエラーを10件、errors.jsonに書き出す
2. 工程2のスクリプトで10件分のMarkdownを生成する
3. 出力を全件読み、間違いを自分の手で直す
4. 工程3でHTML化し、GitHubリポジトリの docs/ に置く
5. リポジトリのPages設定で docs/ を公開する

最重要なのは手順3です。10件なら全数を目視できます。ここでプロンプトの弱点（断定しすぎる、再現コードが動かない等）が見つかり、テンプレートの改善に直結します。量を増やすのは、その後で十分です。

### 量産は成功を保証しない

最後に正直に書きます。ページ数を増やしても、検索流入は保証されません。検索エンジンの評価基準は変動しますし、中身の薄いページの量産はむしろ逆効果になり得ます。この仕組みが機能する前提は、1ページずつが「実在する問題を正しく解決する」中身を持つことです。自動化が短縮してくれるのは作業時間であって、価値づくりの手抜きではありません。次章では、機械学習と組み合わせたもう1つの実践事例を紹介します。
