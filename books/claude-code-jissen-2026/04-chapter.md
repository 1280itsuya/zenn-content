前章まででClaude Codeの準備は整いました。本章は日常編です。リファクタリング、バグ調査、テスト作成、コードレビュー。この4場面で「効く指示の型」をビフォーアフター形式で紹介します。私自身、記事生成から投稿までのパイプライン開発でClaude Codeを毎日使っており、ここで挙げる型はすべて実際の開発で磨いたものです。

### 場面1: リファクタリング——範囲と不変条件を固定する

まず悪い例です。

```text
このコードをきれいにリファクタリングして
```

これで返ってくるのは「AIの考える理想形」です。命名からファイル構成まで変わり、差分が読めなくなります。

```text
src/posters/qiita.py の post_article 関数だけを
リファクタリングしてください。
目的: リトライ処理の重複をなくす。
条件: 関数のシグネチャと外から見た挙動は変えない。
完了後、変更点を箇条書きで説明してください。
```

型は「範囲・目的・不変条件」の3点セットです。特に「変えないこと」の明示が効きます。AIは指示のない部分を自由に解釈するため、守るべき境界を先に引くのです。

### 場面2: バグ調査——修正を頼まず、まず仮説を出させる

```text
動かないので直してください
```

これだと、原因を特定しないまま「それらしい修正」が入ることがあります。直っていないのにコードだけ変わる、最悪のパターンです。

```text
バグ調査をお願いします。まだ修正はしないでください。
現象: 投稿スクリプトが週末だけ空の本文を投稿する。
期待: 平日と同じく本文が入る。
再現: python post.py --date 2026-06-21 で再現。
ログ: logs/post.log
原因の仮説を3つ、根拠のコード箇所つきで挙げてください。
```

ポイントは2つあります。「現象・期待・再現手順」を渡すこと。そして調査と修正を分けることです。仮説を読み、納得してから「仮説2の線で修正してください」と進めます。エラーメッセージやスタックトレースがあるなら、要約せず原文のまま貼るのも重要です。人間には雑音に見える1行が、AIには決定的な手がかりになることがよくあります。私は機械学習の予想サイトで「本番だけ精度が落ちる」問題をこの型で調べ、学習時と本番とで特徴量がずれていた原因にたどり着きました。

### 場面3: テスト作成——先に仕様を言語化させる

```text
この関数のテストを書いて
```

これでも動くテストは出ます。ただし「実装をなぞっただけのテスト」になりがちです。それでは今あるバグごと固定してしまいます。

```text
calc_payout 関数のテストを書く前に、この関数が満たすべき
仕様をあなたの言葉で箇条書きにしてください。
私が確認したら、pytest で正常系3件・異常系3件を書いて
ください。境界値(0、上限)を必ず含めてください。
```

先に仕様を言わせると、AIの誤解がコードになる前に見つかります。箇条書きが自分の認識とずれていたら、その場で直せばよいのです。

### 場面4: コードレビュー——観点を絞り、複数回に分ける

```text
このPRをレビューして
```

漠然と頼むと、重要な指摘と些末な指摘が混ざった長いリストが返ってきます。

```text
git diff main の差分をレビューしてください。
今回の観点は「エラー処理の抜け」だけです。
指摘は 重大度(高/中/低)・該当箇所・理由・修正案 の形式で。
問題がなければ「なし」で構いません。
```

観点を1つに絞り、セキュリティ、性能、可読性と回を分けて実行します。「なしで構いません」と添えるのも大切です。こう書かないと、AIは無理にでも指摘をひねり出そうとするからです。

4場面に共通する原則をまとめておきます。第一に、対象範囲をファイル名や関数名で特定すること。第二に、出力してほしい形式を先に指定すること。第三に、「やらないでほしいこと」を明示することです。指示の巧拙は語彙力ではなく、この3点を書いたかどうかで決まります。

### 丸投げの失敗談——意図とずれた大変更

型の話の裏返しとして、私の失敗を書きます。あるとき「投稿パイプライン全体をエラーに強くして」と丸投げしました。返ってきたのは10ファイル超の大改修です。独自のリトライ基盤が新設され、設定ファイルの形式まで変わっていました。技術的には立派でしたが、私の意図とは別物です。差分が大きすぎてレビューしきれず、結局すべて破棄しました。

曖昧で大きなタスクを渡すと、AIは空白を「AIなりの正解」で埋めます。それが自分の設計意図と一致する保証は、どこにもありません。

#### 対策: 計画→分割→一歩ずつ

```text
投稿失敗時のリトライを追加したいです。まだ実装はせず、
変更が必要なファイルと作業手順を番号つきで提案してください。
```

まず計画だけを出させ、それをレビューします。手順に納得したら「手順1だけ実装してください」と進め、1ステップ終わるごとに差分を確認します。目安は「1回の依頼は1コミットに収まる粒度」です。大きな仕事ほど、指示は小さく刻みます。途中で方向がずれたと感じたら、会話の流れで軌道修正するより、いったん変更を捨てて指示を書き直すほうが早いことも多いです。ずれた前提の上に修正を重ねると、会話もコードも複雑になっていくからです。

### AI併用前提のgit運用

段階的な指示は、gitの習慣とセットで初めて機能します。

#### こまめなコミットは巻き戻しボタン

AIの変更を確認して受け入れたら、その場でコミットします。1ステップ=1コミットです。次のステップでAIがコードを壊しても、直前の状態へ一瞬で戻れます。コミットメッセージ自体もClaude Codeに書かせられます。

#### ブランチで隔離する

AIに任せる作業は、必ず作業ブランチで行います。

```bash
git switch -c ai/add-retry
# ここで Claude Code に作業させる
git diff main --stat  # まず変更規模だけ確認
git diff main         # 中身を読む
```

mainを直接触らせない。これだけで事故の影響範囲は大きく変わります。

#### 差分レビューを習慣にする

受け入れ前に git diff を必ず読みます。読み切れない量の差分は受け入れず、「差分が大きすぎます。◯◯の変更だけに縮めてください」とやり直させます。読める差分だけを積む。これがAI併用開発の生命線です。

まとめると、ワークフローは次の通りです。ブランチを切る→小さく指示する→diffを読む→コミットする→これを繰り返す→最後に全体を通しでレビューしてマージする。

### 結論: AIの出力は必ずテストと動作確認で検証する

本章の型をすべて使っても、AIの出力は間違うことがあります。もっともらしいコードほど危険です。だから最後の砦は昔ながらの方法、つまりテストの実行と実際の動作確認です。私も生成されたコードや記事は、公開前に必ずスクリプトと目視で検証しています。「AIを頼るが、検証なしでは信用しない」。これが本章の結論です。

そしてこの「人間による検証」は、後の章で自動化の主役になります。第5章の claude -p で実行が無人化すると、人間が毎回diffを読む運用は成立しません。第6章では hooks を使い、本章で手動で行っていた検証をAIの動作に自動で差し込む「品質ゲート」へ進化させます。手で覚えた検証だけが、自動化できる検証です。まずは本章の型を、日々の開発で手に馴染ませてください。
