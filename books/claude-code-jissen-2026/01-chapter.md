### 1.1 ある夜のデバッグ、二つのやり方

まず、二つの作業風景を比べてください。

**チャットAIを使う場合。** テストが落ちました。エラーメッセージをコピーし、ブラウザのチャット欄に貼り付けます。「関連するコードも見せてください」と言われ、ファイルを探してまた貼り付けます。修正案をエディタに貼り戻し、テストを実行します。まだ落ちます。新しいエラーをまたコピーします。この往復を、人間が全部担当します。

**Claude Codeを使う場合。** ターミナルでこう打つだけです。

```
claude "テストが落ちている。原因を調べて修正し、テストが通ることを確認して"
```

Claude Codeは自分でテストを実行し、エラーを読みます。関連ファイルを開き、原因を特定し、コードを編集し、もう一度テストを走らせます。通るまで繰り返し、最後に「何をどう直したか」を報告します。あなたの仕事は、結果のdiffを確認することだけです。

### 1.2 決定的な違いは「手」があること

チャットAIは頭脳だけの存在です。優秀ですが、あなたのPCには触れません。だからコピペという「人間の手」が常に必要でした。

Claude Codeはターミナルで動くCLIツールです。同じ頭脳に、次の「手」が付いています。

- ファイルを読む・検索する・編集する
- シェルコマンドを実行する(ビルド、テスト、git操作)
- 実行結果を見て、自分で次の行動を決める

この最後の一つが本質です。結果を観察し、失敗なら方針を変えて再試行する。この自律ループを回せるAIを、本書では「エージェント型」と呼びます。コピペの往復が消えるだけでなく、「人間が張り付かなくても仕事が進む」という別の性質が生まれます。

なお、勝手に何でも実行するわけではありません。危険な操作には許可を求める仕組みがあり、どこまで自動で許すかはsettings.jsonで制御できます(第3章)。定型指示を登録するスラッシュコマンドや、作業を任せるサブエージェントなどの機能も、本書で順に使っていきます。

### 1.3 本書を読み終えると作れるもの

抽象論ではなく、私が毎日動かしている実物を挙げます。

- **記事の全自動パイプライン。** 毎朝、Claude Codeが技術記事を生成し、Qiita・Zenn・Dev.toへ自動で投稿します。人間は起きたら結果を眺めるだけです。
- **824ページの静的サイト。** 開発エラーの解決法を1エラー1ページで解説するサイトを、Claude Codeとスクリプトで自動生成し、GitHub Pagesで公開しています。
- **機械学習とのハイブリッド。** LightGBMの予想モデルとClaude Codeを組み合わせた、毎日自動更新の公営競技AI予想サイトです。数値計算は機械学習、文章化と更新はClaudeという分担です。

これらはすべて、Windowsのタスクスケジューラから毎朝定時に、無人で動いています。本書は、この状態に到達するまでの道のりをそのまま章にしたものです。

### 1.4 本書のロードマップ——三つの段階

Claude Codeの習熟は、次の三段階で進みます。

```
┌────────────────┐   ┌────────────────────┐   ┌────────────────────┐
│ ①対話で使う      │ → │ ②スクリプトから呼ぶ   │ → │ ③無人で回す           │
│   claude        │   │   claude -p "指示"   │   │   タスクスケジューラ    │
│   第2〜4章      │   │   第5〜7章           │   │   第8〜10章           │
└────────────────┘   └────────────────────┘   └────────────────────┘
```

**第一段階「対話で使う」(第2〜4章)。** ターミナルで対話しながら開発します。第2章でセットアップ、第3章でCLAUDE.mdとsettings.jsonによる「プロジェクトの教え方」、第4章で成果が出る指示の型を扱います。

**第二段階「スクリプトから呼ぶ」(第5〜7章)。** `claude -p`によるヘッドレス実行を覚えると、Claude Codeはスクリプトの一部品になります。第5章がその中核、第6章はhooksによる自動品質チェック、第7章はMCPによる外部サービス接続です。

**第三段階「無人で回す」(第8〜10章)。** タスクスケジューラで毎朝AIを働かせます。第8章で無人運用の設計、第9章で824ページサイトの実例、第10章で機械学習連携の実例と運用の教訓をまとめます。

### 1.5 向いているタスク、向かないタスク

期待値を正しく設定するため、実運用からの正直な所感を書きます。

**向いているタスク。**

- **定型作業の反復。** 記事生成、フォーマット変換、決まった手順のリリース作業。手順を言語化できる仕事には強いです。
- **コードの生成と修正。** テストという「正解の判定器」があると特に安定します。通るまで自分で直せるからです。
- **大量コンテンツの量産。** 824ページのサイトは、この特性の産物です。1件ずつは単純でも、総量が人間には無理な仕事です。

**向かないタスク。**

- **正確な数値や事実が命の文章。** AIは平気で数値を捏造します。私は生成文に架空の統計が混ざった経験から、公開前に数値を機械的に検査する工程を必ず挟んでいます。
- **一発で完成品が必要な仕事。** 出力は毎回揺れます。検証と再試行の仕組みごと設計するのが前提です。
- **判断そのものの委譲。** 何を作るか、公開してよいかという判断は人間の仕事です。私の無人運用でも、重要な操作には人間の承認を残しています。

Claude Codeは魔法ではありません。失敗も止まることもあります。ただし、失敗を前提に仕組みを組めば、寝ている間も働く実用的な相棒になります。その組み方を、次章から一つずつ手を動かして身につけていきましょう。
