## 自動化は「動かして終わり」ではない

パイプラインを稼働させた直後は達成感がある。しかし1週間後、あなたのダッシュボードには無情な数字が並ぶ――Zennのビュー数は2桁、アフィリ収益はゼロ。自動化が「育っていない」のだ。

自動化システムを収益エンジンに変えるには、**計測 → 学習 → 改善 → スケール**のループを意図的に設計する必要がある。

---

## ステップ1　計測基盤を先に作る

改善できるのは計測できるものだけだ。最低限、次の3指標をログに残す。

```python
# 毎投稿後に追記する軽量ログ
import json, datetime, pathlib

def log_result(channel, title, views=0, clicks=0):
    row = {
        "ts": datetime.date.today().isoformat(),
        "channel": channel,
        "title": title,
        "views": views,
        "clicks": clicks,
    }
    with open("data/results.jsonl", "a", encoding="utf-8") as f:
        f.write(json.dumps(row, ensure_ascii=False) + "\n")
```

ZennはAPIでビュー数を取得できる。24時間後にクロールして `views` を埋めるスクリプトを別タスクに仕込んでおく。「公開したら忘れる」では、何が当たったか永遠にわからない。

---

## ステップ2　週次レポートで勝ちパターンを抽出する

週に1回、`results.jsonl` を読んでビュー上位5記事のタイトル・タグ・投稿日時を抽出するスクリプトを走らせる。

```bash
python src/winner_extractor.py --top 5 --metric views
```

出力例：

```
1位 「FastAPI × Supabaseで認証APIを10分で作る」  views=318  tag=Python,FastAPI
2位 「Claude MCPの設定でハマった3つの罠」        views=241  tag=Claude,MCP
```

このデータを次の生成プロンプトに差し込む。「過去の上位記事と同じタグ・語感でタイトルを提案せよ」と指示するだけで、LLMは自然に勝ちパターンを踏襲する。感覚ではなく実測値で記事の方向性を決める――これが「育てる」の核心だ。

---

## ステップ3　改善を自動適用する仕組み

手動で設定ファイルを書き換えていると、改善が億劫になって止まる。`config/agents.json` に自動更新フックを組む。

```python
# winner_extractor.py の末尾
top_tags = extract_top_tags(results, n=3)
patch_config("agents.json", "preferred_tags", top_tags)
```

変更はGitでdiffを残す。「なぜこのタグに変えたのか」が履歴に刻まれ、後から検証できる。

---

## ステップ4　スケールは「横展開」と「縦深化」の2軸で

勝ちチャンネルが見つかったら、スケールには2方向ある。

**横展開**：同じロジックを別プラットフォームに流す。ZennでビューがとれたテーマをQiitaとDev.toにも投稿する。コードは1本、設定ファイルで投稿先を増やすだけだ。

**縦深化**：記事1本を5本に分割し、シリーズ化する。「MCPの設定」という1記事が読まれたなら、「MCPのデバッグ」「MCPとClaude連携」と枝を伸ばす。Zennはシリーズに独自の流入経路があるため、深さがそのままSEO資産になる。

---

## ロードマップまとめ

| 週 | アクション |
|---|---|
| 1週目 | 計測ログ設置・初投稿 |
| 2週目 | winner_extractor 初回実行・設定反映 |
| 4週目 | 勝ちチャンネルを別プラットフォームへ横展開 |
| 8週目 | シリーズ化・縦深化でロングテール蓄積 |

自動化は「育てるもの」だ。計測して、学んで、設定を更新する――そのループが回り始めたとき、システムは初めて本当の意味で「寝ている間に稼ぐ」エンジンになる。
