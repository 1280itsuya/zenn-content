## 第7章：月100本体制の運用安定化とパイプライン拡張ロードマップ

月100本の記事を安定して量産するには、生成ロジックより「壊れても気づける仕組み」の方が先に必要だ。一度整えれば手放しで動き続けるが、放置すると無音でコストが溶ける。この章では本番運用に不可欠な3つのガードと、note・はてなへの横展開手順を具体的に示す。

---

## 7-1 エラー通知：失敗を無音にしない

パイプラインが夜中に静かに落ちていても、翌朝まで気づかない。最低限、Discordへのエラー通知を仕込む。

```python
# utils/notify.py
import os, requests

def notify(msg: str, level: str = "INFO"):
    url = os.getenv("DISCORD_WEBHOOK")
    if not url:
        return
    emoji = {"INFO": "ℹ️", "WARN": "⚠️", "ERROR": "🔴"}.get(level, "")
    requests.post(url, json={"content": f"{emoji} {level}: {msg}"}, timeout=5)
```

各エージェントの `except` 節でこれを呼ぶだけでよい。**落とし穴①**：`timeout=5` を省くとネットワーク障害時に通知自体がハングし、パイプライン全体が止まる。必ず短いタイムアウトを設定すること。

## 7-2 コスト上限ガード：青天井を防ぐ

Claude APIは使った分だけ課金される。バグで無限ループが起きると請求が爆発する。実行前にその日の累積コストをチェックする仕組みを入れる。

```python
# utils/cost_guard.py
import json, pathlib

COST_LOG = pathlib.Path("data/daily_cost.json")
DAILY_LIMIT_USD = float(os.getenv("DAILY_COST_LIMIT_USD", "2.0"))

def check_and_record(cost_usd: float):
    today = __import__("datetime").date.today().isoformat()
    data = json.loads(COST_LOG.read_text()) if COST_LOG.exists() else {}
    total = data.get(today, 0.0) + cost_usd
    if total > DAILY_LIMIT_USD:
        raise RuntimeError(f"日次コスト上限超過: ${total:.3f} > ${DAILY_LIMIT_USD}")
    data[today] = total
    COST_LOG.write_text(json.dumps(data))
```

生成後にAPIレスポンスの `usage.input_tokens + output_tokens` からコストを計算し、この関数を呼ぶ。**落とし穴②**：制限を厳しくしすぎると月初に枠が尽きる。まず実績を1週間計測してから上限値を設定する。

## 7-3 ログ監視：翌朝30秒で状態確認

毎日の実行ログを構造化しておくと、週次レビューが30秒で終わる。

```python
# utils/run_logger.py
import json, pathlib, datetime

LOG_FILE = pathlib.Path("data/run_log.jsonl")

def log_run(channel: str, status: str, articles: int, cost_usd: float):
    entry = {
        "ts": datetime.datetime.now().isoformat(timespec="seconds"),
        "channel": channel,
        "status": status,   # "ok" | "partial" | "error"
        "articles": articles,
        "cost_usd": round(cost_usd, 4),
    }
    with LOG_FILE.open("a", encoding="utf-8") as f:
        f.write(json.dumps(entry, ensure_ascii=False) + "\n")
```

翌朝は `tail -20 data/run_log.jsonl | jq .` で前日分を確認。`status=error` が続くチャンネルは即座に調査に入れる。

## 7-4 note・はてなへの横展開

記事本文はすでに生成済みなので、投稿先を増やすだけでリーチが倍になる。

**noteへの投稿（半自動）**

note APIは非公開のため、`pyperclip` でクリップボードへコピーし手貼りが現実的。全自動を狙うなら `playwright` でSPAを操作するが、ログインセッション維持に手間がかかる。月100本体制の序盤は「生成は全自動・投稿は半自動」の分業が安定する。

**はてなブログへの投稿（AtomPub API）**

```python
import requests
from requests.auth import HTTPBasicAuth

def post_hatena(title: str, body_html: str):
    hatena_id = os.getenv("HATENA_ID")
    api_key   = os.getenv("HATENA_API_KEY")
    blog_id   = os.getenv("HATENA_BLOG_ID")
    url = f"https://blog.hatena.ne.jp/{hatena_id}/{blog_id}/atom/entry"
    xml = f"""<?xml version="1.0" encoding="utf-8"?>
<entry xmlns="http://www.w3.org/2005/Atom">
  <title>{title}</title>
  <content type="text/html">{body_html}</content>
  <app:control xmlns:app="http://www.w3.org/2007/app">
    <app:draft>no</app:draft>
  </app:control>
</entry>"""
    r = requests.post(url, data=xml.encode(),
                      auth=HTTPBasicAuth(hatena_id, api_key),
                      headers={"Content-Type": "application/atom+xml"})
    r.raise_for_status()
    return r.headers.get("Location")
```

**落とし穴③**：はてなは1日の投稿数に非公式の速度制限がある。連続投稿は `time.sleep(10)` を挟み、1時間に10本以下に抑えると安全だ。

## 7-5 拡張ロードマップ

| フェーズ | 追加施策 | 目安期間 |
|---|---|---|
| 現状 | Zenn・Qiita・はてな3本柱 | — |
| Phase 1 | note半自動＋Dev.to英語化 | 2週間 |
| Phase 2 | 勝ちテーマ自動増幅（view上位→再生成） | 1ヶ月 |
| Phase 3 | Shorts原稿自動生成（テキスト→台本） | 2ヶ月 |

Phase 2の「勝ちテーマ増幅」が最も費用対効果が高い。APIでview数を取得し、上位5記事のタグ・構成をextractorで分析して次サイクルのプロンプトへ還元する。これだけで同じコストでクリック数が1.5〜2倍になる実績がある。

---

本番運用は「動かすこと」より「壊れたことに気づけること」が難しい。通知・コストガード・ログの3点セットを最初に仕込んでおけば、あとは月次レビューでチューニングするだけで100本体制は無理なく維持できる。
