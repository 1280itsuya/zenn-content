## 自動化収益を「仕組み」として固める：スケールと撤退の判断軸

自動化で収益を得るとき、最大の罠は「動いているから成功」と思い込むことだ。スクリプトが毎朝走り、記事が量産されていても、収益が生まれなければそれは単なるコスト源である。仕組みとして固めるには、**スケールすべき条件**と**撤退すべき条件**を数値で定義しておく必要がある。

## スケールの判断軸：3つの数値を見る

スケールを決める前に、以下の3指標を30日間記録する。

```bash
# 日次ログから集計
python scripts/kpi_summary.py --days 30
# 出力例: views=1240, revenue=3200, cost_yen=480
```

- **ROI（費用対効果）**：収益 ÷ 運用コスト ≥ 3 を満たすチャネルのみ拡大対象とする
- **流入源の集中度**：上位1チャネルが流入の70%超なら先にリスク分散を優先する
- **1記事あたりの平均view**：30日平均が前月比+10%以上なら、同テーマで記事数を2倍に増やす

たとえばZennのテック記事が月1,200view・ROI=4を達成したなら、投稿頻度を週2本から週4本へ引き上げ、同時にDev.toへのクロス投稿を自動化する。

```bash
# チャネル追加は.envのフラグを切り替えるだけ
DEVTO_ENABLED=true
```

コードを書き直す必要はない。環境変数で制御できる設計にしておくことが、スケール時の工数を最小化する。

## 撤退の判断軸：3ヶ月ルール

撤退判定は感情ではなく期限で行う。「3ヶ月でview 500未満・収益¥0」のチャネルは問答無用で停止する。これを**3ヶ月ルール**と呼ぶ。

```python
# config/agents.json でチャネルごとに有効/無効を管理
{
  "comparison_articles": { "enabled": false },  # 停止例
  "zenn_tech":           { "enabled": true  }
}
```

停止の手順は①`enabled: false`に変更 → ②cronからタスクを除外 → ③ログに撤退理由を記録、の3ステップで完結する。コードを削除しないのは、後から類似チャネルを試すときの雛形になるからだ。

撤退時に重要なのは「なぜ刺さらなかったか」の仮説を残すことだ。`data/channel_postmortem.md`に競合・タグ・ターゲット読者のズレを1行でメモしておくと、次のチャネル設計で同じ失敗を繰り返さない。

## 仕組みとして固める：KPIダッシュボードと週次レビュー

スケール・撤退の判断を属人化しないために、KPIを自動集計して毎週レポートを受け取る仕組みを作る。

```bash
# Task Schedulerで毎週月曜9時に実行
python scripts/weekly_kpi_report.py --output data/weekly_report.md
```

レポートには「スケール候補チャネル」「撤退候補チャネル」を自動でリストアップさせる。人間がやることは、その判定を承認するだけだ。

収益の仕組み化とは、毎日の意思決定をゼロにすることではない。**判断の頻度を下げ、判断の質を上げる**ことだ。数値に基づいたスケール・撤退の基準を事前に決めておくだけで、感情的な「もう少し様子を見よう」という先送りを排除できる。それが自動化収益を本物の「仕組み」として機能させる唯一の方法である。
