## 第5章：個人開発パイプラインの落とし穴15選——ハング・403・タイトル本文ドリフトの実体験デバッグ録

自動化パイプラインは「動き始めたら終わり」ではない。筆者が実運用で踏んだ失敗を、原因・症状・修正パッチとともに列挙する。

---

## 1. `claude -p` パイプがオーケストレーター内でハング

**症状：** タスクスケジューラ起動から6〜12時間、CPU使用率ほぼ0%のまま進行なし。投稿もレポートも届かない。

**原因：** `claude -p` はサブプロセスとして標準入力を待ち続ける場合がある。パイプラインが `subprocess.run()` でラップしていると、子プロセスが終了しないまま親もブロックされる。

**修正：**

```python
result = subprocess.run(
    ["claude", "-p", prompt],
    capture_output=True, text=True,
    timeout=120,          # 必ずタイムアウトを設定
    check=True
)
```

さらにオーケストレーターがハングした際に残るロックファイルは手動で除去する必要がある。

```bash
Remove-Item "$env:TEMP\pipeline.lock" -Force
```

## 2. `~/.claude.json` 書き込み競合

**症状：** 対話セッション中にパイプラインをテスト起動すると、最初の生成だけ破損する。2回目以降は正常。

**原因：** Claude CLIは起動時に `~/.claude.json` を読み書きする。対話セッションと並走させると同一ファイルへの競合書き込みが発生する。

**対策：** パイプラインはTask Schedulerなど対話セッション外から起動する。開発中のテスト実行は必ずCLIを終了してから行う。

## 3. Qiita 429（レート超過）

**症状：** 連続投稿後に `HTTP 429 Too Many Requests`。以降の投稿が全スキップされる。

**修正：** posterに集中レートゲートを設ける。

```python
QIITA_MIN_INTERVAL_HOURS = float(os.getenv("QIITA_MIN_INTERVAL_HOURS", "5"))

def can_post_qiita(last_posted: datetime) -> bool:
    elapsed = (datetime.now() - last_posted).total_seconds() / 3600
    return elapsed >= QIITA_MIN_INTERVAL_HOURS
```

---

## 4. Qiita 403（トークン失効）

**症状：** ある日突然、全投稿が `403 Forbidden`。コードに変更はない。

**原因：** Qiita APIトークンは手動で再発行が必要なケースがある（セキュリティポリシー変更等）。

**対策：** `.env` の `QIITA_TOKEN` を確認し、Qiita設定画面でトークンを再発行して差し替える。発生したら数日は投稿を休止する。

---

## 5. Zenn git push がTTY待ちで2時間ハング

**症状：** Zenn投稿スクリプトが `git push` でフリーズ。ログに何も出ない。

**原因：** GitHub認証ダイアログがTTYを要求し、非対話プロセスが無限待ちになる。

**修正：**

```bash
# .env に追加
GIT_TERMINAL_PROMPT=0

# リモートURLにPATを直接埋め込む
git remote set-url origin https://$GITHUB_PAT@github.com/user/zenn-content.git
```

PAT平文埋め込みはセキュリティリスクがあるため、リポジトリはプライベートに設定すること。

## 6. Zenn投稿が静かにスキップされる

**症状：** スクリプトはエラーなく終了するが、Zennに記事が公開されない。

**原因：** `load_dotenv()` の呼び出し漏れ。環境変数 `GITHUB_PAT` が読まれず、push処理が `if not PAT: return` で無言スキップしていた。

**修正：**

```python
from dotenv import load_dotenv
load_dotenv()  # ← ファイル先頭に必ず置く

PAT = os.getenv("GITHUB_PAT")
if not PAT:
    raise EnvironmentError("GITHUB_PAT が未設定です")
```

筆者のコードベースで同じ漏れが6ファイルで確認された。新規posterを追加するたびに確認する。

## 7. タイトルと本文のドリフト

**症状：** 「節約レシピ10選」というタイトルなのに本文はAI副業の話になっている。

**原因：** エージェントが長いプロンプトの途中でテーマを忘れ、直前のコンテキストに引きずられる。

**修正：** 生成後に検証ステップを挟む。

```python
def validate_content(title: str, body: str) -> bool:
    title_keywords = set(title.split())
    body_sample = body[:300]
    overlap = sum(1 for kw in title_keywords if kw in body_sample)
    if overlap < 2:
        raise ValueError(f"タイトルと本文の乖離を検出: {title}")
    return True
```

## 8. テンプレートタイトルによるView≒0

**症状：** Qiitaに毎日投稿しているが、view数がずっと0〜2。

**原因：** タイトルが `「{topic}」について解説します` のような汎用テンプレート。検索に引っかからない。

**修正：** LLMに具体的なタイトルを生成させ、`{topic}` などのプレースホルダが残っていたら投稿を中止する。

```python
if re.search(r'\{.+?\}', title):
    raise ValueError(f"未展開のプレースホルダ検出: {title}")
```

## 9. 学習とlive予測の特徴量不一致

**症状：** 機械学習モデルが学習時は高精度なのに、live予測では `NaN` だらけで的中しない。

**原因：** 学習データには `crs_avg_time` 等の特徴量があったが、live取得スクリプトが同じカラムを生成していなかった。

**対策：** 学習と推論で同じ前処理関数を共有する。

```python
# features.py（学習・live共通）
def build_features(df: pd.DataFrame) -> pd.DataFrame:
    df["crs_avg_time"] = df.groupby("racer_id")["time"].transform("mean")
    return df
```

## 10. 検証がレース前キャッシュを読んで実績ゼロ

**症状：** 競輪AI予想の実績が何週間経っても0件のまま。

**原因：** 実績検証スクリプトが `use_cache=True` のままで、レース前のキャッシュ（結果未確定）を読んでいた。

**修正：**

```python
result = fetch_race_result(race_id, use_cache=False)  # 必ずfreshデータを取得
```

## 11. killしたプロセスのロックファイル残留

**症状：** パイプラインを強制終了後、再起動しても「既に実行中」とスキップされる。

**原因：** ロックファイルが削除されずに残っている。

**対策：** 起動時に古いロックを検出して除去する。

```python
LOCK = Path("pipeline.lock")
if LOCK.exists():
    age = time.time() - LOCK.stat().st_mtime
    if age > 3600:  # 1時間以上古ければ無効とみなす
        LOCK.unlink()
```

## 12. 一括削除の厳密マッチ漏れ

**症状：** 「AIの副業」というタイトルのゴミ記事を削除しようとしたが、「AIの副業入門」が残った。

**原因：** `if title == target_title` の完全一致ではなく `in` で部分マッチしていた。逆パターンも危険。

**対策：** 削除処理は完全一致のみ。削除前に対象一覧を表示して確認ステップを入れる。

---

## 13. Bluesky投稿の重複タスク

**症状：** Blueskyに同じ記事が1日に9回投稿される。

**原因：** 複数のスケジューラタスク（`BlueskyBot_1h`〜`BlueskyBot_9h`）が全て有効になっていた。

**対策：** 1チャネル1タスクを原則とする。既存タスクを無効化してから新規追加する。コメントで「再有効化禁止」を明記。

---

## 14. Gumroad商品が無言で下書き状態のまま

**症状：** GumroadのAPIが `200 OK` を返しているのに、商品が公開されない。

**原因：** APIパラメーターに `published: true` を渡し忘れていた。レスポンスにエラーは出ない。

**修正：**

```python
payload = {
    "name": title,
    "price": 980,
    "published": True,   # ← 明示的に指定
}
```

---

## 15. `load_dotenv()` の多重漏れ

**症状：** 本番環境で特定のチャネルだけAPIキーエラーになる。ローカルでは再現しない。

**原因：** ローカルはシェルに環境変数がセットされているため動作する。CI/スケジューラ環境では `.env` を明示的に読まないと未定義になる。

**対策：** プロジェクト全体で `load_dotenv()` の存在をgrepで一括確認する。

```bash
# load_dotenvなしのpyファイルを検出
grep -rL "load_dotenv" src/ --include="*.py"
```

パイプラインのバグは「エラーが出ない」ことが最も危険だ。静かなスキップ・キャッシュ読み・下書き状態——いずれも正常終了コードを返しながら収益ゼロにつながる。自動化を信頼する前に、必ず「実際に公開されたか」を外部から確認する習慣を持つこと。
