## 結論：30記事を¥36で回した内訳と、損失を生む失敗4パターン

GPT-4o-mini で「執筆→judge採点→リライト」を30記事回した実測コストは **¥36（$0.24）／合計1.2M tokens／平均18.4秒/記事**。内訳を1記事単位でCSV化したところ、コストの62%が「無駄な再課金」に消えていた。原因は4パターン——①judge再実行のトークン二重課金（¥9）②長文記事のコンテキスト肥大（¥7）③JSON崩れによる再試行（¥4）④5並列時のレート制限429リトライ（¥2.4）。本章はこの計測コードと、スループットを3記事/分→11記事/分へ上げた並列化を丸ごと渡す。

```csv
article_id,in_tokens,out_tokens,jpy,sec,retries,judge_runs
001,1820,940,1.18,17.2,0,1
007,4310,1280,2.61,29.8,2,3   # コンテキスト肥大+JSON崩れ
019,1760,910,1.14,16.1,0,1
```

## tiktoken と usage 返り値で 1記事ごとに ¥・秒・トークンを記録する

`response.usage` は実課金トークンと一致するが、**API送信前に予算超過を弾く**には tiktoken で事前計測する。GPT-4o-mini の単価は入力$0.15/出力$0.60（per 1M）。以下を全呼び出しに挟むだけでCSVが埋まる。

```python
import tiktoken, time, csv
enc = tiktoken.encoding_for_model("gpt-4o-mini")
IN, OUT = 0.15/1e6, 0.60/1e6  # USD/token
JPY = 150

def measure(client, msgs, article_id, log="cost.csv"):
    pre = sum(len(enc.encode(m["content"])) for m in msgs)
    if pre > 8000:                      # 肥大を送信前に検知
        raise ValueError(f"context bloat: {pre}tok")
    t0 = time.perf_counter()
    r = client.chat.completions.create(model="gpt-4o-mini", messages=msgs)
    u = r.usage
    jpy = (u.prompt_tokens*IN + u.completion_tokens*OUT) * JPY
    with open(log, "a", newline="") as f:
        csv.writer(f).writerow(
            [article_id, u.prompt_tokens, u.completion_tokens,
             round(jpy, 3), round(time.perf_counter()-t0, 1)])
    return r.choices[0].message.content, jpy
```

## 失敗4パターンの実損額：judge二重課金・肥大・JSON崩れ・429

最大の損は **judge再実行のトークン二重課金（¥9, 全体の25%）**。リライトのたびにjudgeへ全文を再投入するため、3回ループした記事は本文トークンを3倍課金していた。差分のみ渡す `diff_only_judge` で¥9→¥2.8へ削減。

```python
def judge_payload(draft, prev_score=None, prev_draft=None):
    if prev_draft is None:               # 初回は全文
        return f"採点せよ(JSON):\n{draft}"
    # 2回目以降は変更箇所のみ → 入力トークン6割減
    import difflib
    d = "\n".join(difflib.unified_diff(
        prev_draft.splitlines(), draft.splitlines(), lineterm=""))
    return f"前回{prev_score}点。差分のみ再採点(JSON):\n{d}"
```

JSON崩れ（③, ¥4）は `response_format={"type":"json_object"}` の付け忘れが原因で、30記事中5記事がパース失敗→再試行していた。指定後は崩れ0件。

## asyncio で 5並列化し 3記事/分 → 11記事/分 にする

逐次実行は18.4秒×記事のためボトルネックはI/O待ち。`asyncio.Semaphore(5)` で5並列にすると **3.2記事/分 → 11.3記事/分（3.5倍）**、totalコストは不変（¥36）。

```python
import asyncio
from openai import AsyncOpenAI
client = AsyncOpenAI()
sem = asyncio.Semaphore(5)               # 同時5本まで

async def run_one(article_id, topic):
    async with sem:
        msgs = [{"role":"user","content":f"{topic}の記事を書け"}]
        r = await client.chat.completions.create(
            model="gpt-4o-mini", messages=msgs)
        return article_id, r.usage.total_tokens

async def main(jobs):
    return await asyncio.gather(*[run_one(i, t) for i, t in jobs])
```

## 429 を回避する exponential backoff（5並列で発生したレート制限）

並列度を5に上げた瞬間 **429が30記事中6回**発生（④, リトライで¥2.4）。`Retry-After` を尊重しつつ指数バックオフ（1→2→4→8秒）で全6回を吸収し、ドロップ0件にした。

```python
async def with_backoff(coro_fn, max_retry=5):
    for n in range(max_retry):
        try:
            return await coro_fn()
        except Exception as e:
            if "429" not in str(e) or n == max_retry-1:
                raise
            wait = min(2**n, 8)          # 1,2,4,8,8秒
            await asyncio.sleep(wait)
```

この計測CSVを1週間貯めると、記事あたり¥単価とjudge再実行回数の相関が見え、`max_retry` と並列度のチューニング点が定量で決まる。30記事¥36のうち¥15は本章の4対策だけで削れた。
