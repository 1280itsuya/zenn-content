## 90日でprobe故障を3回早期検知、平均6.2日→当日に短縮した生ログ

結論から並べます。23テスト導入前は計測probeの故障発見まで平均6.2日・KPI CSVの欠損平均41行。導入後は3回の故障すべて当日検知・欠損0行に収束しました。下のログは集計をそのまま貼ったものです。

```text
# probe_incidents_90d.csv (抜粋)

2026-04-02,0,0,test_likes_monotonic_increase

```

3回の内訳は「Zennのいいねカウント用DOMクラス名変更」「view列が全0で返るAPI仕様変更」「行数が前日比-38%で欠落」の3系統でした。

## CI実行は90日で約120分・追加費用¥0だったGitHub Actions内訳

23テストはZenn計測probeの直後にCIで回します。90日間の総実行時間は約120分でGitHub Actions無料枠(月2000分)に完全に収まり、追加費用は¥0でした。1回あたりの実測は平均39秒。

```yaml
# .github/workflows/probe-guard.yml
name: probe-guard
on:
  schedule: [{cron: "10 22 * * *"}]  # 計測probeの直後に起動
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - run: pip install pytest pandas
      - run: pytest tests/ -q --junitxml=report.xml
```

`timeout-minutes: 5` が暴走防止の保険になり、月のActions消費は90回×39秒≒59分。再実行込みでも120分で着地しました。

## 誤検知2件はZennの一時502が原因、閾値を3連続失敗へ変更した

90日で誤検知(false positive)が2件発生。両方ともZenn側の一時的なHTTP 502で、probeが空CSVを書き `test_view_not_all_zero` が落ちました。単発の502で夜間通知を鳴らさないため、判定を「3連続失敗で初めてFAIL」に変更します。

```python
# tests/conftest.py
import pandas as pd

def consecutive_failures(metric: str, n: int = 3) -> bool:
    hist = pd.read_csv("probe_runs.csv").tail(n)
    return (hist[metric] == 0).all()  # n回連続0のみ実故障扱い

def test_view_not_all_zero():
    assert not consecutive_failures("view"), "view 3連続0=実故障"
```

この変更後、誤検知は残り期間0件。502由来のノイズ通知が消え、Discord通知の信頼度が戻りました。

## KPI CSVから「いいね伸び率TOP記事」を週次集計し次テーマへ回す

ここが課金分の本題です。守ったKPI CSVを死蔵させず、いいね伸び率(7日差分÷初日いいね)で記事をランク化し、上位ニッチを次の執筆テーマへ昇格させます。

```python
# weekly_theme_picker.py
import pandas as pd
df = pd.read_csv("zenn_kpi.csv", parse_dates=["date"])
piv = df.pivot_table(index="slug", columns="date", values="likes")
growth = (piv.iloc[:, -1] - piv.iloc[:, -8]) / piv.iloc[:, -8].clip(lower=1)
top = growth.sort_values(ascending=False).head(3)
top.to_csv("next_themes.csv")
print(top)  # 例: claude-code-ci 0.92 / furusato-6sha 0.71
```

`next_themes.csv` の上位slugのタグを再利用すれば、勘ではなく実測いいね伸び率で次章テーマを決められます。

## crontabで週次集計を金曜7時に固定し、計測を意思決定へ接続する

最後に集計を運用へ固定します。毎週金曜7時にpickerを回し結果を執筆キューへ流す1行crontabで、計測→検証→テーマ選定の閉ループが完成します。

```bash
# crontab -l
0 7 * * 5 cd ~/auto-money && python weekly_theme_picker.py >> theme_log.txt 2>&1
```

23テストで欠損0行を保証したCSVだからこそ、この自動選定が誤った数字で暴走しません。probeを守ることは、次の収益記事を当てるための前提条件でした。
