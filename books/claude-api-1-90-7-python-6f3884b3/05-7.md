## 落とし穴1：バースト送信でレート制限に即死する

ループで100件を連続送信すると、30件目でこのエラーが出る。

```
anthropic.RateLimitError: 429 {"error":{"type":"rate_limit_error"}}
```

対処はexponential backoffの一択。

```python
import time, anthropic

client = anthropic.Anthropic()

def call_with_retry(prompt: str, max_retries: int = 5) -> str:
    for attempt in range(max_retries):
        try:
            resp = client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=1024,
                messages=[{"role": "user", "content": prompt}]
            )
            return resp.content[0].text
        except anthropic.RateLimitError:
            wait = 2 ** attempt
            time.sleep(wait)
    raise RuntimeError("Max retries exceeded")
```

## 落とし穴2：システムプロンプトを毎回丸ごと渡してコストが3倍になる

本当の罠は出力トークンではなく**入力トークンの肥大化**だ。500トークンの背景説明を100件流せば5万トークン≒¥200が無駄になる。プロンプトキャッシュを使えば2回目以降のコストを最大90%削減できる。

```python
resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system=[{
        "type": "text",
        "text": load_full_context(),
        "cache_control": {"type": "ephemeral"}
    }],
    messages=[{"role": "user", "content": prompt}]
)
```

## 落とし穴3：モデルをダウングレードしたら出力フォーマットが崩壊した

コスト削減でopus→sonnetに変えると、JSON出力を期待していたのにMarkdownコードブロックで返ってくる、キーが欠ける、といった現象が起きる。品質ゲートで必ず止める。

```python
import json

def generate_structured(prompt: str) -> dict:
    raw = call_with_retry(prompt)
    try:
        cleaned = raw.strip().removeprefix("```json").removesuffix("```").strip()
        return json.loads(cleaned)
    except json.JSONDecodeError:
        raise ValueError(f"出力がJSON形式でない: {raw[:200]}")
```

無言でスルーして後続処理に空データを流すのが最悪のパターン。

## 落とし穴4：並列化でTPM上限を超えてAPIキーが一時停止された

`asyncio`でスループットを上げると、TPM（Tokens Per Minute）上限を一気に突破する。Tier1のデフォルト上限は40,000 TPM。Semaphoreで同時実行数を絞り、最低限のインターバルを設ける。

```python
import asyncio

sem = asyncio.Semaphore(3)

async def bounded_call(client, prompt: str) -> str:
    async with sem:
        resp = await client.messages.create(...)
        await asyncio.sleep(0.5)
        return resp.content[0].text
```

## 落とし穴5：長期運用で出力が「ドリフト」する

同じプロンプトを毎日実行しても、モデルのマイナー更新や温度設定のブレで品質が少しずつ劣化する。1週間後に気づいたとき100件分のゴミ記事が量産済み——という事例は珍しくない。

```python
def quality_check(text: str, min_chars: int = 300) -> bool:
    if len(text) < min_chars:
        return False
    red_flags = ["以下の通りです", "まとめると", "いかがでしたか"]
    return sum(1 for kw in red_flags if kw in text) < 2

if not quality_check(output):
    raise ValueError("品質基準未達：再生成または人間レビューへ")
```

## 落とし穴6：temperature=1のままで再現性がゼロになる

創作系なら問題ないが、分類・抽出・構造化タスクで`temperature=1`を使うとデバッグが不可能になる。タスク性質に合わせて使い分ける。

```python
# 分類・抽出タスク
resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    temperature=0.0,   # 決定論的な出力
    messages=[{"role": "user", "content": prompt}]
)
```

## 落とし穴7：例外を握りつぶして空データを量産する

```python
# やりがちな罠
try:
    result = call_api(prompt)
except Exception:
    result = ""  # ← 後続処理に空が流れ続ける
```

空文字を後続に流すと、DBには空レコード、投稿システムには空記事が積み上がる。必ずログに残し、パイプラインを止めるか通知を出す。

```python
import logging
logger = logging.getLogger(__name__)

try:
    result = call_with_retry(prompt)
except Exception as e:
    logger.error("API call failed: %s | prompt_hash=%s", e, hash(prompt))
    raise
```

---

7つの落とし穴に共通するのは「動いている間は気づかない」という点だ。対策をコードに埋め込み、問題が表面化する前に検出する仕組みを最初から組み込む。それが長期安定運用の前提条件になる。
