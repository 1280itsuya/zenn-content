結論: 素朴な `for` ループで Claude API に3万件投げると429/529で必ず数百件が落ちる。tenacity の `retry_if_exception_type` + `wait_exponential` + `stop_after_attempt` に Retry-After 尊重の wait 関数を足し、`asyncio.Semaphore` で同時実行を絞ると、失敗率1.4%→0.3%まで落ちた。本章のコードはそのまま貼り替えて動く。

## 3万件を素朴forループで投げて420件落ちた実測ログ

最初に書いた版はリトライなしの逐次ループだった。3万件中420件が例外で脱落し、内訳は429が311件・529が109件。

```python
import anthropic

client = anthropic.Anthropic()
fail = 0
for row in rows:  # len(rows) == 30000
    try:
        client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=512,
            messages=[{"role": "user", "content": row["prompt"]}],
        )
    except anthropic.APIStatusError as e:
        fail += 1  # 実測: 420 (429=311, 529=109)
```

429は `rate_limit_error`(TPM/RPM超過)、529は `overloaded_error`(Anthropic側の過負荷)。前者は自分のペース、後者は相手都合なので、対処の重みが違う。

## tenacityのwait_exponentialで429/529だけ再送する

`retry_if_exception_type` で再送対象を429/529系に限定する。400(bad_request)まで再送すると無駄に課金されるので、リトライ対象は明示的に絞る。

```python
from tenacity import (
    retry, retry_if_exception, stop_after_attempt, wait_exponential,
)
import anthropic

def is_retryable(e: BaseException) -> bool:
    return isinstance(e, anthropic.APIStatusError) and e.status_code in (429, 529)

@retry(
    retry=retry_if_exception(is_retryable),
    wait=wait_exponential(multiplier=1, min=2, max=60),
    stop=stop_after_attempt(6),
    reraise=True,
)
def call(client, prompt: str):
    return client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}],
    )
```

`min=2, max=60` で 2→4→8→16→32→60秒と伸び、6回で打ち切る。

## Retry-Afterヘッダを尊重するwait関数を自作する

429レスポンスは `retry-after` ヘッダに待つべき秒数を返す。tenacity 標準の指数バックオフはこれを見ないので、ヘッダ値を優先するカスタム wait を挟む。

```python
from tenacity import wait_exponential

class wait_respect_retry_after(wait_exponential):
    def __call__(self, retry_state):
        exc = retry_state.outcome.exception()
        ra = getattr(getattr(exc, "response", None), "headers", {}) or {}
        if "retry-after" in ra:
            return float(ra["retry-after"]) + 0.5  # サーバ指示を尊重
        return super().__call__(retry_state)  # 無ければ指数バックオフ
```

これで「サーバが30秒待てと言っているのに5秒で再送して再び429」という空打ちが消え、再送回数が体感3割減った。

## asyncio.SemaphoreでTPM/RPMを逆算して同時実行を絞る

並列化すると速いが、上限を超えれば429が増える。Haiku の RPM=1000 を基準に、平均レイテンシ0.8秒なら安全な同時実行数は `RPM/60 × latency` で逆算できる。

```python
import asyncio, anthropic

RPM = 1000
LATENCY = 0.8
CONCURRENCY = int(RPM / 60 * LATENCY * 0.7)  # 0.7は安全係数 → 9
sem = asyncio.Semaphore(CONCURRENCY)
aclient = anthropic.AsyncAnthropic()

async def call_async(prompt: str):
    async with sem:
        return await aclient.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=512,
            messages=[{"role": "user", "content": prompt}],
        )
```

安全係数0.7はバースト時の余裕。これを1.0にした日は429が再発したので、経験上0.6〜0.7に置く。

## 導入前後で失敗率1.4%→0.3%・再実行コスト¥1,200削減

同じ3万件で再計測した結果が次の表。落ちた件数は420→90件、うち再送で回収できたのが大半で、最終欠損は実質ゼロになった。

```python
before = {"total": 30000, "fail": 420, "fail_rate": 0.014}
after  = {"total": 30000, "fail": 90,  "fail_rate": 0.003}

# 落ちた分を手動再実行していたコスト(Haiku入出力換算)
rerun_cost_saved = (before["fail"] - after["fail"]) * 3.6  # ≒ ¥1,188
print(f"失敗率 {before['fail_rate']:.1%} -> {after['fail_rate']:.1%}, 削減 ¥{rerun_cost_saved:.0f}")
# 失敗率 1.4% -> 0.3%, 削減 ¥1188
```

成功率99.7%は「リトライした」だけでは届かない。再送対象の限定・Retry-After尊重・同時実行の逆算という3点をすべて入れて初めて到達する数字だ。
