## 失敗7件と損失額｜output_tokens青天井で1本300円が出た原因ログ

6か月で出た失敗は7件、累計損失は約$42。最大は `max_tokens` 未指定での暴走で、1本$2.1(約300円)が3本連続した。原因は「指定しなければモデル上限まで使う」仕様を握り潰したこと。cost_tracker のログを `usd` 降順で並べると一目で分かる。

```python
# losses.py — 失敗を損失額で順位付け
import sqlite3
rows = sqlite3.connect("cost.db").execute(
    "SELECT title, usd, out_tok FROM runs WHERE usd > 0.5 ORDER BY usd DESC LIMIT 7"
).fetchall()
for t, usd, o in rows:
    print(f"${usd:.2f}  out={o:>6}  {t[:24]}")
# $2.10  out=  8021  事業計画テンプレ全部入り...
```

## prompt cache前方不一致でヒット率0%継続を検知する

2番目の損失は、system プロンプトに日付を埋め込んだことでキャッシュの前方一致が毎回崩れ、ヒット率0%が3週間続いた件($9相当)。OpenAI は `prompt_tokens_details.cached_tokens` を返すので、これを保存して比率を出せば即検知できる。

```python
def cache_hit_rate(usage) -> float:
    cached = usage.prompt_tokens_details.cached_tokens
    return cached / max(usage.prompt_tokens, 1)

r = cache_hit_rate(resp.usage)
if r < 0.3:                      # 60本中ヒット率3割未満は異常
    raise RuntimeError(f"cache miss streak: {r:.0%} — 可変要素をsuffixへ移動せよ")
```

修正は1行、可変の日付を system 先頭から user 末尾へ移すだけ。これでヒット率は0%→61%に戻った。

## $31→$18への内訳｜gpt-4o-mini中心化とmax_tokens上限の効果分解

月次請求 $31→$18(▲42%)は3施策に分解できる。gpt-4o中心→gpt-4o-mini中心化で▲$8、キャッシュ整列で▲$3、`max_tokens=1500` 上限で▲$2。施策ごとの寄与はモデル別に月次集計する。

```sql
-- 月別・モデル別の請求寄与
SELECT strftime('%Y-%m', ts) m, model, ROUND(SUM(usd),2) usd, COUNT(*) n
FROM runs GROUP BY m, model ORDER BY m DESC;
-- 2026-06  gpt-4o-mini  14.2  57
-- 2026-06  gpt-4o        3.8   3   ← 重要記事のみ残す
```

## budget_guardで月$20を物理的に超えさせない実装

Batch API の失敗を握り潰し二重課金した4件目以降を潰すため、予算ガードを generator の前段に置く。当月累計が閾値を超えたら例外で停止し、無人運用でも青天井を物理的に断つ。

```python
def assert_budget(db, cap=20.0):
    spent = db.execute(
        "SELECT COALESCE(SUM(usd),0) FROM runs "
        "WHERE strftime('%Y-%m',ts)=strftime('%Y-%m','now')"
    ).fetchone()[0]
    if spent >= cap:
        raise SystemExit(f"budget ${spent:.2f} >= ${cap} — 当月停止")
    return cap - spent
```

## 再現用リポジトリ構成と移植チェックリスト7項目

cost_tracker / budget_guard / generator を1リポジトリにまとめ、別ジャンルへ移植する手順を確定する。下のチェックを上から通せば、本書の計測基盤がそのまま動く。

```bash
repo/
├── cost.db            # runs(ts,model,usd,in_tok,out_tok,cached)
├── tracker.py         # usage→DB記録 + cache_hit_rate
├── budget_guard.py    # assert_budget(cap=20)
└── generator.py       # assert_budget→生成→tracker記録
# 移植7項目:
# [1] cached_tokens保存 [2] max_tokens必須化 [3] 可変要素をsuffixへ
# [4] cap=月予算設定   [5] hit_rate<0.3で停止 [6] model別月次SQL
# [7] Batch結果のstatus検証(failedを記録)
```

このまま `python generator.py` を回せば、1本ごとのドル建て請求とキャッシュヒット率が `cost.db` に積み上がり、$31→$18の再現を自分のテーマで検証できる。
