## 第5章　自律的に稼ぎ続ける仕組みの育て方

### 「動かす」から「育てる」へ

AIに稼がせる仕組みを作れたとしても、それはゴールではない。仕組みは放置すれば必ず腐る。APIの仕様変更、プラットフォームの規約改定、読者の飽き——外部環境は常に動いている。自律的に稼ぎ続けるには、仕組みを「育てる」フェーズに移行しなければならない。

### 計測なき改善は存在しない

最初にすべきことは、**何が機能しているかを数値で把握する**ことだ。たとえばZennやQiitaに記事を自動投稿しているなら、記事ごとのview数をAPIで毎日回収してCSVに記録する。

```bash
python src/winner_extractor.py --platform zenn --days 7
```

このスクリプトは直近7日間のview上位記事を抽出し、共通するタグ・タイトルパターン・投稿時間帯を`data/winners.json`に書き出す。感覚ではなく、実測値から「勝ちパターン」を特定する。

### 勝ちパターンを次世代生成に反映する

抽出した勝ちパターンは、次の生成プロンプトに自動でフィードバックする。

```python
# orchestrator.py の一部
winners = load_winners("data/winners.json")
prompt = build_prompt(topic, exemplars=winners[:3])
```

これにより、モデルは過去に実際にviewを集めた記事のトーンと構成を模倣しながら新記事を生成する。人間が毎回プロンプトを書き直す必要はない。仕組みが仕組みを改善する、という循環が生まれる。

### 死活監視と自動復旧

仕組みが止まっても気づかない状態は最も危険だ。Pythonのスケジューラと組み合わせて、定期実行の成否をSlackやDiscordへ通知する。

```python
# health_check.py
if last_run_at < now - timedelta(hours=25):
    notify_discord("⚠️ 自動投稿が25時間以上停止しています")
```

さらに、ロックファイル（`.lock`）を使ってプロセスの二重起動を防ぎ、異常終了時はロックを自動除去してから再起動するリカバリロジックを入れておく。手作業での復旧は属人化の入口になる。

### 環境変化への適応設計

プラットフォームのAPIが変わった際、修正が一か所で済む設計にしておくことが長期運用の命綱だ。投稿ロジックはすべて`posters/`以下に分離し、オーケストレーター側は呼び出すだけにする。新しい投稿先を追加するときも、`.env`に1行加えるだけでパイプラインに組み込める構造にする。

```
HASHNODE_TOKEN=xxx  # これを追記するだけで自動投稿が始まる
```

### 月次レビューで仕組みをアップデートする

毎月1回、以下の3点を確認する習慣を作る。

1. **収益・view数のトレンド**——上昇しているか、横ばいか、下落か
2. **停止・スキップされたタスクの原因**——ログを見て根本原因を潰す
3. **新チャネルの追加候補**——競合が薄く、自動化しやすい領域を1つ試す

仕組みは一度作れば終わりではなく、毎月少しずつ賢くなっていくものだ。その積み重ねが、半年後には「寝ている間に収益が入る」状態を現実にする。
