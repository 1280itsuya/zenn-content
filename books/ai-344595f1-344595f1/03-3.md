## 自動化ルーティンの「骨格」を先に決める

収益が出ない自動化の多くは、「とりあえずスクリプトを書いた」状態で止まっている。設計の順番が逆なのだ。正しい順番は次の3ステップだ。

1. **収益動線を決める**（何をどこで売るか）
2. **ボトルネックを特定する**（人手が介在している箇所）
3. **そこだけを自動化する**

たとえば「AI記事 → Zenn公開 → アフィリリンク → 収益」という動線なら、ボトルネックは「記事生成」と「投稿」の2点だけ。そこを機械に渡せばよい。

## 毎朝7時に動く「3段ロケット」構造

実際に筆者が運用しているルーティンの構造を示す。

```
07:00  orchestrator.py 起動
 └─ 01_generate.py   # Claude APIで記事生成
 └─ 02_review.py     # LLM-as-judgeで品質スコアリング
 └─ 03_post.py       # Zenn/Qiita/はてなへ自動投稿
```

Task Schedulerへの登録は1行で済む。

```powershell
schtasks /create /tn "AutoMoney" /tr "python orchestrator.py" /sc daily /st 07:00
```

ポイントは**生成・審査・投稿を分離**することだ。一体化すると品質チェックが飛ばせず、後からチャンネルを追加するときに修正箇所が爆発する。

## 「スキップ設計」で壊れないルーティンにする

自動化が止まる最大の原因は、1つの失敗が全体を止めることだ。各ステージは必ず**独立して失敗できる**構造にする。

```python
if not os.getenv("ZENN_PAT"):
    logging.warning("ZENN_PAT未設定 → Zenn投稿をスキップ")
    return
```

`.env`の変数が未設定のチャンネルは静かにスキップされる。これにより、新しいプラットフォームの追加も「環境変数を1行追加する」だけで済む。

## 週次レポートで「何が稼いでいるか」を可視化する

自動化は「動いていること」が目的ではない。**どのチャンネルが何件のviewと収益を生んでいるか**を毎週数値で見ることで、伸びているものにリソースを集中できる。

```python
# winner_extractor.py の抜粋
top_articles = df.nlargest(5, "views")[["title", "views", "channel"]]
```

上位5記事のタイトル・タグ・チャンネルを抽出し、次週の生成プロンプトに差し込む。これが「勝ちパターンの増幅ループ」だ。自動化は一度作れば終わりではなく、**計測→改善→再投入**のサイクルを回すことで複利的に育っていく。
