## 自動エンジンが止まる3つの理由

自動収益エンジンは「作って終わり」ではない。実運用では必ず止まる。原因はほぼ3パターンだ。①外部API仕様変更（Qiita・Zennのレート制限など）、②認証トークンの失効、③スクリプトのハングアップである。止まっても気づかなければ損失は静かに積み上がる。「動いているはず」という思い込みが最大の敵だ。

## ヘルスチェックを毎朝自動実行する

Task Schedulerに以下のコマンドを登録し、毎朝6:55（本番起動の5分前）に実行させる。

```powershell
python scripts/health_check.py --notify discord
```

`health_check.py` は各チャンネルの「前日投稿件数」と「直近エラーログ」を確認し、0件または例外検出時にDiscordのWebhookへ警告を送る。これだけで「今日は何も投稿されていない」を人間が気づく前に捕捉できる。

## 改善ループの回し方

運用改善はPDCAではなく**OAR（観測→分析→書き換え）**サイクルで回す。

**1. 観測（毎日自動）**
```bash
python scripts/winner_extractor.py --days 7
```
Zenn・Qiitaのview数上位3記事のタグ・テーマ・文字数を抽出し `data/exemplars.json` に保存する。

**2. 分析（週1回・5分）**
exemplarsを眺め「なぜ伸びたか」を1行メモする。例：「`#Python` × エラー実例記事がview 340→他ニッチより3倍高い」。

**3. 書き換え（週1回・10分）**
```python
# config/agents.json の topic_pool を更新
"topic_pool": ["TypeErrorの直し方", "FastAPI 422エラー完全解決", ...]
```
エージェントが次サイクルから新テーマで生成を始める。コード修正はゼロ、設定だけで軌道修正できる構造が重要だ。

## 長期ハング対策：タイムアウトとロック管理

orchestratorが外部API応答待ちで数時間フリーズする事故が起きやすい。必ず `subprocess.run` にタイムアウトを設定し、ロックファイルを使って二重起動を防ぐ。

```python
import subprocess, os, sys

LOCK = "run.lock"
if os.path.exists(LOCK):
    sys.exit("already running")
open(LOCK, "w").close()
try:
    subprocess.run(["python", "orchestrator.py"], timeout=3600)
finally:
    os.remove(LOCK)
```

タイムアウト後にロックが残る場合は `Remove-Item run.lock` で手動除去してから再実行する。

## KPIを数値で見る習慣

週次で以下の3指標だけを確認する。

| 指標 | 目標 | 確認コマンド |
|------|------|-------------|
| 週間投稿本数 | ≥21本 | `python scripts/post_summary.py` |
| Zenn週間view | 前週比+10% | Zennダッシュボード |
| エラー率 | ≤5% | `grep ERROR logs/orchestrator.log \| wc -l` |

数値が目標を下回った週だけexemplarsとtopic_poolを見直す。それ以外は触らない。**動いているエンジンを過剰に弄るのが最大のリスク**だと覚えておこう。
