## 結論：最小5秒間隔＋指数バックオフで100記事を429ゼロ巡回できる

Zennの統計エンドポイント `/api/me/dashboard/articles` は、短時間に連続で叩くと `429 Too Many Requests` を返す。検証では1秒間隔で巡回したところ12記事目、28記事目、41記事目の計3回ブロックされた。対策は単純で、最小5秒間隔・指数バックオフ・ジッタの3点を入れるだけで100記事を429ゼロで巡回できた。逐次処理のまま、総所要時間は約9分に収まる。

```bash
# 1秒間隔で叩いた再現（3回429を踏んだ実測コマンド）
for id in $(cat article_ids.txt); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Cookie: $ZENN_COOKIE" \
    "https://zenn.dev/api/me/dashboard/articles/$id" && sleep 1
done | sort | uniq -c
# => 97 200 / 3 429
```

## Retry-Afterヘッダを実測して待機秒数を10秒に固定する

429時のレスポンスヘッダを丸ごと記録すると、`Retry-After: 10` が返っていた。Zennはリトライ秒数を明示してくるので、これを無視して即リトライすると連鎖ブロックになる。`Retry-After` があればその値を、なければバックオフ計算値を採用する。

```python
import time

def wait_seconds(resp, attempt: int, base: float = 5.0) -> float:
    ra = resp.headers.get("Retry-After")
    if ra is not None:
        return float(ra)                     # 実測では 10 秒
    return base * (2 ** attempt)             # 5, 10, 20, 40...
```

## 指数バックオフ＋ジッタのリトライ装飾子をprobeに実装する

最小5秒間隔を保証しつつ、429時は `5→10→20` 秒と倍化、さらに0〜1.5秒のジッタを足して同期的な再衝突を避ける。装飾子にして全probe呼び出しへ一括適用した。

```python
import functools, time
from random import uniform

def throttled(min_interval=5.0, max_retries=5):
    last = {"t": 0.0}
    def deco(fn):
        @functools.wraps(fn)
        def wrap(*a, **k):
            for attempt in range(max_retries):
                gap = min_interval - (time.monotonic() - last["t"])
                if gap > 0:
                    time.sleep(gap)
                resp = fn(*a, **k)
                last["t"] = time.monotonic()
                if resp.status_code != 429:
                    return resp
                time.sleep(wait_seconds(resp, attempt) + uniform(0, 1.5))
            raise RuntimeError("429 retries exhausted")
        return wrap
    return deco
```

## asyncioの同時8接続で429が17回噴出した失敗と逐次にした理由

`asyncio` ＋ `aiohttp` で同時8接続にしたところ、最初の10秒で429が17回発生し、全体の所要時間はむしろ14分へ悪化した。Zenn側のレート判定はIP単位で、並列数を増やすほど単位時間あたりのリクエストが密集して逆効果になる。1記事あたり平均応答0.3秒のエンドポイントを相手に並列化する旨味はなく、逐次＋5秒スリープが最も速く安全という結論に達した。

```python
# 採用しなかった並列版（429が17回噴出）
import asyncio, aiohttp
async def fetch_all(ids, cookie):
    sem = asyncio.Semaphore(8)              # この8が事故の原因
    async with aiohttp.ClientSession(headers={"Cookie": cookie}) as s:
        async def one(i):
            async with sem, s.get(URL.format(i)) as r:
                return await r.json()
        return await asyncio.gather(*[one(i) for i in ids])
```

## before/afterの実測値とアカウントに優しいprobe設計指針

3つの指標をbefore/afterで並べると効果が明確になる。最小5秒間隔を守るだけで、リトライ発生率は3%から0%へ、巡回完了率は97%から100%へ改善した。

| 指標 | before(1秒間隔) | after(5秒+backoff) |
|---|---|---|
| 1記事平均応答 | 0.31秒 | 0.30秒 |
| 100記事総所要 | 約2分(3回失敗) | 約9分(失敗0) |
| リトライ発生率 | 3.0% | 0.0% |

```python
# 行儀の良いprobeの最小実装
@throttled(min_interval=5.0)
def probe(article_id, cookie):
    return requests.get(URL.format(article_id),
                        headers={"Cookie": cookie,
                                 "User-Agent": "zenn-stats-probe/1.0"})
```

総所要9分はcronで毎朝1回回せば十分実用域に入る。速度より「アカウントへ負荷を残さない」ことを優先し、5秒間隔・明示User-Agent・`Retry-After`尊重の3点を守れば、統計収集でブロックされるリスクはほぼ消える。
