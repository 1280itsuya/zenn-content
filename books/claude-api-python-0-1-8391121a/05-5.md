## パイプライン運用で踏んだ落とし穴5選と回避策

---

## 落とし穴①：レート制限で記事生成が途中で止まる

Claude APIにはRPM（1分あたりリクエスト数）とTPM（1分あたりトークン数）の二重制限がある。記事を並列生成すると数本目で`RateLimitError`が飛び、その後のバッチが全滅した。

**回避策：指数バックオフ付きリトライ**

```python
import anthropic
import time

client = anthropic.Anthropic()

def call_with_backoff(prompt: str, max_retries: int = 5) -> str:
    for attempt in range(max_retries):
        try:
            msg = client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=2048,
                messages=[{"role": "user", "content": prompt}]
            )
            return msg.content[0].text
        except anthropic.RateLimitError:
            wait = 2 ** attempt  # 1, 2, 4, 8, 16秒
            print(f"Rate limit hit. Waiting {wait}s...")
            time.sleep(wait)
    raise RuntimeError("Max retries exceeded")
```

並列度を上げたいなら`asyncio.Semaphore`で同時実行数を3〜5本に絞る。

## 落とし穴②：コスト爆死——1日で¥3,000溶けた

プロンプトに「前回の記事全文」を毎回貼り付けていたら、入力トークンが膨れ上がり翌朝に請求通知が届いた。

**回避策：トークンを事前にカウントして上限ガード**

```python
def safe_generate(prompt: str, budget_tokens: int = 1500) -> str:
    # client.beta.messages.count_tokens() で事前計測
    token_count = client.beta.messages.count_tokens(
        model="claude-sonnet-4-6",
        messages=[{"role": "user", "content": prompt}]
    )
    if token_count.input_tokens > budget_tokens:
        raise ValueError(
            f"Prompt too large: {token_count.input_tokens} tokens "
            f"(budget: {budget_tokens})"
        )
    return call_with_backoff(prompt)
```

さらに月次コストは`usage.input_tokens + usage.output_tokens`を`sqlite3`に記録し、日次集計でSlack通知する。

## 落とし穴③：記事ドリフト——タイトルと本文がズレる

「Python 非同期処理の基礎」というタイトルで生成させると、本文がいつの間にか「FastAPIの使い方」になっていた。LLMは長い生成の途中でテーマが滑る。

**回避策：生成後にテーマ一致を検証**

```python
def verify_topic_match(title: str, body: str) -> bool:
    check_prompt = f"""
以下の記事タイトルと本文が同じテーマを扱っているか答えてください。
「はい」か「いいえ」だけ返してください。

タイトル: {title}
本文（先頭300字）: {body[:300]}
"""
    result = call_with_backoff(check_prompt)
    return result.strip().startswith("はい")

title = "Python 非同期処理の基礎"
body = generate_article(title)

if not verify_topic_match(title, body):
    raise ValueError(f"Topic drift detected: {title}")
```

不一致が続く場合はプロンプト冒頭に`## 必ずこのテーマだけ書くこと: {title}`を強制挿入する。

## 落とし穴④：投稿重複——同じ記事がZennに二重公開された

ネットワーク瞬断のリトライで投稿関数が2回走り、同一タイトルの記事が2本並んだ。

**回避策：ファイルロック＋投稿済みIDの永続化**

```python
import fcntl, json, pathlib

POSTED_LOG = pathlib.Path("data/posted_ids.json")

def load_posted() -> set:
    if POSTED_LOG.exists():
        return set(json.loads(POSTED_LOG.read_text()))
    return set()

def mark_posted(article_id: str):
    ids = load_posted()
    ids.add(article_id)
    POSTED_LOG.write_text(json.dumps(list(ids)))

def post_once(article_id: str, content: str):
    if article_id in load_posted():
        print(f"Already posted: {article_id}")
        return
    # 実際の投稿処理
    do_post(content)
    mark_posted(article_id)
```

`article_id`はタイトルのSHA-256ハッシュで生成するとDB不要で冪等になる。

## 落とし穴⑤：APIタイムアウト——長文生成が途中で切れる

2,000字超の記事を一度に生成させると`APITimeoutError`が頻発した。ネットワーク遅延とサーバー処理時間の合算が60秒を超えるためだ。

**回避策：ストリーミング＋チャンク分割生成**

```python
def generate_long_article(title: str) -> str:
    sections = ["導入", "本論①", "本論②", "まとめ"]
    full_text = ""

    for section in sections:
        prompt = f"記事「{title}」の「{section}」節を300字で書いてください。"
        # ストリーミングで受け取り途中切断を検知
        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=512,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            chunk = ""
            for text in stream.text_stream:
                chunk += text
            full_text += chunk + "\n\n"

    return full_text
```

セクションごとに分割することで1リクエストのトークン数と処理時間を抑え、部分失敗時も該当節だけ再試行できる。

---

これら5つの問題は「動いているように見えて実は壊れている」状態で数日気づけないことが多い。**生成→検証→投稿**の各ステップにガード処理を入れ、ログを`sqlite3`か`jsonl`で必ず残すのがパイプラインを安定させる最短路だ。
