## 無料/Basicプランの実枠をYAMLで定数化する

X API v2のレート制限はエンドポイント単位で異なる。`GET /2/users/:id/tweets` は無料枠で月100件・15分窓1リクエスト、Basic($200/月)で月15,000件・15分窓5リクエストまで。副業運用では「月間取得上限」が先に枯れるため、ここを定数化して超過前に止める。

```yaml
# config/x_limits.yaml — 2026-06時点の実枠
free:
  user_timeline: { monthly_cap: 100, window_req: 1, window_sec: 900 }
basic:
  user_timeline: { monthly_cap: 15000, window_req: 5, window_sec: 900 }
  search_recent: { monthly_cap: 15000, window_req: 60, window_sec: 900 }
```

## tweepyではなく素のhttpxで叩く

tweepyは429時に内部で勝手にsleepし、`Retry-After` の実値を握り潰す。残り枠を1秒単位で制御したいので、httpxで叩いてレスポンスヘッダを直接読む。

```python
import httpx

def fetch_timeline(uid: str, token: str, since_id: str | None):
    params = {"max_results": 100, "tweet.fields": "created_at"}
    if since_id:
        params["since_id"] = since_id
    r = httpx.get(
        f"https://api.twitter.com/2/users/{uid}/tweets",
        headers={"Authorization": f"Bearer {token}"},
        params=params, timeout=20,
    )
    return r
```

## 429を指数バックオフ＋Retry-Afterで捌く

429時は `x-rate-limit-reset`(UNIX秒)が返る。これを尊重しつつ、ヘッダ欠落時のみ指数バックオフへフォールバックする。リトライ上限は5回。

```python
import time

def with_retry(call, max_retry=5):
    for i in range(max_retry):
        r = call()
        if r.status_code != 429:
            return r
        reset = r.headers.get("x-rate-limit-reset")
        wait = max(int(reset) - int(time.time()), 1) if reset else 2 ** i
        time.sleep(min(wait, 900))
    raise RuntimeError("429 retry exhausted")
```

## SQLiteで取得済みIDを重複排除する

`since_id` だけだと取りこぼし時に同一ツイートを再取得し、無料枠100件を無駄に焼く。取得済みIDをSQLiteのUNIQUE制約で弾く。

```python
import sqlite3

db = sqlite3.connect("tweets.db")
db.execute("CREATE TABLE IF NOT EXISTS seen(id TEXT PRIMARY KEY)")

def dedup(tweets: list[dict]) -> list[dict]:
    fresh = []
    for t in tweets:
        try:
            db.execute("INSERT INTO seen(id) VALUES(?)", (t["id"],))
            fresh.append(t)
        except sqlite3.IntegrityError:
            pass
    db.commit()
    return fresh
```

## 実運用3週間の実測値と損益分岐

Basicプランで21日連続稼働した結果が下表。429は計12回踏み、`Retry-After` 尊重後の再取得成功率は100%(12/12)。1日あたり消費は平均約480件。

```text
期間: 2026-05-14〜06-03 (21日)
429発生回数 : 12回 / 総リクエスト 1,008回 (1.19%)
リトライ成功 : 12/12 (100%)
1日平均取得  : 約480件 → 月換算 約14,400件
損益分岐     : Basic月15,000件 ÷ 480件/日 ≒ 31アカウント/日 が上限
```

480件/日で回すと、収集対象を1日あたり最大31アカウントまで広げてもBasic枠内に収まる。無料枠(月100件)は1アカウントの1回分で枯れるため、複数アカウント運用ならBasic一択になる。
