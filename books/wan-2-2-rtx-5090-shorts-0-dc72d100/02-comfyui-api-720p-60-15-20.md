## ComfyUI APIモード起動 — `--listen 0.0.0.0` の1フラグで有効化

ComfyUIをHTTP APIサーバーとして起動するには `--listen` と `--port` を追加するだけでよい。

```bash
# RTX 5090(32GB)向け — --highvram でモデルをVRAMに全載せ
python main.py --listen 0.0.0.0 --port 8188 --highvram
```

起動後、`curl http://localhost:8188/queue` がJSON応答を返せば準備完了。`--highvram` を省くと ComfyUI が自動でRAMオフロードを試み、5090でも初回ロードが40秒以上かかる。

## Wan 2.2 縦型720×1280ワークフローJSON — 崩壊を生む3ノードの正体

ワークフローはJSON形式のまま `/prompt` にPOSTできる。`CLIPTextEncode` → `WanVideoSampler` → `VHS_VideoCombine` の3ノードがパラメータ起因崩壊の根源。以下は最小構成の全文。

    },

## 崩壊ゼロへのパラメータ20点 — Steps=15以下でフレームが溶ける

RTX 5090実測での危険域と推奨値。**Steps=15以下は無条件でフレーム崩壊**が発生した。

| # | パラメータ | 推奨値 | 危険域 | 崩壊症状 |
|---|---|---|---|---|
| 1 | steps | 20 | ≤15 | フレーム溶解 |
| 2 | cfg | 6.0 | ≥9.0 | 彩度爆発 |
| 3 | width | 720 | ≥768 | VRAM OOM |
| 4 | height | 1280 | ≥1440 | VRAM OOM |
| 5 | video_length | 81 | ≥121 | 後半フリーズ |
| 6 | motion_bucket_id | 127 | ≤64 | カメラ完全静止 |
| 7 | motion_bucket_id | 127 | ≥200 | ブレ過多 |
| 8 | fps | 24 | 30(1280px時) | OOM落ち |
| 9 | scheduler | euler | dpm++ | ノイズ残留 |
| 10 | denoise | 1.0 | ≤0.7 | 前フレーム透過 |
| 11 | negative prompt | 必須入力 | 省略 | 文字幽霊 |
| 12 | seed | 固定値 | -1(都度乱数) | 再現不能 |
| 13 | clip_skip | 1 | 2以上 | プロンプト無視 |
| 14 | batch_size | 1 | ≥2(1280px時) | OOM確定 |
| 15 | sampler | euler_a | deis | 輪郭崩壊 |
| 16 | model dtype | fp16 | fp32 | VRAM消費2倍 |
| 17 | VAE tile size | 512 | 1280(無分割) | OOM |
| 18 | frame_rate(VHS) | 24 | 60 | 出力容量4倍 |
| 19 | loop_count | 0 | ≥1 | 生成時間2倍 |
| 20 | format | h264-mp4 | hevc | ffmpeg再エンコ必須 |

motion_bucket_id は「動きの強さ」制御子で、ドキュメントに推奨範囲の記載がない。64未満は静止画と判別不能なレベルで動きが消え、200超えはカメラが振動しているように見える。127が中央値として機能する。

## Pythonキュー投入スクリプト — 10本分を連続POSTする実装

```python
import json
import uuid
import httpx
from pathlib import Path

COMFY_URL = "http://localhost:8188"
WORKFLOW_PATH = Path("workflow_wan22_vertical.json")

def queue_job(prompt_text: str, seed: int) -> str:
    workflow = json.loads(WORKFLOW_PATH.read_text())
    workflow["prompt"]["1"]["inputs"]["text"] = prompt_text
    workflow["prompt"]["2"]["inputs"]["seed"] = seed
    payload = {
        "prompt": workflow["prompt"],
        "client_id": str(uuid.uuid4()),
    }
    res = httpx.post(f"{COMFY_URL}/prompt", json=payload, timeout=30)
    res.raise_for_status()
    return res.json()["prompt_id"]

PROMPTS = [
    ("a cat chasing sakura petals in Tokyo, cinematic vertical 4k", 1001),
    ("rainy night alley cat staring at camera, neon reflections, slow motion", 1002),
    ("white cat jumping rooftop at sunset, drone shot angle, vertical", 1003),
]

for text, seed in PROMPTS:
    pid = queue_job(text, seed)
    print(f"queued: {pid}")
```

`client_id` を毎回ランダムにすることで、ComfyUI側がジョブを別リクエストとして扱い、キューが積み上がる。同じ `client_id` を使い回すと前のジョブがキャンセルされる罠がある。

## SQLite再開機能 — OOMクラッシュ後に中断ジョブを自動復旧

バッチ中にOOMで落ちた場合、SQLiteのステータス管理で `pending` のジョブだけ再投入する。完了済みジョブは二重実行されない。

```python
import sqlite3, json, httpx
from pathlib import Path

DB = "queue.db"
COMFY_URL = "http://localhost:8188"

def init_db():
    con = sqlite3.connect(DB)
    con.execute("""
        CREATE TABLE IF NOT EXISTS jobs (
            id      INTEGER PRIMARY KEY,
            prompt  TEXT,
            seed    INTEGER,
            status  TEXT DEFAULT 'pending',
            comfy_id TEXT
        )
    """)
    con.commit()
    return con

def add_jobs(con, job_list: list[tuple[str, int]]):
    con.executemany(
        "INSERT OR IGNORE INTO jobs (prompt, seed) VALUES (?, ?)", job_list
    )
    con.commit()

def run_pending(con):
    wf = json.loads(Path("workflow_wan22_vertical.json").read_text())
    rows = con.execute(
        "SELECT id, prompt, seed FROM jobs WHERE status='pending'"
    ).fetchall()
    for jid, prompt, seed in rows:
        wf["prompt"]["1"]["inputs"]["text"] = prompt
        wf["prompt"]["2"]["inputs"]["seed"] = seed
        res = httpx.post(
            f"{COMFY_URL}/prompt",
            json={"prompt": wf["prompt"]},
            timeout=30,
        )
        comfy_id = res.json()["prompt_id"]
        con.execute(
            "UPDATE jobs SET status='queued', comfy_id=? WHERE id=?",
            (comfy_id, jid),
        )
        con.commit()
        print(f"[{jid}] queued → {comfy_id}")

if __name__ == "__main__":
    con = init_db()
    add_jobs(con, [
        ("cat in Tokyo rain, vertical cinematic 4k", 2001),
        ("neon alley cat, slow motion vertical", 2002),
    ])
    run_pending(con)
```

クラッシュ後は `python resume_queue.py` を再実行するだけ。生成完了確認は `/history/{prompt_id}` をポーリングして `status.status_str == "success"` になったタイミングで `status='done'` に更新する。次章ではこのポーリングループとffmpegパイプラインを接続する。
