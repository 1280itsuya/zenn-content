## 第5章　収益を加速させる自動化の最適化と次のステップ

自動化パイプラインを稼働させただけでは、収益は伸び続けない。数字を見て改善し、勝ちパターンを増幅させる「最適化ループ」こそが、月5万円を超えるための鍵だ。

## 5-1　ログを収益に変える：勝ちパターンの抽出

まず、どの記事・どのチャンネルが実際に閲覧・収益を生んでいるかを毎週集計する。Zennであれば公式APIでビュー数を取得できる。

```bash
python src/winner_extractor.py --platform zenn --days 7
```

このスクリプトはビュー上位10記事のタグ・テーマ・文字数を抽出し、`data/winners.json`に保存する。次の生成サイクルでは、このファイルを読み込んで「勝ちテーマ」を優先的に展開する。感覚ではなくデータで手を打つことが、試行錯誤の速度を上げる。

## 5-2　コスト管理：無料枠で最大化する

Claude Code Max（定額）やローカルLLMを使っていても、API呼び出しの無駄は積み重なる。生成ごとにトークン数をログに残し、月次でコスト試算を確認する。

```python
# config/agents.json の budget_alert_yen を設定
"budget_alert_yen": 500
```

閾値を超えた時点でSlackまたはDiscordに通知を飛ばし、翌日以降の生成本数を自動で絞る仕組みを入れておく。収益ゼロの月に無駄なAPI費用だけが積み上がるのを防ぐためだ。

## 5-3　自動化の"詰まり"を先手で潰す

パイプラインが静かにスキップしている状態が最も危険だ。Zennのgit pushが認証でハングしたり、QiitaトークンがRateLimit 429で弾かれたりすることは実際に起きる。毎朝のオーケストレーター実行後に、ヘルスチェックスクリプトを走らせる。

```bash
python src/health_check.py --alert-on-skip
```

前日の投稿件数がゼロのチャンネルを検出したらDiscordに警告を送る。「動いているつもりで止まっている」状態を48時間以内に検知するのが目標だ。

## 5-4　次の収益レールを敷く

既存の自動化が安定したら、新チャンネルを1本ずつ追加する。優先順位の基準は「自分が集客しなくてよいか」だ。プラットフォーム側に検索流入があるZenn・Qiita・Dev.toは最優先。次にBooth・Gumroadの自助販売、そしてクラウドソーシングへの自動出品と続く。

新チャンネルは`.env`に1行追加するだけで有効化できるよう設計しておく。

```
ENABLE_DEVTOOLS_SITE=1
ENABLE_BOOTH_KIT=1
```

一度に全部立ち上げようとすると管理コストが爆発する。月1本のペースで追加し、2週間でビュー・成約が出なければ停止する。この「小さく試して、勝ちを育てる」サイクルが自動化の真の強みだ。

## 5-5　長期視点：自動化を資産にする

個別スクリプトの寄せ集めを「事業」に変えるには、再現性のある記録が必要だ。週次でGitにコミットし、何が変わって何が効いたかをコミットメッセージに残す。それ自体がいずれOSSや教材として売れる資産になる。自動化は一度作れば終わりではなく、改善し続けることで複利的に価値が増す。次の章では、この仕組みをチームや外部委託へ展開する方法を解説する。
