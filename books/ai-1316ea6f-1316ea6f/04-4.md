## 自動化パイプラインは「動かして終わり」ではない

パイプラインを本番稼働させた瞬間、本当の仕事が始まる。スクリプトを書いてcronに登録しただけでは収益は生まれない。運用・計測・最適化の三つのサイクルを回し続けることが、副業を「自動機械」へ育てる唯一の道だ。

## まず計測基盤を先に置く

何も測らずに改善はできない。最初の一週間は収益よりもロギングを優先する。

```python
# ops/logger.py
import json, datetime, pathlib

LOG_PATH = pathlib.Path("data/ops_log.jsonl")

def log_event(channel: str, status: str, views: int = 0, revenue: int = 0):
    entry = {
        "ts": datetime.datetime.now().isoformat(),
        "channel": channel,
        "status": status,
        "views": views,
        "revenue": revenue,
    }
    with LOG_PATH.open("a", encoding="utf-8") as f:
        f.write(json.dumps(entry, ensure_ascii=False) + "\n")
```

各エージェントの末尾でこの関数を呼ぶだけでよい。一週間後には「どのチャンネルが何ビューを稼いでいるか」が一目で分かるようになる。

## 日次レポートで「詰まり」を即発見する

パイプラインは静かにハングする。cronジョブが走ったように見えてもAPIレート制限・認証エラー・ファイルロックで途中停止していることがある。

```bash
# 前日のエラー行をSlackやDiscordへ通知
python ops/daily_report.py --date yesterday
```

`daily_report.py` は `ops_log.jsonl` を読み、`status != "ok"` の行を集計して通知する。筆者の環境では毎朝7時の自動投稿後に7時30分にレポートを走らせ、8時前にスマホで確認するルーティンにした。

## 「稼ぎながら」の意味——Winner Amplification

計測データが溜まり始めたら次のステップへ進む。ビューが伸びた記事のタグ・テーマ・構成をフィードバックループに組み込む。

```python
# ops/winner_extractor.py
import json, pathlib, collections

logs = [json.loads(l) for l in pathlib.Path("data/ops_log.jsonl").read_text().splitlines()]
top = sorted(logs, key=lambda x: x["views"], reverse=True)[:10]
tags = collections.Counter()
for entry in top:
    for t in entry.get("tags", []):
        tags[t] += 1
print(tags.most_common(5))
```

出力例が `[("Python", 8), ("Claude", 6), ("自動化", 5)]` であれば、次の生成プロンプトにその上位タグを必ず含めるよう `config/agents.json` を更新する。負け筋を繰り返す消耗と、勝ち筋を増幅する複利——どちらを選ぶかは計測しているかどうかだけの差だ。

## 最適化は「削る」こと

週次レビューで確認すべき指標は三つだけに絞る。**投稿本数・平均ビュー数・収益**。複雑なダッシュボードは判断を遅らせる。ビュー数が伸びていないチャンネルは即停止し、Claude APIコストを勝ち筋チャンネルへ再配分する。

```bash
# チャンネル別集計（7日間）
python ops/weekly_summary.py --days 7
```

自動化の罠は「動いている=稼いでいる」と錯覚することだ。稼ぎながら改善するとは、毎日ログを読み、毎週一つ止め、毎月一つ伸ばす小さな反復を継続することに尽きる。パイプラインはコードではなく、この反復サイクル自体が資産になる。
