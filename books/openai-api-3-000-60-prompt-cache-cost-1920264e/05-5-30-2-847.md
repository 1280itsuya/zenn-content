## 30日週次チューニング表：cache hit率と1本原価の推移

結論から言うと、cache未設計の初日¥9,200ペースを、30日で月¥2,847・1本平均¥47.5へ着地させた。鍵は「few-shotを削ってヒット率を上げる」週次の意思決定で、感覚ではなく下表の実数で回した。このBookの想定topicsは `openai` / `python` / `automation` / `api` / `cost` の5スラッグで、本章はその `cost` を実測で潰す。

cost_trackerのDBから週次集計したSQLと結果がこれだ。

```sql
SELECT
  CAST((julianday(created_at) - julianday('2026-05-01')) / 7 AS INT) AS week,
  ROUND(AVG(cache_hit_rate), 2)  AS hit_rate,
  ROUND(AVG(cost_jpy), 1)        AS avg_cost,
  ROUND(SUM(cost_jpy), 0)        AS cum_cost
FROM generation_log
GROUP BY week;       -- W1:0.89/¥58/¥812   W2:0.93/¥41/¥1,386
                     -- W3:0.94/¥44/¥2,011 W4:0.93/¥41/¥2,847
```

## 第2週にfew-shotを3→1へ削りヒット率89%→93%

第1週はfew-shot例を3本固定で前置きしていたが、例文がプロンプト全体の62%を占めキャッシュ境界を押し下げていた。例を1本へ削った結果、cacheable prefixが安定し、ヒット率89%→93%、1本¥58→¥41まで下がった。

```python
# before: few-shot 3本（cacheable prefix が毎回ズレる）
SYSTEM = STYLE_GUIDE + EX1 + EX2 + EX3   # 約4,100 tokens

# after: 1本へ圧縮し prefix を固定
SYSTEM = STYLE_GUIDE + EX1               # 約1,500 tokens
messages = [
    {"role": "system", "content": SYSTEM,
     "cache_control": {"type": "ephemeral"}},  # prefix を固定キャッシュ
    {"role": "user", "content": topic_prompt},
]
```

## テンプレ全置換でcache全ミス、1日だけ原価3倍の事故

第3週の火曜にSTYLE_GUIDEを全面改稿したところ、cacheable prefixが1文字目から変わり当日分が全ミス。1本¥41→¥126へ跳ね、その日だけ累計が想定より¥510上振れした。教訓は「prefixは追記のみ・改稿は週初の1回だけ」。

```python
def assert_prefix_stable(new_prefix: str, locked: str) -> None:
    # prefix 変更を検知したら CI で落とす（事故の再発防止）
    if not new_prefix.startswith(locked):
        raise SystemExit(
            f"cache prefix changed: hit率全損リスク 差分={len(new_prefix)-len(locked)}t")
```

cost_trackerのテーブルはこの定義をそのまま流用してよい。

```sql
CREATE TABLE generation_log (
  id INTEGER PRIMARY KEY,
  created_at TEXT NOT NULL,
  prompt_tokens   INTEGER,
  cached_tokens   INTEGER,
  cache_hit_rate  REAL,   -- cached / prompt
  cost_jpy        REAL
);
```

## GPT-4o-miniからGPT-4oへ移す際のコスト試算式

同じ60本をgpt-4o（入力$2.50/Mtok、cache読み出し$1.25/Mtok）へ移すと原価は約7.6倍になる。切替前にこの式で月額を見積もる。1本平均prompt 5,000token・hit率0.93・$1=¥157前提だ。

```python
def monthly_jpy(in_price, cache_price, hit=0.93, tok=5000, n=60, usd=157):
    cached = tok * hit
    fresh  = tok * (1 - hit)
    per = (cached / 1e6 * cache_price + fresh / 1e6 * in_price) * usd
    return round(per * n, 0)

print(monthly_jpy(0.15, 0.075))  # gpt-4o-mini ≈ ¥2,847
print(monthly_jpy(2.50, 1.25))   # gpt-4o      ≈ ¥21,600
```

gpt-4oで品質を底上げするなら、本数を60→20へ絞り1本¥360を許容する判断が現実的だ。hit率を1ポイント上げるより、まずprefixを固定し全ミス事故をゼロにする方が原価への効きが大きい。
