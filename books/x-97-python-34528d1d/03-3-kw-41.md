## 1日300件通知・有用17件：初期KWマッチ実装の崩壊数値

最初に作った通知フィルターは8行のPython関数だった。`["副業", "稼げる", "AI", "フリーランス", "月収"]` のいずれかを含む投稿を全件通知する単純な `in` 演算子チェックだ。

```python
KEYWORDS = ["副業", "稼げる", "AI", "フリーランス", "月収"]

def is_relevant(text: str) -> bool:
    return any(kw in text for kw in KEYWORDS)
```

3日間の実測結果：

| 日付 | 通知件数 | 実際に有用 | 偽陽性率 |
|------|----------|------------|----------|
| 6/1  | 312      | 19         | 93.9%    |
| 6/2  | 287      | 16         | 94.4%    |
| 6/3  | 301      | 18         | 94.0%    |
| 平均 | 300      | **17.7**   | **94.1%** |

94%がノイズ。通知確認の行為そのものが副業の実作業時間を食う、逆説的な状態に陥った。

## 「副業には向かない」が素通りする：正規表現が否定文脈を無視する構造欠陥

偽陽性100件を手動ラベリングした結果、最大カテゴリは**否定文脈**（41件）だった。

```
「副業には向かない人の特徴3選」     → KW「副業」でHIT
「AIを使っても稼げるわけがない現実」 → KW「AI」「稼げる」でHIT
「フリーランスを辞めた理由10選」    → KW「フリーランス」でHIT
```

Pythonの `in` 演算子は文字列の存在だけを確認する。否定助詞（「には」「わけがない」「辞めた」）は完全に無視される。

正規表現で否定パターンを弾こうとすると次のようになる：

```python
import re

NEGATIVE_PATTERNS = [
    r"副業には",
    r"副業をやめ",
    r"稼げるわけがない",
    r"フリーランスを辞め",
]

def has_negative_context(text: str) -> bool:
    return any(re.search(p, text) for p in NEGATIVE_PATTERNS)
```

実測では20パターンを追加しても偽陽性率は94%→79%にしか改善しなかった。否定表現のバリエーションは無限に列挙できるが、それはホワイトリストのメンテナンス地獄を意味する。

## スコアリング関数v1：KW重み×フォロワー係数の設計

二値判定（pass/fail）から**スコア（0.0〜1.0）**に切り替え、閾値以上のみ通知する設計にする。スコアの構成要素は2つ：KWごとの実測ノイズ率から逆算した重みと、フォロワー数を使った信頼係数だ。

```python
from dataclasses import dataclass

KW_WEIGHTS: dict[str, float] = {
    "副業収入": 0.5,    # 高精度KW：否定文脈でほぼ使われない
    "月収公開": 0.4,
    "副業": 0.2,        # 汎用・ノイズ多い
    "稼げる": 0.15,
    "AI": 0.1,
    "フリーランス": 0.1,
}

NEGATIVE_RE = re.compile(
    r"(には向かない|わけがない|をやめ|辞め|失敗|騙|詐欺|稼げない)"
)
NEGATIVE_PENALTY = 0.4

@dataclass
class Post:
    text: str
    follower_count: int

def follower_coefficient(count: int) -> float:
    """フォロワー数を0.8〜1.2の係数に変換（実測ノイズ率から算出）"""
    if count >= 10_000:
        return 1.2
    elif count >= 1_000:
        return 1.0
    return 0.8

def score_v1(post: Post) -> float:
    kw_score = sum(w for kw, w in KW_WEIGHTS.items() if kw in post.text)
    if NEGATIVE_RE.search(post.text):
        kw_score -= NEGATIVE_PENALTY
    return min(max(kw_score * follower_coefficient(post.follower_count), 0.0), 1.0)
```

通知条件: `score_v1(post) >= 0.5`

## 偽陽性41%→22%：v1評価スクリプトと比較数値

手動ラベリング済み100件のデータセットでv1を評価する：

```python
from typing import NamedTuple

class LabeledPost(NamedTuple):
    post: Post
    is_relevant: bool

def evaluate(dataset: list[LabeledPost], threshold: float = 0.5) -> dict:
    tp = fp = tn = fn = 0
    for lp in dataset:
        predicted = score_v1(lp.post) >= threshold
        if predicted and lp.is_relevant:       tp += 1
        elif predicted and not lp.is_relevant:  fp += 1
        elif not predicted and lp.is_relevant:  fn += 1
        else:                                    tn += 1
    precision = tp / (tp + fp) if (tp + fp) else 0
    recall    = tp / (tp + fn) if (tp + fn) else 0
    fpr       = fp / (fp + tn) if (fp + tn) else 0
    return {"precision": round(precision, 2),
            "recall": round(recall, 2),
            "false_positive_rate": round(fpr, 2)}

# 実測結果
# {'precision': 0.69, 'recall': 0.78, 'false_positive_rate': 0.22}
```

| 指標 | v0（単純KW） | v1（スコアリング） |
|------|-------------|-------------------|
| Precision | 0.06 | **0.69** |
| Recall | 1.00 | 0.78 |
| 偽陽性率 | 0.94 | **0.22** |
| 1日あたり通知件数 | 300 | **約67** |

見逃しが22%発生した代わりに、通知件数を300→67まで削減できた。

## v1が取りこぼす「文脈依存ノイズ」：LLMレイヤが必要な4カテゴリ

v1通過後の誤検知をさらに分析した結果、次の4カテゴリはパターンマッチで対処不能とわかった：

| カテゴリ | 件数 | 例 |
|----------|------|----|
| 皮肉・反語 | 9件 | 「副業収入で生活できると本気で思ってる？」 |
| 否定KWなし否定文脈 | 8件 | 「副業月収公開→3,000円でした」 |
| 広告・スパム | 5件 | 「副業収入UP！無料セミナー↓」 |
| KW重複の過剰スコア | 3件 | 「AI副業×AI月収×AI稼げる」 |

「3,000円でした」が失望報告か成功報告かは、文の意味を理解しないと判別できない。正規表現を増やしてもこの壁は越えられない。

v1はLLMスコアリング（第4章）の**前処理レイヤ**として位置づける。1日300件→67件まで削減してからLLMに渡すことで、Claude APIの呼び出しコストを78%カットしながら全体の精度向上を達成する設計になる。
