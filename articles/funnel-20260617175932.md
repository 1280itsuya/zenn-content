# ベクタDBなしで作る最小RAG：自社ドキュメントにClaudeで出典付きQ&A

社内規定・仕様書・マニュアルを対象に「どこに書いてあるか」を即答したい。しかし Pinecone や Chroma の導入・運用コストをかけるほどではない——ドキュメント数が数百件以下ならベクタDBを使わず、**BM25キーワード検索＋Claudeのコンテキスト窓**で十分動く。

## 全体像

```
docs/*.txt
  ↓ 500文字チャンクに分割
BM25インデックス（メモリ上）
  ↓ クエリで上位5件取得
Claude API（検索結果＋質問をプロンプト注入）
  ↓
出典ファイル名付き回答
```

依存ライブラリは `rank-bm25` と `anthropic` のみ。

## 実装

### 1. チャンク分割

```python
def chunk_text(text: str, source: str, size: int = 500, overlap: int = 100):
    chunks, start = [], 0
    while start < len(text):
        chunks.append({"text": text[start:start + size], "source": source})
        start += size - overlap
    return chunks
```

### 2. BM25インデックス構築

```python
from rank_bm25 import BM25Okapi

def build_index(chunks):
    tokenized = [c["text"].split() for c in chunks]
    return BM25Okapi(tokenized), chunks
```

日本語の場合は `split()` を `fugashi`（MeCab バインディング）に差し替える。`pip install fugashi unidic-lite` で導入でき、形態素単位の検索精度が上がる。

### 3. 検索＋回答生成

```python
import anthropic

def answer(query: str, bm25, chunks, top_k: int = 5) -> str:
    scores = bm25.get_scores(query.split())
    top_idx = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:top_k]
    context = "\n\n".join(
        f"[出典: {chunks[i]['source']}]\n{chunks[i]['text']}"
        for i in top_idx
    )
    client = anthropic.Anthropic()
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": (
            "以下のドキュメント抜粋のみを根拠に質問へ答えてください。"
            "記載がなければ「情報なし」と答え、回答末尾に使用した出典を列挙してください。\n\n"
            f"{context}\n\n質問: {query}"
        )}]
    )
    return msg.content[0].text
```

### 4. 実行

```python
import pathlib

chunks = []
for p in pathlib.Path("docs").glob("**/*.txt"):
    chunks.extend(chunk_text(p.read_text(encoding="utf-8"), source=p.name))

bm25, chunks = build_index(chunks)
print(answer("有給休暇の申請期限は？", bm25, chunks))
```

実際の社内規定を入力した場合の出力イメージ：

```
有給休暇は取得希望日の3営業日前までに申請が必要です。

【出典】
- hr_policy_2025.txt
```

## つまずきポイント

**日本語の検索精度**  
`split()` は半角スペース区切りのため日本語でほぼ機能しない。`fugashi` が難しければ、文字列を3〜4文字ずつずらす文字 n-gram でも BM25 は動く。精度は落ちるが依存ゼロで試せる。

**チャンクサイズとコスト**  
Claude Sonnet-4 のコンテキスト窓は広いが、`top_k × size` がトークン換算で大きいほど費用が増える。`size=500文字 / top_k=5` を起点に、回答品質を見ながら調整する。

**ハルシネーション**  
プロンプトに「記載がなければ情報なし」を明示するだけで根拠のない回答は大幅に減る。さらに `scores[top_idx[0]] < 閾値` のときは Claude を呼ばずに「該当文書なし」を返す実装を加えると安全性が上がる。

## まとめ

数百件規模の社内ドキュメントへの Q&A であれば、ベクタDB不要・ライブラリ最小構成でも十分実用になる。まず動くものを作り、精度が物足りなくなった段階で埋め込みベクタ化へ移行する段階的アプローチが現実的だ。
