## なぜVPSなのか

### 要件を整理する

自動化botに必要なものはシンプルだ。

- **常時起動**：cronやsystemdで定時実行する
- **外部API呼び出し**：TwitterAPI・OpenAI・スクレイピング等
- **ファイル永続化**：ログ・DBの保存
- **SSH接続**：デプロイ・デバッグのため

共有レンタルサーバー（ロリポップ・さくらのレンタルサーバーなど）はSSHが制限されていたり、常時起動プロセスを禁止していたりするため、botには向かない。Lambdaなどのサーバーレスは15分タイムアウトがあり、長時間ジョブには使えない。VPSはこれらの制約がなく、自由度が高い。

## サーバーを選ぶ際の実務チェックリスト

### 1. OSはUbuntu LTS一択でいい

CentOS 7はすでにEOL。RedHat系に強い理由がなければ**Ubuntu 22.04 LTS**を選ぶ。Pythonの新しいバージョンが素直に入り、apt/pip周りの情報量が最多だ。

### 2. SSHキー認証を初日に設定する

VPSを起動したら最初にやることは以下の3つだ。

```bash
# ローカルでキー生成（既にあるなら不要）
ssh-keygen -t ed25519 -C "your@email.com"

# サーバー側に公開鍵を登録
ssh-copy-id -i ~/.ssh/id_ed25519.pub root@<サーバーIP>

# パスワード認証を無効化
sudo sed -i 's/^PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

パスワード認証を残したままにするとブルートフォース攻撃を常時受ける。`/var/log/auth.log`を見れば、世界中から毎分試されているのがわかるはずだ。

### 3. ポートを変える・UFWを入れる

デフォルトの22番ポートはスキャンされ続ける。SSHポートを変更し、UFWで必要なポートのみ開けておく。

```bash
sudo ufw allow 2222/tcp   # 変更後のSSHポート例
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

### 4. cronよりsystemd timerを使う

cronは手軽だが、実行ログがsyslogに散らばり、失敗しても気づきにくい。systemd timerを使うと`journalctl -u my-bot.service`で実行履歴が一元管理できる。

```ini
# /etc/systemd/system/my-bot.timer
[Unit]
Description=Run my bot every hour

[Timer]
OnCalendar=*:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

### 5. ドメインとDNSは後から足せる

botだけなら最初はIPアドレス直打ちで十分だ。ダッシュボードや通知エンドポイントを公開するときに、お名前.comなどでドメインを取得し、VPSのIPにAレコードを向ければいい。TTLは最初300秒（5分）に設定しておくと、IP変更時の切り替えが速い。

## よくある落とし穴

**メモリ不足で無音死亡する**：512MBプランでChromeのヘッドレスブラウザを動かすと、OOM killerにプロセスを殺される。スクレイピングにはPlaywrightより`requests`+`httpx`で済む設計を先に検討する。

**ディスクI/Oがボトルネックになる**：SQLiteへの頻繁な書き込みはVPSのディスクスペックに依存する。大量ログは書き出し頻度を落とすか、外部ストレージ（S3互換等）に逃がす。

**再起動後にbotが起動しない**：systemdのenable忘れが原因の大半だ。

```bash
sudo systemctl enable my-bot.timer
```

これを忘れると月1回のOSアップデート後に静かに止まる。
