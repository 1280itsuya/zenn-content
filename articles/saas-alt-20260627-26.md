## 結論（おすすめ1つ）

**Bruno** への移行を迷わず勧める。

コレクションをローカルのテキストファイル（`.bru` 形式）として保存するため、Git で差分管理・PR レビューが自然に行える。クラウドアカウント不要でネットワーク制約やプロキシ設定の影響を受けず、Postman の JSON エクスポートをそのままインポートできるため移行作業の大半は数ステップで完結する。チームの規模を問わず、「コレクションをコードと同じリポジトリに置く」という運用が即座に実現できる点が他ツールとの決定的な差だ。

---

## 比較表（料金/無料枠/移行コスト/対応言語）

| ツール | 料金・無料枠 | 移行コスト | 対応プロトコル | Git 連携 |
|---|---|---|---|---|
| **Bruno** | OSS・完全無料 | 低（Postman JSON インポート対応） | REST / GraphQL / gRPC | ネイティブ（.bru ファイル群） |
| **Hoppscotch** | 無料枠あり。チームプランは公式の料金ページで要確認 | 中（インポートあり、一部手動修正が発生することがある） | REST / GraphQL / WebSocket / SSE | 公式の料金ページで要確認 |
| **Insomnia** | 無料枠あり。有料プランは公式の料金ページで要確認 | 低（Postman インポート対応。操作感も近い） | REST / GraphQL / gRPC / WebSocket | プラグイン経由 |
| **HTTPie CLI** | OSS CLI は無料。GUI（Desktop / Cloud）は公式の料金ページで要確認 | 高（GUI 操作から CLI 操作モデルへの切り替えが必要） | REST / HTTP | 設定ファイルをリポジトリに含める運用が可能 |

## 移行手順

Bruno を例に、Postman からの移行をそのまま実行できる手順を示す。

**Step 1: Bruno のインストール**

```bash
# macOS (Homebrew)
brew install --cask bruno

# Windows (Scoop)
scoop install bruno

# Linux: 公式サイトから .deb または .AppImage を取得してインストール
```

**Step 2: Postman コレクションのエクスポート**

Postman で対象コレクションを右クリック → 「Export」→ `Collection v2.1` 形式の JSON ファイルを書き出す。Environment も同様に「Export」できる。

**Step 3: Bruno へインポート**

```
Bruno を起動
→ 「Import Collection」→ エクスポートした JSON ファイルを選択
```

インポート後、コレクションは `~/bruno/<コレクション名>/` 配下に `.bru` ファイル群として展開される。

**Step 4: 環境変数の設定**

`environments/` ディレクトリに `.bru` 形式のファイルを作成する。機密値は直接書かず、シェル環境変数で注入する。

```ini
# environments/local.bru
vars {
  base_url: http://localhost:3000
  api_key: {{process.env.MY_API_KEY}}
}
```

**Step 5: Git リポジトリに追加してチームと共有**

```bash
cd ~/bruno/my-api-collection

git init
echo "*.env" >> .gitignore

git add .
git commit -m "chore: migrate API collection from Postman to Bruno"

git remote add origin git@github.com:<org>/<repo>.git
git push -u origin main
```

以降はチームメンバーが `git clone` するだけで同一のコレクションを使い始められる。環境変数はローカルの `.env` または CI の Secrets に分離しておくことで、機密情報をリポジトリに含めずに済む。

## 向き不向き

**Bruno が向くケース**

- API コレクションをソースコードと同じリポジトリで管理し、変更を PR レビューしたいチーム
- クラウドへのデータ送信を禁じるセキュリティポリシーがある組織
- メンバー全員が Git に慣れており、コレクションのバージョン履歴を重視する場合
- Postman の無料プランでコレクション共有の制限にぶつかった小規模チームや個人開発者

**Hoppscotch が向くケース**

- インストール不可の環境でブラウザのみ使えるケース
- WebSocket や SSE など双方向通信も同一ツールでテストしたい場合
- セルフホスト（OSS 版 Docker イメージあり）でデータを社内に完全に閉じたい組織

**Insomnia が向くケース**

- gRPC や GraphQL を日常的に使うチームで GUI の操作感を維持したい場合
- Postman に近い UX のままツールを変えたい、移行コスト最小を優先するチーム

**HTTPie CLI が向くケース**

- ターミナル中心のワークフローで、シェルスクリプトや CI パイプラインに API テストを組み込むことが主目的の場合
- GUI を一切使わず、コマンド1行でレスポンスを確認したい[エンジニア](https://www.amazon.co.jp/s?k=%E3%82%A8%E3%83%B3%E3%82%B8%E3%83%8B%E3%82%A2%20%E6%9C%AC&tag=1280itsuya22-22)個人

**避けるべきケース**

- Bruno: コレクションを非エンジニア（QA・デザイナー・営業）と共有する必要があるチーム。`.bru` ファイルはテキストで読めるが、操作には一定のリテラシーが要る
- HTTPie CLI: GUI に慣れたメンバーが多い混成チームへの一斉移行は摩擦が大きく、習熟コストが移行メリットを上回ることがある
- Hoppscotch クラウド版: 本番環境の機密 API キーを扱うリクエストはセルフホスト版か別ツールで行うことを検討すること
