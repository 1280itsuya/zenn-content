## 結論から言う

**月額1,000〜2,000円台のVPS一択。** 共用レンタルサーバーや無料枠クラウドは「常時起動」「外部API叩き放題」「cronの自由度」の三点でbotには向かない。まずVPSを契約し、SSHで入ってsystemdかcronで動かす。これが最短で最も安定する構成だ。

---

## なぜ共用サーバーはダメか

共用サーバー(いわゆるレンタルサーバー)はWebサイト向けに最適化されている。以下の制約がbotには致命的になる。

- **cronの実行間隔が5〜15分に制限**されているサービスが多い
- **外向きのHTTP/TCPが絞られる**場合があり、外部APIへの接続が不安定
- 常時起動のプロセスを立ち上げると規約違反になるケースがある
- [Python](https://www.amazon.co.jp/s?k=Python%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22)のバージョンやパッケージを自由に入れられない

無料枠のクラウド(AWS Lambda無料枠など)はイベント駆動には向くが、ステートフルなbotや「起動したままAPIをポーリングする」用途には設計が合わない。

## 実際のセットアップ手順

### 1. VPSを契約してSSHで接続する

VPS契約直後はrootのパスワード認証が有効になっていることが多い。最初にやることは**パスワード認証の無効化とSSH鍵認証への切り替え**だ。

```bash
# ローカルで鍵ペアを生成
ssh-keygen -t ed25519 -C "bot-server"

# 公開鍵をVPSに転送
ssh-copy-id -i ~/.ssh/id_ed25519.pub root@<VPSのIPアドレス>
```

`/etc/ssh/sshd_config` で以下を設定してサービスを再起動する。

```
PasswordAuthentication no
PermitRootLogin prohibit-password
```

### 2. botをsystemdサービスとして登録する

cronよりもsystemdのほうがログ管理・自動再起動・依存関係の制御がしやすい。

```ini
# /etc/systemd/system/mybot.service
[Unit]
Description=My Automation Bot
After=network.target

[Service]
User=botuser
WorkingDirectory=/home/botuser/mybot
ExecStart=/home/botuser/mybot/.venv/bin/python main.py
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
systemctl enable mybot
systemctl start mybot
journalctl -u mybot -f   # ログ確認
```

`Restart=on-failure` を設定しておくと、例外でクラッシュしても自動で復帰する。

### 3. UFWでポートを絞る

botがAPIを叩くだけで外部から接続を受け付けない場合でも、ファイアウォールは設定しておく。

```bash
ufw default deny incoming
ufw allow ssh
ufw enable
```

Webhookを受け取る場合はそのポートだけ追加で開ける。

## よくあるハマりポイント

**タイムゾーン**: VPSのデフォルトはUTCのことが多い。cronで「毎朝7時」と設定したのにJSTで動かないのはこれが原因だ。`timedatectl set-timezone Asia/Tokyo` で揃えておく。

**メモリ不足でbotが落ちる**: 最小プランは512MB〜1GBが多い。Pythonのbotで複数ライブラリを読み込むと意外と食う。スワップを設定するか、最初から1GBプランを選んだほうが安定する。

**IP制限**: 一部の外部APIはIP単位でレート制限をかけてくる。VPSの固定IPを取得してAPIのホワイトリストに登録しておくと安定する。
