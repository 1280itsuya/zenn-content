## 発生条件

- `git pull` や `git merge` 時、**ローカルリポジトリとリモートリポジトリに共通の祖先コミットが存在しない**場合に発生する
- `git init` で初期化したローカルリポジトリに、既存コミットを持つ GitHub リポジトリをリモートとして追加し、そのまま `git pull` を実行したとき
- 別々に育てた2つのリポジトリを `git remote add` で結合しようとしたとき

## 原因

最も典型的なケースは、ローカルで `git init` → コミット → GitHub に空でないリポジトリを作成 → `git pull origin main` という手順を踏んだときだ。

```bash
# ローカル側
mkdir myproject && cd myproject
git init
echo "hello" > README.md
git add . && git commit -m "first commit"

# リモート側（GitHub 上で README.md 付きで作成済み）
git remote add origin https://github.com/yourname/myproject.git

# ここで fatal が発生する
git pull origin main
# fatal: refusing to merge unrelated histories
```

`fatal: refusing to merge unrelated histories` というメッセージは、Git が「この2つのブランチには共通の祖先が一切ない、意図しないマージかもしれない」と判断し、安全のために拒否していることを意味する。Git 2.9 以降、このチェックがデフォルトで有効になっている。

## 修正方法

`--allow-unrelated-histories` オプションを付けることで、Git に「意図的なマージである」と明示できる。

```bash
git pull origin main --allow-unrelated-histories
```

実行するとエディタが開いてマージコミットメッセージの入力を求められる。内容を確認して保存すれば、2つの履歴が1本に統合される。

競合が発生した場合は通常の `git merge` と同様に解消する。

```bash
# 競合ファイルを確認
git status

# 競合を手動で編集後
git add <conflicted-file>
git commit
```

**注意点として**、このオプションは「関係ない歴史を持つブランチを意図的に合流させる」ための例外措置だ。本来は同一プロジェクトを別々に育ててしまった設計ミスを後から修正する用途に使う。たとえば、以下のように最初からリモートをクローンして作業を始めれば、この問題は原則として発生しない。

```bash
# 最初からクローンして始める（推奨）
git clone https://github.com/yourname/myproject.git
cd myproject
# → 以降は git pull で fatal が出ない
```

`--allow-unrelated-histories` を使った後は `git log --oneline --graph` で履歴が正しく統合されているかを確認しておくとよい。
