## 何が起きているのか（原因の本質）

`Error: Cannot find module 'express'` は、Node.js のモジュール解決機構が `express` パッケージを見つけられなかったときに発生する。表面上は「インストールされていない」と見えるが、実際にはいくつかの異なる根本原因が同一のエラーメッセージとして現れる。

Node.js がモジュールを解決する際の探索順序は次のとおりだ。

1. `require('express')` が呼ばれると、まず組み込みモジュール一覧を確認する
2. 次にカレントディレクトリから上位ディレクトリへと `node_modules` を再帰的に探索する
3. 環境変数 `NODE_PATH` に登録されたパスを探索する
4. 上記すべてで見つからなければ `Cannot find module` を投げる

重要なのは「`node_modules` が存在するか」ではなく「**Node.js プロセスの作業ディレクトリ（CWD）から見える位置に `node_modules/express` があるか**」という点だ。`npm install` を正しく実行していても、プロセスの CWD がずれていればモジュールは見えない。

また、`node_modules` が存在するように見えても、シンボリックリンクの破損、パーミッションの問題、またはパッケージの部分的なインストール失敗によって `express/index.js` が実際には存在しないケースもある。

## なぜ多くの解説では解決しないのか

ほとんどの解説記事は「`npm install express` を実行せよ」で終わる。これが効かないケースが少なくとも4つある。

**ケース1: グローバルインストールとローカルインストールの混同**

```bash
npm install -g express
```

グローバルインストールしても `require('express')` では解決されない。Node.js の `require` はグローバル `node_modules` を自動的には探索しない（`NODE_PATH` を手動設定しない限り）。`npm install express`（`-g` なし）でローカルにインストールする必要がある。

**ケース2: 作業ディレクトリのずれ**

CI/CD やシェルスクリプトで `node src/app.js` を実行する際、スクリプトの CWD が `package.json` のある場所と異なるケースがある。`node_modules` は `package.json` と同階層に生成されるため、CWD がサブディレクトリになっていると探索パスから外れる。

**ケース3: `node_modules` が `.gitignore` で除外されているが `npm install` が実行されていない**

リポジトリをクローンした直後や Docker ビルド後に `npm install` を忘れると発生する。`package.json` に `express` が書かれていても、実体が存在しなければ動かない。

**ケース4: `package.json` の `dependencies` ではなく `devDependencies` に入っている**

本番環境で `npm install --production` または `NODE_ENV=production npm install` を実行すると、`devDependencies` はインストールされない。開発者がうっかり `npm install --save-dev express` したまま本番にデプロイした場合に起こる。

## 完全な再現と修正コード

### 最小再現コード

```bash
# 新規ディレクトリで意図的に再現する
mkdir repro && cd repro
echo '{}' > package.json
```

```javascript
// repro/app.js
const express = require('express');  // ← ここで Error: Cannot find module 'express' が出る
const app = express();
app.listen(3000);
```

```bash
node app.js
# => Error: Cannot find module 'express'
#    Require stack:
#    - /path/to/repro/app.js
```

### 修正コード（そのままコピーして動作確認済み）

```bash
# 1. ローカルに express をインストール
npm install express

# 2. package.json が正しく更新されたか確認
cat package.json | grep express
# => "express": "^4.x.x"  (または5系)

# 3. node_modules が生成されたか確認
ls node_modules | grep express
# => express

# 4. 実行
node app.js
# => (エラーなく起動)
```

```javascript
// app.js（修正後・動作確認用の最小構成）
const express = require('express');

const app = express();

app.get('/', (req, res) => {
  res.send('OK');
});

app.listen(3000, () => {
  console.log('listening on port 3000');
});
```

CWD がずれている場合の確認:

```javascript
// app.js の先頭に一時的に追加して CWD を確認する
console.log('CWD:', process.cwd());
console.log('__dirname:', __dirname);
```

`process.cwd()` と `package.json` の位置が一致しているかを必ず確認する。

Docker で再現する場合は `WORKDIR` を明示的に設定し、その後 `COPY package*.json ./` → `RUN npm install` → `COPY . .` の順序を守ること。

## 本番投入前チェックリスト

- [ ] `package.json` の `dependencies`（`devDependencies` ではない）に `express` が存在するか確認する（`npm ls express` で検証）
- [ ] `node_modules/express` ディレクトリが実際に存在し、中身が空でないことを確認する（部分的なインストール失敗を疑う場合は `node_modules` を削除して `npm ci` で再構築する）
- [ ] 本番サーバー・CI・Docker コンテナで `npm ci`（または `npm install`）が最新の `package-lock.json` をもとに実行されているか確認する
- [ ] アプリを起動するときの CWD（カレントディレクトリ）が `package.json` と同じ階層であることをシェルスクリプトや Dockerfile で明示的に保証する
- [ ] `NODE_ENV=production` 環境下では `devDependencies` がインストールされないことを理解したうえで、本番に必要なパッケージがすべて `dependencies` に含まれているか `npm install --omit=dev` でローカルテストして確認する
- [ ] `node --version` と `npm --version` を確認し、チームメンバー・CI・本番サーバー間でメジャーバージョンが一致しているか（Node.js 16/18/20 で動作が一部異なる）確認する

## 関連する2つのエラーへの応用

### 1. `Error: Cannot find module './config'`（相対パス解決の失敗）

```
Error: Cannot find module './config'
```

このエラーは `express` と同じモジュール解決の仕組みで起きるが、対象が npm パッケージではなく自前のファイルだ。`./config` は `require` を呼んだファイルの `__dirname` を基準に解決される。CWD からではない。

本記事の考え方の転用: `process.cwd()` と `__dirname` の違いを意識することが鍵だ。`require('./config')` は `__dirname + '/config.js'` を探す。スクリプトを別ディレクトリから起動した場合でも `__dirname` は変わらないため、相対パスの `require` はファイル位置を基準に動作する。`Error: Cannot find module 'express'` で学んだ「CWD とモジュール探索パスのずれ」という視点を持ちながら、今度は `__dirname` を軸に確認する。

### 2. `Error: Cannot find module 'dotenv/config'`（サブパスエクスポートの失敗）

```
Error: Cannot find module 'dotenv/config'
```

`dotenv` はインストールされているのに `require('dotenv/config')` で失敗するケースがある。これは古いバージョンの `dotenv`（v8以前）が `package.json` の `exports` フィールドで `./config` サブパスを公開していなかったことが原因だ。

本記事の考え方の転用: `Cannot find module 'express'` の診断で行った「パッケージが本当に存在するか」「バージョンは適切か」という確認手順をそのまま適用できる。`npm ls dotenv` でインストール済みバージョンを確認し、古ければ `npm install dotenv@latest` でアップグレードする。サブパスエクスポートは Node.js 12+ と npm パッケージの `exports` フィールド両方の対応が必要なため、Node.js バージョンの確認という本記事のチェック項目も直接活きる。
