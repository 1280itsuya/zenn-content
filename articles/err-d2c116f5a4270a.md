## 発生条件

- スクリプトや設定ファイルに記述したパスが実際のファイルシステム上の配置と一致しないとき
- シェルスクリプトを Windows 環境で編集して Linux/Mac で実行すると、shebang 行末の `\r` が原因で `No such file or directory` が出る
- 環境変数やコマンド引数で渡されたパスに意図しないスペース・タイポ・大文字小文字の差異があるとき

## 原因

典型例は次の 2 パターンです。

**パターン 1 — 単純なパス誤り**

```bash
#!/bin/bash
cat /home/user/config/setting.yml   # 実際は /home/user/configs/setting.yml
```

```
cat: /home/user/config/setting.yml: No such file or directory
```

ディレクトリ名のスペルミスや、ステージング環境と本番環境でディレクトリ構成が異なることで起きます。

**パターン 2 — CRLF 改行による shebang 壊れ**

Windows の Git や VSCode で改行コードを `CRLF` のまま保存したスクリプトを Linux で実行すると、shebang の `#!/bin/bash` が `#!/bin/bash\r` として解釈され、`/bin/bash\r` というパスを探しに行きます。

```bash
# file: deploy.sh (CRLF で保存されている)
#!/bin/bash
echo "deploy start"
```

```
bash: /usr/bin/env: 'bash\r': No such file or directory
```

`file` コマンドで確認すると改行コードが露見します。

```bash
file deploy.sh
# deploy.sh: POSIX shell script, ASCII text executable, with CRLF line terminators
```

## 修正方法

**パターン 1 — パスを確認して修正する**

```bash
# 実在するパスを ls で確認
ls /home/user/

# 正しいパスに修正
cat /home/user/configs/setting.yml
```

変数経由でパスを渡している場合は `set -x` でデバッグ出力を有効にすると展開後の実際の値を確認できます。

```bash
#!/bin/bash
set -x
CONFIG_DIR="/home/user/configs"
cat "${CONFIG_DIR}/setting.yml"
```

**パターン 2 — CRLF を LF に変換する**

```bash
# dos2unix が使える環境
dos2unix deploy.sh

# dos2unix がない場合は sed で代替
sed -i 's/\r//' deploy.sh

# 変換後に確認
file deploy.sh
# deploy.sh: POSIX shell script, ASCII text executable
```

Git リポジトリで管理している場合は `.gitattributes` を追加してリポジトリ側で改行コードを制御するのが確実です。

```gitattributes
# .gitattributes
*.sh text eol=lf
```

既存ファイルへの適用は次の手順で行います。

```bash
git add --renormalize .
git commit -m "fix: normalize line endings to LF"
```

---

`No such file or directory` はエラーメッセージそのままの通り「指定されたパスが見つからない」という事実だけを示しており、原因はパスのスペルミスから CRLF 混入まで幅広い。`file` コマンドや `set -x` でまず実態を確認し、パス誤りか改行コード問題かを切り分けることが解決の最短ルートです。
