## 発生条件

- サーバーサイドレンダリング（SSR）時に `window` や `localStorage` など**ブラウザ専用API**をコンポーネントのトップレベルで参照したとき
- `Math.random()` や `new Date()` のように**実行タイミングで値が変わる式**をSSR中に使ったとき
- `typeof window !== 'undefined'` による条件分岐をレンダリングロジックの中に直接書いたとき

## 原因

`Error: Hydration failed because the initial UI does not match` は、[React](https://www.amazon.co.jp/s?k=React%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22)がサーバーで生成したHTMLとクライアントで初回レンダリングした仮想DOMを突合したときに、内容が一致しないと投げるエラーです。

典型的な再現コードを示します。

```tsx
// NG: サーバーとクライアントで window.innerWidth の値が異なる
export default function Banner() {
  const width = typeof window !== 'undefined' ? window.innerWidth : 0;

  return <p>画面幅: {width}px</p>;
}
```

サーバー実行時は `window` が存在しないため `0` が埋め込まれたHTMLが返ります。クライアントは実際の `window.innerWidth`（例: `1440`）でレンダリングしようとするため、DOMが一致せずハイドレーションが失敗します。`Date.now()` を直接JSXに書いた場合も同じ理由で再現します。

## 修正方法

### パターン1: `useEffect` でブラウザAPIを後処理する

```tsx
import { useState, useEffect } from 'react';

export default function Banner() {
  const [width, setWidth] = useState(0); // 初期値はサーバーと同じ値にする

  useEffect(() => {
    // useEffect はクライアント側でしか実行されないため安全
    setWidth(window.innerWidth);
  }, []);

  return <p>画面幅: {width}px</p>;
}
```

`useState` の初期値をサーバーで出力される値（ここでは `0`）に揃えておくのがポイントです。`useEffect` はハイドレーション完了後に実行されるため、`Error: Hydration failed because the initial UI does not match` は発生しません。

### パターン2: `next/dynamic` で SSR を無効化する

コンポーネント全体がブラウザ依存で、サーバーでのレンダリング自体が不要な場合はこちらが簡潔です。

```tsx
// components/ClientOnlyWidget.tsx
export default function ClientOnlyWidget() {
  return <p>現在時刻: {new Date().toLocaleTimeString()}</p>;
}
```

```tsx
// pages/index.tsx または app/page.tsx
import dynamic from 'next/dynamic';

const ClientOnlyWidget = dynamic(
  () => import('../components/ClientOnlyWidget'),
  { ssr: false } // サーバーレンダリングをスキップ
);

export default function Home() {
  return <ClientOnlyWidget />;
}
```

`ssr: false` を指定すると対象コンポーネントはサーバーでレンダリングされず、クライアントのみで描画されます。そのためサーバーとクライアントの不一致が原理的に起きません。ただし初回表示時にコンポーネントが遅れて現れるため、SEOや初期描画が重要な箇所への適用は慎重に判断してください。
