# 1コードでWan/SDXL/Flux切替：失敗7回でたどり着いた設計

結論から言う。モデルごとにスクリプトを分けるな。

自動生成パイプラインを組んでいると、必ずこの問題にぶつかる。今月はSDXL、来月はFlux.1-dev、依頼によってはWan2.1——という具合にモデルが切り替わるたびに `generate_sdxl.py`、`generate_flux.py` と増殖し、3ヶ月後には誰も触れないコードの墓場ができあがる。

私が7回失敗した記録だ。

**失敗1〜3回目：** プロバイダ別クラスを作ったが、プロンプト最適化ロジックが各クラスに散らばり、SDXL用に加えた「ネガティブプロンプト自動付与」がFlux側に反映されず2日気づかなかった。生成コスト試算が`$0.04/枚`ずれたまま本番稼働した。

**失敗4〜5回目：** 設定ファイルで切り替えを試みた。`config.yaml`のモデル名を変えるだけで動くはずが、Wan2.1の解像度制約（縦横比16:9固定、短辺≥480px）をSDXL向けコードが無視してサイレントに歪み画像を吐き続けた。1,200枚生成後に発覚。

**失敗6〜7回目：** `if model == "flux"` の分岐をメイン処理に直書きした。当然、Wan2.2が出たタイミングで全部書き直しになった。

---

たどり着いた設計は **Adapter + Validator の2層** だ。

```python
# 呼び出し側はこれだけ
result = generate(prompt, model="flux-dev", size=(1024, 1024))
```

内部でAdapterがモデル固有の制約（FLUXの`guidance_scale=0`問題、Wanの縦横比スナップ、SDXLのnegative prompt必須）を吸収し、Validatorが出力サイズ・ファイルサイズ・生成時間を事後検証してSlackに異常通知を飛ばす。

この構成に変えてから、モデル追加コストが平均 **4時間→45分** に落ちた。月次の生成失敗率も **11% → 1.3%** まで下がり、再生成コストで月 **$23** 節約できている。
