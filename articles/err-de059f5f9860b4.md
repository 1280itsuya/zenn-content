## 発生条件

- `webpack` や `vite build`、`next build` など大規模なバンドル処理を実行したとき
- `ts-node` や `ts-jest` で型定義が多いプロジェクトをフルコンパイルしたとき
- CI/CD 環境（GitHub Actions, Docker コンテナ等）のようにデフォルトメモリが少ないホストでビルドを走らせたとき

## 原因

Node.js の V8 エンジンにはヒープメモリの上限がデフォルトで設定されており（環境により 512MB〜1.5GB 程度）、それを超えると `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` が発生してプロセスが強制終了する。

以下は最小再現例。巨大な配列を一括生成すると上限に達する。

```js
// heap-overflow.js
const data = [];
while (true) {
  // 100万要素のバッファを際限なく積み上げる
  data.push(Buffer.alloc(1024 * 1024));
}
```

```bash
node heap-overflow.js
# => FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory
```

ビルドツールの場合は単一ファイルではなくモジュールグラフ全体をメモリ上に展開するため、プロジェクトが大きくなるほど同じエラーに遭遇しやすくなる。

## 修正方法

`NODE_OPTIONS` 環境変数で `--max-old-space-size` を指定し、V8 の旧世代ヒープ上限を引き上げる。単位は MB。

```bash
# シェルで一時的に設定してビルドを実行する例
export NODE_OPTIONS="--max-old-space-size=4096"
npm run build
```

```json
// package.json の scripts に直接埋め込む場合
{
  "scripts": {
    "build": "NODE_OPTIONS=--max-old-space-size=4096 webpack --config webpack.config.js"
  }
}
```

```bash
# .env や CI の環境変数として設定する場合（GitHub Actions 等）
NODE_OPTIONS=--max-old-space-size=4096
```

値の目安はホストの物理メモリの 50〜75% 程度に留める。たとえば 8GB マシンなら `4096`、16GB なら `8192` を上限として試す。無制限に増やすとホスト全体のスワップを圧迫するため注意。

また、`--max-old-space-size` はあくまで上限の緩和であり、根本的なメモリリークやビルド設定の問題を隠蔽することがある。エラーが繰り返し発生する場合は、`--inspect` フラグとヒープスナップショット（`v8.writeHeapSnapshot()`）を組み合わせてメモリ使用量の内訳を確認し、不要なモジュールのバンドル除外や `splitChunks` 設定の見直しも検討する。
