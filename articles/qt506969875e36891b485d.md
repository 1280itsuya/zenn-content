RTX 5090（VRAM 32GB）とWan2.2のローカル生成だけで、API課金0円のまま縦動画（720×1280 / 5秒）を1本あたり実測4〜6分で吐き出し、cronで毎朝7時に3本自動生成 → YouTube Shortsへ予約アップロードする所までをこの記事で動かせるようにする。Kling APIの1本¥30〜60を払い続けるのが嫌になった人向けの、私の3週間ぶんの失敗ログ込みの実装だ。

## 結論：A14Bは5090で罠、5B(TI2V)を使えば0円で回る

結論から言うと、Wan2.2の最上位 `Wan2.2-T2V-A14B` は VRAM 32GB の RTX 5090 では実用にならなかった。offloadを有効にすると生成1step目で必ず segfault で落ちる（後述の実測ログあり）。知らないと損するポイントはここで、SNSで「5090なら14B余裕」と書かれているのを真に受けて2日溶かした。

実際に毎日量産が回ったのは軽量版の `Wan2.2-TI2V-5B`。720p・5秒・24fpsで生成し、1本の実測値は次の通り。

- 生成時間: 4分12秒〜6分08秒（steps=30, cfg=5.0）
- VRAM使用: ピーク18.6GB（offloadなしで収まる）
- 電気代: 1本あたり約¥1.8（450W×6分×¥28/kWh換算）
- API比較: Kling I2V APIなら同尺で¥30〜60 → 1日3本×30日で月¥2,700〜5,400の[節約](https://hb.afl.rakuten.co.jp/ichiba/5429ef72.6d629ef3.5429ef73.a7c64ecd/?pc=https%3A%2F%2Fsearch.rakuten.co.jp%2Fsearch%2Fmall%2F%25E7%25AF%2580%25E7%25B4%2584%2520%25E3%2582%25B0%25E3%2583%2583%25E3%2582%25BA%2F)

つまり5Bに割り切れば、初期のGPU代を除けば限りなく0円に近い。14Bの画質に未練を残してoffloadで粘るのが一番の損だ。

## RTX5090 + PyTorch 2.6 + CUDA 12.8 環境構築（Blackwell sm_120対応）

5090（Blackwell, compute capability sm_120）は古いPyTorchホイールでは認識されない。`torch.cuda.is_available()` がTrueでも生成時に `no kernel image is available` で落ちるので、CUDA 12.8対応のnightly〜2.6系を入れるのが命綱…ではなく必須条件だ。

```bash
# [Python](https://www.amazon.co.jp/s?k=Python%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22) 3.11 の venv を作る（3.12はxformersでハマるので避ける）
python3.11 -m venv .venv && source .venv/bin/activate

# Blackwell(sm_120)対応の PyTorch 2.6 + CUDA 12.8
pip install torch==2.6.0 torchvision==0.21.0 \
  --index-url https://download.pytorch.org/whl/cu128

# Wan2.2 本体と依存
git clone https://github.com/Wan-Video/Wan2.2.git
cd Wan2.2
pip install -r requirements.txt

# 5Bモデルを取得（約16GB、14Bは落とさない）
pip install "huggingface_hub[cli]"
hf download Wan-AI/Wan2.2-TI2V-5B --local-dir ./models/TI2V-5B

# sm_120 がビルドに含まれるか確認（ここがTrueでないと全部無駄になる）
python -c "import torch; print(torch.cuda.get_arch_list())"
# => ['sm_80', 'sm_86', 'sm_90', 'sm_120'] のように sm_120 が出ればOK
```

`get_arch_list()` に `sm_120` が無ければ、その時点でホイールが古い。私は最初 `whl/cu124` を踏んで30分ビルドした挙句に弾かれた。CUDAは12.8で揃える。

## Pythonで5秒720p縦動画を1本生成する最小スクリプト

下のスクリプトがそのまま動く本体だ。`--offload_model` は5Bでは付けない（付けると遅くなるだけで、18.6GBに収まるので不要）。縦動画なので解像度は `720*1280` を明示する。

```python
# gen_short.py — Wan2.2-TI2V-5B で縦5秒動画を1本生成
import subprocess, sys, datetime, pathlib

MODEL_DIR = "./models/TI2V-5B"
OUT_DIR = pathlib.Path("./out"); OUT_DIR.mkdir(exist_ok=True)

def generate(prompt: str) -> pathlib.Path:
    ts = datetime.datetime.now().strftime("%Y%m%d_%H%M%S")
    out = OUT_DIR / f"short_{ts}.mp4"
    cmd = [
        sys.executable, "generate.py",
        "--task", "ti2v-5B",
        "--size", "720*1280",        # 縦Shorts。横は1280*720
        "--frame_num", "121",         # 24fps×5秒+1
        "--sample_steps", "30",       # 50→30で画質ほぼ同等、速度1.6倍
        "--sample_guide_scale", "5.0",
        "--ckpt_dir", MODEL_DIR,
        "--prompt", prompt,
        "--save_file", str(out),
    ]
    print("[gen]", out.name)
    subprocess.run(cmd, check=True)
    return out

if __name__ == "__main__":
    generate("a neon koi fish swimming through rainy Tokyo alley at night, "
             "cinematic, vertical, slow motion, 9:16")
```

ここで実際にハマった失敗談を3つ残しておく。

1. **`--frame_num 120` だと最後のフレームが欠ける**。Wan2.2は内部で4n+1フレーム前提なので `121`（=4×30+1）にしたら破綻が消えた。
2. **`--sample_steps 50` のデフォは過剰**。30に落としても主観でほぼ差がなく、1本あたり2分以上短縮できた。50→30で約38%速くなった実測。
3. **プロンプト末尾に `9:16` `vertical` を入れないと中央に横構図が描かれる**。縦の指定はsizeだけでは足りず、文言の二重指定が効いた。

参考までに、14Bで落ちた時のログはこれだ。offload有効時に100%再現した。

```text
[1/30] denoising...
Segmentation fault (core dumped)
# nvidia-smi 上は VRAM 31.4GB/32GB まで張り付いた直後にプロセス消失
```

VRAMが31.4GBまで埋まった瞬間に死ぬので、32GBでは14B+offloadはギリギリ足りていない。これがKlingやRunwayに課金せずローカルで14Bを回す夢を諦めた根拠だ。

## ffmpegで音源とテロップを焼き込み、Shorts規格に整える

生成したmp4は無音・メタデータ無しなので、このまま上げてもShortsとして弱い。フリーBGMを重ね、解像度を1080×1920へアップスケールし、メタを付与する。

```bash
# 720x1280 -> 1080x1920 にアップスケール + BGM合成 + メタ付与
ffmpeg -i out/short_xxxx.mp4 -stream_loop -1 -i bgm/free_lofi.mp3 \
  -vf "scale=1080:1920:flags=lanczos,fps=30" \
  -c:v libx264 -preset slow -crf 18 \
  -c:a aac -b:a 192k -shortest \
  -metadata title="Neon Koi Tokyo" \
  -movflags +faststart \
  final/ready_short.mp4
```

`scale` のフィルタを `bicubic` から `lanczos` に変えただけで、夜景のネオンのにじみが目に見えて減った。`-crf 18` はファイルサイズと画質のバランス点で、23だと暗部にブロックノイズが乗った。`-movflags +faststart` を忘れるとアップロード後の処理が遅くなる。

## cron + YouTube Data API v3 で毎朝7時に3本を自動投稿

最後に自動化だ。生成3本→ffmpeg整形→YouTube Data API v3でアップロードまでを1スクリプトにまとめ、Linuxなら `crontab`、Windowsならタスクスケジューラで毎朝7時に叩く。`youtube.upload` スコープのOAuthトークンを事前に1回取得しておく。

```python
# upload.py — YouTube Data API v3 でShortsとして予約アップロード
from googleapiclient.discovery import build
from googleapiclient.http import MediaFileUpload
from google.oauth2.credentials import Credentials

creds = Credentials.from_authorized_user_file("token.json",
        ["https://www.googleapis.com/auth/youtube.upload"])
yt = build("youtube", "v3", credentials=creds)

def upload(path: str, title: str):
    body = {
        "snippet": {
            "title": f"{title} #Shorts",        # #Shorts でShorts判定
            "description": "Wan2.2でローカル生成 #AI #Shorts",
            "tags": ["AI動画", "Wan2.2", "Shorts"],
            "categoryId": "24",
        },
        "status": {"privacyStatus": "private", "selfDeclaredMadeForKids": False},
    }
    media = MediaFileUpload(path, chunksize=-1, resumable=True)
    req = yt.videos().insert(part="snippet,status", body=body, media_body=media)
    res = req.execute()
    print("uploaded:", res["id"])

if __name__ == "__main__":
    upload("final/ready_short.mp4", "Neon Koi Tokyo")
```

```bash
# crontab -e に追加：毎朝7時に生成3本→整形→アップロード
0 7 * * * cd /home/me/Wan2.2 && ./.venv/bin/python daily_batch.py >> log/daily.log 2>&1
```

YouTube Data APIの1日のアップロード割り当ては既定で 10,000 quota、動画insertは1本あたり約1,600 quota消費するので、無料枠でも1日6本が現実的な上限だ。私は3本に絞り、`private` で上げてから手で公開チェックする運用にしている。完全無人で `public` にして1本ポリシー違反が出ると、チャンネルごと飛ぶリスクがあるからだ。

## まとめ：0円量産の損益分岐は約20日

5090で14Bを諦めて5Bに割り切れば、API課金0円で縦動画が毎日回る。Kling I2V換算で月¥2,700〜5,400の節約、電気代は月¥162（3本×30日）程度。GPUの初期[投資](https://px.a8.net/svt/ejp?a8mat=4B3XB4+6AU69E+1IRY+25KCOX)を別にすれば、Kling APIを使い続けるより約20日で運用コストが逆転する計算だ。14B+offloadで2日溶かす前に、まず5Bで1本動かすのが最短ルートだと断言する。
