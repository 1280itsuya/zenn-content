## ここでハマると数時間溶ける

GitHub Actions で「デプロイが走った直後に別のデプロイがキャンセルされる」「Reusable Workflow を呼んだら親も子も巻き込まれて落ちる」という挙動に遭遇したら、原因はほぼ `concurrency` の設定です。ログには `Canceling since a higher priority waiting request for '...' exists` とだけ出て、コードのバグに見えないのが厄介なところです。

## つまずく原因

`concurrency` は「同じグループに属する実行を1つに絞る」機能で、`cancel-in-progress: true` を付けると新しい実行が古い実行をキャンセルします。問題は **group キーの組み立て方** です。

```yaml
# caller.yml(よくある失敗)
concurrency:
  group: ${{ github.workflow }}
  cancel-in-progress: true

jobs:
  deploy:
    uses: ./.github/workflows/deploy-reusable.yml
```

`github.workflow` は呼び出し元のワークフロー名を返します。Reusable Workflow 側にも同じ `concurrency` を書いていると、親と子で group キーが衝突したり、複数ブランチ・複数環境のデプロイが「同じグループ」と判定されて、無関係なデプロイ同士が互いをキャンセルし合います。GitHub Actions の `concurrency` はワークフローをまたいで group 名だけで突き合わせるため、名前が被れば別物でも同一グループ扱いになるのです。

## 結論

group キーに **ブランチ(ref)とデプロイ先(environment)を必ず含め**、`concurrency` は呼び出し元(caller)に一本化する。Reusable Workflow 側には書かない。これで衝突は消えます。

## 正しい解決手順

### 1. caller 側で一意な group キーを作る

```yaml
# caller.yml
concurrency:
  group: deploy-${{ github.ref }}-${{ inputs.environment || 'production' }}
  cancel-in-progress: true

on:
  push:
    branches: [main]

jobs:
  deploy:
    uses: ./.github/workflows/deploy-reusable.yml
    with:
      environment: production
    secrets: inherit
```

`github.ref`(例 `refs/heads/main`)と環境名を混ぜることで、main の production デプロイと staging デプロイが別グループになります。

### 2. Reusable Workflow 側の concurrency は削除する

```yaml
# deploy-reusable.yml
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string

# concurrency はここに書かない(caller 側に集約)

jobs:
  run-deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - run: echo "deploying to ${{ inputs.environment }}"
```

子に `concurrency` を残すと、GitHub Actions は親の同時実行制御とは別軸で子をキャンセル判定するため、親が生きていても子だけ落ちる事故が起きます。

### 3. 本番デプロイは途中キャンセルを禁止する

PR の CI なら古い実行を捨てて構いませんが、デプロイの中断は中途半端な状態を残します。本番ジョブは `cancel-in-progress: false` にして直列実行(キューイング)へ切り替えます。

```yaml
concurrency:
  group: deploy-${{ github.ref }}-production
  cancel-in-progress: false   # 走行中は待たせる、殺さない
```

## 検証コマンド

設定後、実際に group が分かれているかは実行中の run から確認できます。

```bash
gh run list --workflow=caller.yml --limit 5
gh run view <run-id> --json status,conclusion
```

連続 push して `conclusion` が `cancelled` にならず、片方が `queued` で待てば成功です。GitHub Actions の同時実行制御は group 名の設計がすべてなので、`ref` と `environment` を必ず織り込む——この一点を押さえれば、デプロイが意図せず消える事故は防げます。
