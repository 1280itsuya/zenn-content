## 発生条件

- `.env` ファイルに定義した変数を `import.meta.env.VITE_XXX` で参照しているはずなのに `undefined` が返る
- `vite.config.ts` で `envPrefix` を独自の値に変更したあと、変数名を更新し忘れている
- `.env` ファイルをプロジェクトルートではなく `src/` 配下など誤ったディレクトリに置いている

## 原因

最小再現として、次のような設定が問題になることが多い。

```ts
// vite.config.ts
import { defineConfig } from 'vite'

export default defineConfig({
  envPrefix: 'APP_',   // ← デフォルトの 'VITE_' を上書きしている
})
```

```bash
# .env
VITE_API_URL=https://api.example.com
```

```ts
// src/main.ts
console.log(import.meta.env.VITE_API_URL) // undefined
```

Vite は起動時に `.env` ファイルを読み込み、`envPrefix` に一致するキーだけをクライアントバンドルへ埋め込む。上記では `envPrefix` を `APP_` に変更しているため、`VITE_` で始まる変数はフィルタリングされて露出しない。セキュリティ上の意図した仕様であり、プレフィックスが一致しない変数はビルド後のコードに含まれない。

また `envPrefix` を変更していない場合でも、`.env` の置き場所が `vite.config.ts` と同じディレクトリ（プロジェクトルート）でなければ読み込まれない点に注意する。

## 修正方法

`envPrefix` を独自値にした場合は、変数名側のプレフィックスをそれに合わせる。

```bash
# .env（APP_ プレフィックスに統一）
APP_API_URL=https://api.example.com
```

```ts
// src/main.ts
console.log(import.meta.env.APP_API_URL) // 'https://api.example.com'
```

`envPrefix` を変更していないなら、デフォルトの `VITE_` を使い続けるのがシンプルで確実だ。

```bash
# .env
VITE_API_URL=https://api.example.com
```

```ts
console.log(import.meta.env.VITE_API_URL) // 'https://api.example.com'
```

加えて、`.env` を編集したあとは開発サーバーを再起動しないと新しい値が反映されない。HMR（Hot Module Replacement）は `.env` の変更を検知しないため、`Ctrl+C` → `vite` で必ずサーバーを立ち上げ直す必要がある。TypeScript を使っている場合は `vite/client` の型定義に `ImportMetaEnv` を拡張しておくと、補完と型チェックの両方が効いて取り違えを防ぎやすい。

```ts
// src/vite-env.d.ts
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_API_URL: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}
