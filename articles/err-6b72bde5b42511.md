## 発生条件

- goroutine がチャネルへの送信または受信でブロックしたまま、対応する受信側・送信側が存在しない状態になった場合
- `sync.WaitGroup` の `Add` / `Done` の数が一致せず、`Wait()` が永久にブロックした場合
- バッファなしチャネルへの送信を同じ goroutine 内で行い、受信する goroutine が起動していない場合

## 原因

Go ランタイムはすべての goroutine がブロック状態と判断したとき、処理を継続できないため `fatal error: all goroutines are asleep - deadlock!` を出力してパニックします。

典型的な最小再現コードは以下の通りです。

```go
package main

func main() {
    ch := make(chan int) // バッファなし

    // 受信する goroutine を起動せずに送信
    ch <- 1 // main goroutine がここで永久ブロック

    val := <-ch
    _ = val
}
```

上のコードでは `ch <- 1` を実行した時点で main goroutine がブロックします。しかし受信側の goroutine は存在しないため、ランタイムがデッドロックを検知してプログラムを強制終了します。`sync.WaitGroup` を使う場合も同様に、`Add(2)` に対して `Done()` を 1 回しか呼ばないと `Wait()` が抜けられず同じエラーになります。

## 修正方法

受信を別の goroutine で行い、`WaitGroup` で完了を待つ形に修正します。

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    ch := make(chan int)
    var wg sync.WaitGroup

    wg.Add(1)
    go func() {
        defer wg.Done()
        val := <-ch // goroutine 内で受信
        fmt.Println("received:", val)
    }()

    ch <- 1 // 受信側が存在するので送信がブロックしない
    wg.Wait()
}
```

要点は次の 3 点です。

- **受信 goroutine を先に起動する**: 送信より前に `go func()` で受信ループまたは単発受信を立ち上げる。
- **バッファ付きチャネルを検討する**: 送受信のタイミングを切り離せる場合は `make(chan int, N)` でバッファを持たせると goroutine の起動順に依存しなくなる。
- **`WaitGroup` の `Add` / `Done` を対称にする**: `Add` に渡した数だけ `Done` が呼ばれることを確認する。`defer wg.Done()` を goroutine の先頭に置くとリターンパスの漏れを防げる。

`fatal error: all goroutines are asleep - deadlock!` はランタイムレベルの検知なので、スタックトレースに各 goroutine の停止位置が出力されます。どのチャネル操作や `Wait` でブロックしているかをトレースで確認し、対応する送受信・`Done` 呼び出しが存在するかをコードと突き合わせるのが最短の原因特定手順です。
