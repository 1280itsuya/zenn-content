# 自動化botを動かすなら「VPS一択」——[個人開発](https://www.amazon.co.jp/s?k=%E5%80%8B%E4%BA%BA%E9%96%8B%E7%99%BA%20%E6%9C%AC&tag=1280itsuya22-22)者のためのレンタルサーバー選び

## 結論

定期実行する自動化bot（cron/常駐プロセス）を動かすなら、**共有レンタルサーバーではなくVPSを選ぶ**べきです。具体的には **ConoHa VPS** か **Xserver VPS** をまず検討し、料金最優先なら **さくらのVPS**、[ドメイン](https://px.a8.net/svt/ejp?a8mat=4B3XB4+4VMW8I+50+2HU3GX)も一括で揃えたいなら **お名前.com** という棲み分けが実務的に扱いやすい、というのが私の運用結論です。

## なぜ共有サーバーではダメなのか

共有レンタルサーバー（いわゆる「○○サーバー」のスタンダードプラン）は、Webサイト公開向けに最適化されていて、**SSHでの常駐プロセスやcronの自由度が低い**ことが多いです。[Python](https://www.amazon.co.jp/s?k=Python%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22)のバージョンが固定だったり、長時間プロセスが規約で禁止されていたり、`pip install` で詰まる場面に何度も遭遇しました。

VPSなら root 権限が手に入るので、

- 任意のランタイム（Python / Node / Go）を自由に入れられる
- `systemd` や `cron` で常駐・定期実行を自分で組める
- Docker をそのまま動かせる

botのように「自分の好きな環境を、止めずに動かし続けたい」用途とVPSは相性が良いです。

## 料金感（必ず公式で最新を確認すること）

各社とも頻繁にキャンペーンや改定があるため、**金額は必ず公式サイトで最新を確認してください**。ここでは相対的な傾向だけ書きます。

- **さくらのVPS**: 最小構成が安く、長く運用するbot基盤としてコスパ重視向き
- **ConoHa VPS**: 時間課金があり「試して気に入らなければ消す」がやりやすい。GUIも分かりやすい
- **Xserver VPS**: スペックあたりの価格設定が積極的で、CPU/メモリを盛りたい時に候補
- **お名前.com**: VPSとドメインを同一アカウントで管理でき、DNS設定まで一気通貫

bot1個なら最小プラン（メモリ1GB前後）で足りることが多く、画像生成など重い処理を挟むならメモリを盛る、という判断になります。

## 具体的な構築手順

契約後、最低限これだけやれば動き出します。

**1. SSH鍵でログインする**

パスワード認証は総当たりされるので、鍵認証に切り替えます。

```bash
ssh-keygen -t ed25519
ssh-copy-id user@<サーバーIP>
# /etc/ssh/sshd_config で PasswordAuthentication no にして再起動
```

**2. 最低限のセキュリティ**

- `ufw` で必要なポート（22など）だけ開放
- 一般ユーザーを作り、root直ログインは無効化

**3. botを常駐させる**

定期実行なら `crontab -e`、止めたくない常駐型なら `systemd` のユニットを書いて `enable` しておくと、再起動後も自動で立ち上がります。

**4. DNSを向ける（Webhookを受ける場合）**

botが外部からのリクエストを受けるなら、ドメインのAレコードをサーバーIPに向けます。お名前.comなどで取得したドメインの管理画面から、

```
A  bot.example.com  <サーバーIP>
```

を登録するだけです。反映には時間差があるので、慌てず `dig bot.example.com` で確認します。

## まとめ

bot運用の肝は「止まらないこと」と「環境を自分で握れること」。この2点でVPSが共有サーバーに勝ります。まずは時間課金で試せるConoHa、長期はさくら、スペック重視はXserver、ドメイン込みならお名前.com、と使い分けるのがおすすめです。料金とスペックは変動するため、契約前に必ず各社の公式情報を確認してください。
