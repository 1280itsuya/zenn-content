# ベクタDB不要で作る最小RAG：自社Markdownファイル + Claude APIで社内Q&Aを5時間で構築する

社内規約・設計書・議事録をまとめて「これどこに書いてあるっけ？」と検索するたびにフォルダを漁っている——そのコストを減らしたい。しかしChromaやPineconeを導入するほど資料は多くない。そんなケースに刺さる構成を紹介する。

## 前提と構成

- **資料量**: Markdownファイル合計 50〜200KB 程度（数十〜百ファイル）
- **アプローチ**: チャンク分割 → TF-IDFでキーワード検索 → 上位チャンクをClaude APIへ渡す
- **依存パッケージ**: `anthropic`, `scikit-learn` のみ

ベクタDBが要らない理由は単純で、Claude 3.5 Sonnetのコンテキストが約20万トークンあり、絞り込んだチャンク数十個なら余裕で収まるからだ。

## ディレクトリ構成

```
rag/
```

## コード全文

```python
import os, glob, re
from anthropic import Anthropic
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity
from dotenv import load_dotenv

load_dotenv()
client = Anthropic()

def load_chunks(docs_dir="docs", chunk_size=400):
    chunks = []
    for path in glob.glob(f"{docs_dir}/**/*.md", recursive=True):
        text = open(path, encoding="utf-8").read()
        # 空行2つ以上で段落分割、それでも長い段落は文字数で切る
        paragraphs = re.split(r"\n{2,}", text)
        for para in paragraphs:
            para = para.strip()
            if not para:
                continue
            for i in range(0, max(1, len(para)), chunk_size):
                chunk = para[i:i+chunk_size]
                if chunk.strip():
                    chunks.append({"text": chunk, "source": path})
    return chunks

def retrieve(query, chunks, top_k=8):
    texts = [c["text"] for c in chunks]
    vec = TfidfVectorizer(analyzer="char", ngram_range=(2, 3))
    tfidf = vec.fit_transform(texts + [query])
    sims = cosine_similarity(tfidf[-1], tfidf[:-1])[0]
    indices = sims.argsort()[::-1][:top_k]
    return [chunks[i] for i in indices if sims[i] > 0.01]

def answer(query, chunks):
    hits = retrieve(query, chunks)
    if not hits:
        return "関連ドキュメントが見つかりませんでした。"
    context = "\n\n---\n\n".join(
        f"[{h['source']}]\n{h['text']}" for h in hits
    )
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="あなたは社内ドキュメントアシスタントです。必ず以下の資料だけを根拠に答え、資料に書かれていないことは「資料に記載がありません」と答えてください。",
        messages=[
            {"role": "user", "content": f"資料:\n{context}\n\n質問: {query}"}
        ],
    )
    return resp.content[0].text

if __name__ == "__main__":
    print("ドキュメントを読み込み中...")
    chunks = load_chunks()
    print(f"{len(chunks)} チャンク準備完了")
    while True:
        q = input("\n質問> ").strip()
        if not q:
            break
        print(answer(q, chunks))
```

## つまずきポイント3つ

**1. チャンク分割は「段落単位」が基本**
固定文字数だけで切るとMarkdownの表やコードブロックが途中で千切れる。`\n{2,}` で段落に分割してから、長い段落だけを `chunk_size` で再分割するのが安定する。

**2. TF-IDFは日本語で `analyzer="char"` を使う**
形態素解析なしで日本語対応するには `analyzer="char", ngram_range=(2, 3)` が手軽で実用的。英語混じりのドキュメントでも崩れにくい。

**3. Claude に「資料だけを根拠にして」を明示しないと幻覚が混じる**
systemプロンプトで根拠制限を書かないと、不足情報を学習済み知識で補ってしまう。「資料に記載がありません」と返させる一文が必須。

## 限界と次のステップ

TF-IDF検索は「同じ単語が含まれているか」しか見ないため、「社員旅行の申請手順は？」に対して「旅行費用精算規定」が上がらないケースがある。資料が増えてこの限界が気になり始めたら、そのタイミングで初めてベクタDB（ChromaやFaiss）への移行を検討すればいい。最小構成でまず動かすことが先決だ。
