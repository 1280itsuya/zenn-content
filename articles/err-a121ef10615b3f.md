## 発生条件

- nilポインタ（`*T`型）に対してフィールドアクセスやメソッド呼び出しを行ったとき
- インターフェース変数が`nil`のまま、そのメソッドを呼び出したとき
- 関数がエラーとともに`nil`ポインタを返したのに、エラーチェックせず戻り値を使ったとき

## 原因

Goランタイムは、nilポインタへのメモリアクセスを検知すると `panic: runtime error: invalid memory address or nil pointer dereference` を発生させてプログラムを停止させます。

典型的な最小再現コードを示します。

```go
package main

import (
	"fmt"
	"strconv"
)

type Config struct {
	Value int
}

func loadConfig(input string) (*Config, error) {
	n, err := strconv.Atoi(input)
	if err != nil {
		return nil, err
	}
	return &Config{Value: n}, nil
}

func main() {
	cfg, err := loadConfig("invalid")
	// err != nil なのに cfg を使おうとしている
	_ = err
	fmt.Println(cfg.Value) // ここで panic
}
```

`loadConfig` が失敗した場合、`cfg` は `nil` を返します。しかし `err` を確認せずに `cfg.Value` へアクセスしているため、ランタイムがnilポインタへのメモリアクセスを検知してパニックが発生します。

インターフェース経由でも同様のパニックが起きます。

```go
package main

import "fmt"

type Writer interface {
	Write(s string)
}

func main() {
	var w Writer // nil インターフェース
	w.Write("hello") // panic: invalid memory address or nil pointer dereference
}
```

## 修正方法

エラー戻り値を必ず確認してからポインタを使うのが基本です。

```go
package main

import (
	"fmt"
	"log"
	"strconv"
)

type Config struct {
	Value int
}

func loadConfig(input string) (*Config, error) {
	n, err := strconv.Atoi(input)
	if err != nil {
		return nil, err
	}
	return &Config{Value: n}, nil
}

func main() {
	cfg, err := loadConfig("invalid")
	if err != nil {
		log.Fatalf("設定の読み込みに失敗しました: %v", err)
	}
	fmt.Println(cfg.Value)
}
```

ポインタを直接受け取る関数では、使用前に`nil`チェックを入れます。

```go
func printValue(cfg *Config) {
	if cfg == nil {
		fmt.Println("cfg is nil, skipping")
		return
	}
	fmt.Println(cfg.Value)
}
```

インターフェース変数については、型アサーションの結果を確認するか、変数が確実に初期化されてから呼び出すよう設計します。

```go
var w Writer = &ConcreteWriter{} // 必ず具体型で初期化してから使う
w.Write("hello")
```

`panic: runtime error: invalid memory address or nil pointer dereference` はスタックトレースにパニック発生箇所が明示されるため、まずその行を確認し、関連する変数が`nil`になり得るパスを追います。`go vet` や静的解析ツール（`staticcheck`など）は一部のnil参照を事前に検出できるため、CI に組み込んでおくと早期発見に役立ちます。
