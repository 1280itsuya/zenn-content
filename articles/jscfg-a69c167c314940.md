## 発生条件

- ESLint v8以前の`.eslintrc`系設定ファイルと、v9の`eslint.config.js`（Flat Config）が同一プロジェクトに混在している
- `@typescript-eslint/eslint-plugin`や`eslint-plugin-react`など複数プラグインが`no-unused-vars`を上書き・無効化している
- `extends`の順序が誤っており、後から読み込まれた設定が`no-unused-vars`を`off`に上書きしている

## 原因

典型的な競合パターンを示す。

```json
// .eslintrc.json
{
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended",
    "plugin:react/recommended"
  ],
  "rules": {
    "no-unused-vars": "error"
  }
}
```

この設定では`plugin:@typescript-eslint/recommended`が内部で`no-unused-vars: off`を設定している。TypeScriptプラグインは`@typescript-eslint/no-unused-vars`を代わりに使うため、意図的にベースルールを無効化する。

その後に`"no-unused-vars": "error"`を書いても、**`@typescript-eslint/recommended`が持つ`rules`ブロックの`no-unused-vars: off`が`extends`経由で適用された後**に、手動の`rules`が上書きするはずだが、プラグイン側の型チェック連携が壊れる。また、`eslint-plugin-react`の一部バージョンも`no-unused-vars`を独自に操作する。

結果として`no-unused-vars`は構文上`error`に見えながら、TypeScript型情報を参照した未使用変数（型インポート等）は検出されず、ルールが実質的に機能しない状態になる。

Flat Configと旧形式の混在時はさらに深刻で、ESLintがどちらの設定を優先するか環境依存になり、ルール全体が不安定になる。

## 修正方法

TypeScriptプロジェクトでは`no-unused-vars`を直接使わず、`@typescript-eslint/no-unused-vars`に一本化する。

```json
// .eslintrc.json
{
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended"
  ],
  "rules": {
    "no-unused-vars": "off",
    "@typescript-eslint/no-unused-vars": [
      "error",
      {
        "args": "after-used",
        "ignoreRestSiblings": true,
        "varsIgnorePattern": "^_",
        "argsIgnorePattern": "^_"
      }
    ]
  }
}
```

要点は2つ。

1. `"no-unused-vars": "off"`を明示的に書く。`plugin:@typescript-eslint/recommended`がすでに`off`にしているが、後続の設定や他プラグインが再度有効化するケースを防ぐために明示しておく。

2. `@typescript-eslint/no-unused-vars`をルールとして明記し、オプションで実運用に合わせて調整する。`varsIgnorePattern: "^_"`は慣例的に先頭アンダースコアで意図的未使用を示す変数を除外するため有用。

Flat Config（`eslint.config.js`）を使う場合は`.eslintrc`系ファイルを完全に削除する。両形式が共存するとESLintが予期しない順序で設定を解釈し、どのルールが有効かの追跡が困難になる。移行途中のプロジェクトでは`ESLINT_USE_FLAT_CONFIG=false`の環境変数で旧形式に固定するか、移行を完了させるかのどちらかに統一する。
