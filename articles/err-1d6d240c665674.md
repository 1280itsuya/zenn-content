## 発生条件

- `tsconfig.json` で `strictNullChecks`（または `strict`）が有効になっている
- 値の型が `T | null`（あるいは `T | undefined`）で、null の可能性が型として残っている
- その値のプロパティアクセスやメソッド呼び出しを、nullチェックなしで行っている

## 原因

`strictNullChecks` 下では、null になりうる値をそのまま参照すると `TS2531: Object is possibly 'null'.` が報告されます。代表例が `document.getElementById` で、戻り値は `HTMLElement | null` です。

```typescript
// strictNullChecks: true
const el = document.getElementById("app");
// el は HTMLElement | null
el.textContent = "hello";
// Error: TS2531: Object is possibly 'null'.
```

`el` が `null` の場合に `textContent` へ代入すると実行時エラーになるため、コンパイラが事前に弾いています。これは型に `null` が含まれている限り発生し、変数名やAPIに依存しません。

## 修正方法

最も安全なのは、参照前に明示的なnullチェックでガードする方法です。これでブロック内の型は `null` が除外され、`Object is possibly 'null'` は解消されます。

```typescript
// 方法1: 事前にnullチェック（推奨）
const el = document.getElementById("app");
if (el !== null) {
  el.textContent = "hello"; // ここでは el は HTMLElement
}

// 方法2: オプショナルチェーン（参照だけなら簡潔）
document.getElementById("app")?.setAttribute("data-ready", "true");

// 方法3: 非null表明 ! は「絶対にnullでない」と確信できる時だけ
const el2 = document.getElementById("app")!;
el2.textContent = "hello"; // null だった場合は実行時に落ちる
```

要点は次の通りです。

- **方法1（nullチェック）**: 最も安全で、null時の分岐も書ける。基本はこれを選ぶ
- **方法2（`?.`）**: null の時は評価をスキップするだけ。値の取得・副作用呼び出しが目的で、null時に何もしなくてよい場合に向く
- **方法3（`!`）**: 型エラーは消えるが安全性の保証はコンパイラから外れる。null が来ないと確証できる箇所に限定する

`!` で握りつぶすと `TS2531` は消えても実行時の null 参照は防げません。原則は方法1か方法2で、`null` の可能性そのものを型レベルで解消することを優先してください。
