## Zenn公開API `/api/users/{name}/articles` でPV(liked_count)を取得する罠

最初に断っておくと、Zennは記事ごとの**正確なPV数を公開APIで返しません**。ダッシュボード（`/dashboard`）のPVは内部APIで、ログインCookieが必要です。ここを自動化しようとしてCookieをGitHub Secretsに入れたのですが、Zennのセッションは数日で失効し、Actionsが静かに401を返して通知が来なくなりました。これが1つ目の失敗です。

そこで方針を変え、**公開APIで取れる`liked_count`（いいね）と`comments_count`を日次KPIの代理指標**にしました。PVと強く相関し、かつ失効しません。エンドポイントはこれです。

```python
import requests

def fetch_zenn_articles(username: str) -> list[dict]:
    """Zenn公開APIから全記事を取得（ページング対応）"""
    articles, page = [], 1
    while True:
        url = f"https://zenn.dev/api/users/{username}/articles"
        resp = requests.get(url, params={"page": page, "order": "latest"}, timeout=10)
        resp.raise_for_status()
        data = resp.json()
        batch = data.get("articles", [])
        if not batch:
            break
        for a in batch:
            articles.append({
                "slug": a["slug"],
                "title": a["title"],
                "liked": a["liked_count"],
                "comments": a["comments_count"],
            })
        if data.get("next_page") is None:
            break
        page += 1
    return articles
```

実測で、記事30本の取得が平均0.8秒。`order=latest`を付けないと並びが安定せず差分突合がズレるので必須でした。

## 前日スナップショットとの差分をPythonで計算しlikedの増分だけ抽出する

KPI可観測化の肝は「累積値」ではなく「**昨日からの増分**」を見せることです。累積のいいね数を毎日通知しても、伸びているのかは分かりません。スナップショットをリポジトリ内のJSONに保存し、git管理することで履歴も残します。

```python
import json, os
from pathlib import Path

SNAPSHOT = Path("data/zenn_snapshot.json")

def load_prev() -> dict:
    if SNAPSHOT.exists():
        return json.loads(SNAPSHOT.read_text(encoding="utf-8"))
    return {}

def diff_and_save(articles: list[dict]) -> list[dict]:
    prev = load_prev()
    changes, current = [], {}
    for a in articles:
        current[a["slug"]] = a["liked"]
        before = prev.get(a["slug"], 0)
        delta = a["liked"] - before
        if delta > 0:
            changes.append({"title": a["title"], "liked": a["liked"], "delta": delta})
    SNAPSHOT.parent.mkdir(parents=True, exist_ok=True)
    SNAPSHOT.write_text(json.dumps(current, ensure_ascii=False, indent=2), encoding="utf-8")
    changes.sort(key=lambda x: x["delta"], reverse=True)
    return changes
```

ここで2つ目の失敗。初回実行時は`prev`が空なので**全記事の累積いいねがそのまま「増分」として爆撃通知**されました。30件のいいねが全部「+12」みたいに飛んできてDiscordが埋まった。対策は、`SNAPSHOT`が存在しない初回は通知をスキップして保存だけする、を後述のmain側に入れることです。

## Discord WebhookへPython requestsでKPIサマリをPOSTする

SlackでもDiscordでも、Webhook URLにJSONをPOSTするだけです。Discordは`content`が2000文字を超えると400で弾かれるので、上位5件に絞ります。これも実際に長文で400を食らって学びました。

```python
import requests, os

def notify(changes: list[dict]):
    webhook = os.environ["DISCORD_WEBHOOK_URL"]
    if not changes:
        text = "📊 本日のZenn KPI: いいね増分なし（昨日比±0）"
    else:
        total = sum(c["delta"] for c in changes)
        lines = [f"📈 Zenn いいね増分 合計 +{total}（昨日比）", ""]
        for c in changes[:5]:
            lines.append(f"・+{c['delta']}　{c['title']}（累計{c['liked']}）")
        text = "\n".join(lines)
    r = requests.post(webhook, json={"content": text}, timeout=10)
    r.raise_for_status()
```

Slackに送りたい場合は同じ関数で`json={"text": text}`に変えるだけ。キー名が`content`→`text`なのが唯一の差分です。

main側はこう繋ぎます。初回スキップのガードを入れています。

```python
def main():
    is_first = not SNAPSHOT.exists()
    articles = fetch_zenn_articles(os.environ["ZENN_USERNAME"])
    changes = diff_and_save(articles)
    if is_first:
        print("初回実行: スナップショット保存のみ、通知スキップ")
        return
    notify(changes)

if __name__ == "__main__":
    main()
```

## GitHub Actionsのcronを毎朝6時(JST=UTC21時)に設定しsnapshotをコミットする

ここが一番ハマりやすい。GitHub Actionsのcronは**UTC固定**です。日本の朝6時に動かしたいなら`21 21 * * *`ではなく`0 21 * * *`（前日UTC21時）。私は最初JSTで書いて15時に通知が来て混乱しました。

そしてスナップショットJSONを次回のために**Actions内からコミット＆push**します。これを忘れると毎回「初回扱い」になり差分が常に空になります。

```yaml
name: zenn-kpi-daily

on:
  schedule:
    - cron: "0 21 * * *"   # UTC21時 = JST朝6時
  workflow_dispatch:        # 手動実行も可能にする

permissions:
  contents: write           # snapshotのpushに必須

jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install requests
      - name: Run KPI notifier
        env:
          ZENN_USERNAME: ${{ secrets.ZENN_USERNAME }}
          DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}
        run: python kpi_notify.py
      - name: Commit snapshot
        run: |
          git config user.name "github-actions"
          git config user.email "actions@github.com"
          git add data/zenn_snapshot.json
          git diff --cached --quiet || git commit -m "chore: update zenn snapshot"
          git push
```

`permissions: contents: write`を書かないとpushが403で落ちます。デフォルトのGITHUB_TOKENは読み取り専用になっているリポジトリが多く、ここで30分溶かしました。`git diff --cached --quiet || git commit`は、変更がない日に空コミットでコケないためのイディオムです。

## 3週間運用した実測値とcronの遅延という現実

回してみての数字を正直に出します。通知1回あたりのAPI呼び出しは記事30本で4リクエスト、Actionsの実行時間は約35秒、月30回でGitHub無料枠2000分のうち**約18分しか消費しません**。完全に¥0です。

ただしGitHub Actionsの`schedule`は**定刻通りには来ません**。混雑時は3〜15分遅延し、たまにスキップされます。厳密な定時性が必要なKPIには向きません。私は「朝の通知が6:00〜6:20の間に来ればOK」という運用にして許容しました。秒単位の精度が要るなら自前のcronサーバーかCloud Schedulerを使うべきです。

もう一段精度を上げたい人は、`liked_count`だけでなく記事個別ページのOGPやRSS（`https://zenn.dev/{user}/feed`）を併用すると新規公開検知も足せます。私はここまでで「どの記事が今日伸びたか」が毎朝1行で分かるようになり、ネタ選定の打率が体感で上がりました。

この手の自動化を本気で組むなら、ローカル開発の遅さで消耗しないよう一度きちんと環境を整える価値があります。私はVPS上にcronとログ基盤を寄せてからデバッグが一気に楽になりました。さくらVPSやConoHa VPSは月1000円前後で常時稼働のcron基盤として手堅く、Actions無料枠の遅延・スキップ問題を回避したい人には現実的な選択肢です。GitHub ActionsとWebhookの設計をきちんと理解したい人は、CI/CD実践書を1冊通読すると今回のpermissionsやcron UTC問題で詰まらなくなります。
