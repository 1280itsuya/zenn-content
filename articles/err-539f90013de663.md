## 発生条件

- Next.js や Nuxt などの SSR フレームワークで、サーバー側とクライアント側で異なる値（現在時刻・ロケール依存の文字列・乱数など）をレンダリングしたとき
- `new Date()` や `Date.now()`、`Math.random()` など実行タイミングで結果が変わる式をコンポーネントのトップレベルで直接呼び出したとき
- ブラウザ固有の API（`window.innerWidth` など）をサーバーサイド実行フェーズで参照したとき

## 原因

`Error: Text content does not match server-rendered HTML` は、ハイドレーション時に DOM のテキストノードがサーバー出力と一致しないと [React](https://www.amazon.co.jp/s?k=React%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22) が検出して投げるエラーです。

以下は典型的な最小再現コードです。

```tsx
// NG: サーバーとクライアントで new Date() の結果が異なる
export default function Clock() {
  const now = new Date().toLocaleTimeString("ja-JP");
  return <p>{now}</p>;
}
```

サーバーはビルド時 or リクエスト時の時刻でレンダリングし、クライアントはハイドレーション時の（わずかに後の）時刻で同じ箇所を計算します。このズレが `Text content does not match server-rendered HTML` を引き起こします。ロケール（`ja-JP` vs `en-US`）が環境ごとに異なる場合も同様のエラーになります。

## 修正方法

クライアントでのみ確定する値は、マウント後に `useState` + `useEffect` で確定させます。

```tsx
import { useState, useEffect } from "react";

export default function Clock() {
  const [now, setNow] = useState<string | null>(null);

  useEffect(() => {
    // マウント後＝クライアントのみで実行される
    setNow(new Date().toLocaleTimeString("ja-JP"));
  }, []);

  if (now === null) {
    // サーバー側レンダリングと一致させるプレースホルダ
    return <p>--:--:--</p>;
  }
  return <p>{now}</p>;
}
```

要点は 2 つです。

1. **初期値をサーバーと一致させる** — `useState(null)` にして、サーバー HTML 側も同じ `--:--:--` を出力するようにします。これで `Text content does not match server-rendered HTML` は発生しません。
2. **`useEffect` の空配列依存** — マウント時に 1 回だけ実行し、以降は不要な再実行を防ぎます。

ロケール起因の場合は、サーバーとクライアントで `toLocaleTimeString` に渡すロケール文字列を明示的に固定（例: 常に `"ja-JP"` を渡す）するだけで解消するケースもあります。実行環境の `LANG` や `TZ` 設定が揺れている場合はそちらも確認してください。

`suppressHydrationWarning` 属性で警告を抑制するアプローチもありますが、根本的な不一致は残るため、ユーザーに見える値がちらつく可能性があります。原則として上記の `useEffect` パターンで根本修正するのが確実です。
