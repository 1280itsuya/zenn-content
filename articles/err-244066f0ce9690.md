## 発生条件

- Kubernetes Pod が起動直後にプロセス終了コード非ゼロで落ち、kubelet が再起動を繰り返した結果 `CrashLoopBackOff` ステータスに遷移する
- `kubectl get pods` で STATUS が `CrashLoopBackOff` と表示され、RESTARTS カウントが増加し続けている
- コンテナイメージのエントリポイント・環境変数・依存サービスへの接続のいずれかが不正で、プロセスが即座に異常終了している

## 原因

典型的な最小再現例として、必須環境変数が未定義のままコンテナが起動するケースを示す。

```yaml
# deployment.yaml（問題あり）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:latest
          # DATABASE_URL が未定義のまま起動
```

```python
# app.py
import os
import sys

db_url = os.environ["DATABASE_URL"]  # KeyError で即クラッシュ
print(f"connecting to {db_url}")
```

この構成では Pod 起動時に `KeyError: 'DATABASE_URL'` が発生してプロセスが終了し、kubelet が再起動→再クラッシュを繰り返すことで `CrashLoopBackOff` に至る。

まず以下のコマンドでクラッシュ直前のログを確認する。

```bash
# 直前の終了ログを取得（--previous が重要）
kubectl logs --previous <pod-name>

# Pod の詳細イベントも確認する
kubectl describe pod <pod-name>
```

`--previous` フラグを付けないと現在起動中（または起動待ち）のコンテナのログを参照するため、クラッシュ原因が見えないことが多い。`describe` の Events セクションには終了コードや OOMKilled などの追加情報が含まれる。

## 修正方法

環境変数の欠落が原因の場合、Secret または ConfigMap 経由で必要な値を注入する。

```yaml
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secret
type: Opaque
stringData:
  DATABASE_URL: "postgresql://user:pass@db-host:5432/mydb"
```

```yaml
# deployment.yaml（修正後）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:latest
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: myapp-secret
                  key: DATABASE_URL
```

```bash
kubectl apply -f secret.yaml
kubectl apply -f deployment.yaml
kubectl rollout status deployment/myapp
```

環境変数以外の原因として多いのは次のパターンで、それぞれ対処が異なる。

- **依存サービス未起動**: `initContainers` で依存先の疎通確認を行い、準備完了後にメインコンテナを起動する
- **イメージのエントリポイント誤り**: `kubectl describe pod` の `Command` / `Args` を確認し、`docker run` 相当で手元で実行して再現する
- **メモリ不足（OOMKilled）**: `describe` の終了理由が `OOMKilled` であれば `resources.limits.memory` を引き上げるか、アプリのメモリ使用量を見直す

`CrashLoopBackOff` は症状であって原因ではないため、`kubectl logs --previous` と `kubectl describe pod` の出力を必ず確認してから修正方針を決める。
