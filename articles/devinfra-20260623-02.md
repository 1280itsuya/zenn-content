# 自動化botを動かすレンタルサーバー選び──個人開発者のための実践ガイド

## 結論から言う

**ConoHa VPS・Xserver VPS・さくらのVPS のいずれかを最小プランで借り、SSH鍵認証 + systemd で bot を常駐させる** のが現時点で最も手堅い選択肢だ。

理由は三つある。①月額数百円〜千数百円台のプランでも 24/7 稼働できる、②sshd が素直に使える、③インスタンスを自分でコントロールできるため [Python](https://www.amazon.co.jp/s?k=Python%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22) 仮想環境や cron・systemd を自由に組める。

GAS や Heroku の無料枠と迷う人も多いが、無料枠はスリープ・実行時間上限・外部通信制限がつきまといがちで、定期実行 bot には向かない場面が多い。

## 選定基準

### 1. 月額料金と最小スペック

各社とも料金は随時変動するため、必ず **契約直前に公式サイトで確認** すること。以下は記事執筆時点の参考感覚値で、保証値ではない。

- **ConoHa VPS**: 最小プランで月額数百円台〜（512MB RAM 帯）。管理コンソールが直感的で初心者にも扱いやすい。
- **Xserver VPS**: 最小プランで月額千円前後の帯域にある。Xserver ブランドの安心感と国内バックボーンが強み。
- **さくらのVPS**: 最小プランが月額数百円台から。老舗で情報量が豊富、石橋を叩きたい人向け。
- **お名前.com VPS**: キャンペーン価格が安いが、通常価格に戻ったときの落差に注意。[ドメイン](https://px.a8.net/svt/ejp?a8mat=4B3XB4+4VMW8I+50+2HU3GX)と同一ベンダーにまとめたい場合は検討する価値あり。

bot が軽量（Pythonスクリプト1〜2本、API叩いてDB書込み程度）なら **1GB RAM・1vCPU** で十分動く。余裕を持たせたいなら 2GB プランを選ぶ。

### 2. SSH アクセスと鍵認証

レンタルサーバーを選ぶうえで **SSH が普通に使えるか** は最重要チェック項目だ。共用レンタルサーバー（ロリポップ・エックスサーバーのホスティング版など）は SSH が制限されており、systemd が触れない。VPS 一択になる理由がここにある。

初期設定の流れ：

```bash
# ローカルで鍵ペアを生成
ssh-keygen -t ed25519 -C "your_bot_key"

# 公開鍵をサーバーへ転送
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@YOUR_SERVER_IP

# パスワード認証を無効化（/etc/ssh/sshd_config）
PasswordAuthentication no
PubkeyAuthentication yes
```

鍵を設定したら `PermitRootLogin no` にして root 直接ログインも封じておく。

### 3. DNS と固定 IP

外部サービス（Slack Webhook・LINE Notify・各種 API）から **IP 制限を求められる場合**、固定 IP が必須になる。VPS はデフォルトで固定 IP が割り当てられるため、この点は問題ない。

ドメインを使って管理したいなら、Aレコードで VPS の IP を向けるだけでよい。ドメインは利用しているレジストラ（お名前.com など）の管理画面で設定できる。bot 管理 UI を外に公開する際は必ず HTTPS（Let's Encrypt）を通す。

## bot の常駐パターン

### systemd（推奨）

```ini
# /etc/systemd/system/mybot.service
[Unit]
Description=My Automation Bot
After=network.target

[Service]
User=botuser
WorkingDirectory=/home/botuser/bot
ExecStart=/home/botuser/bot/.venv/bin/python main.py
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable mybot
sudo systemctl start mybot
sudo journalctl -u mybot -f  # ログ確認
```

`Restart=on-failure` を付けることで、例外クラッシュ後に自動復帰する。定期実行には systemd timer か cron を使えばよい。

### cron での定期実行

```cron
# crontab -e
30 7 * * * /home/botuser/bot/.venv/bin/python /home/botuser/bot/run.py >> /var/log/mybot.log 2>&1
```

パスを絶対パスで書くのは cron の基本。仮想環境の python バイナリを直接指定することで `source activate` 不要になる。

## コスト最適化の考え方

- **最小プランで始めて、詰まったらスケールアップ**。ほとんどの個人 bot は 1GB RAM で動く。
- スナップショット・バックアップ機能はオプション料金がかかる場合が多い。git で構成管理し、`.env` は別途手元で管理すれば、最悪インスタンスを作り直せる。
- 長期割引（3ヶ月・6ヶ月・12ヶ月前払い）を使うと実質月額が下がるが、サービスを試す前にコミットしすぎない。最初は月単位で契約する。

## まとめ

| 判断ポイント | チェック内容 |
|---|---|
| SSH 鍵認証 | 使えるか（VPS であればほぼ OK） |
| systemd / cron | 触れるか |
| 固定 IP | API の IP 制限がある場合に必須 |
| 料金 | 公式で必ず最新を確認 |
| 復旧容易性 | git 管理 + .env 分離で再構築できるか |

bot 用サーバーは「壊れても 30 分で元に戻せる」構成が理想だ。ConoHa・Xserver・さくら・お名前.com のどれを選んでも、SSH と systemd さえ使えれば大きく外れない。まず最小プランで動かしてみて、ボトルネックが出たら上げる──この順番が一番失敗が少ない。
