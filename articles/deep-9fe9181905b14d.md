## 何が起きているのか（原因の本質）

`TS2322: Type 'string' is not assignable to type 'number'.` — このエラーはTypeScriptの**静的型システムが型の非互換性を検出したとき**に発生する。

TypeScriptはコンパイル時に各変数・引数・プロパティの型を追跡している。ある型 `T` として宣言された場所に、`T` のサブタイプでも `T` に代入可能な型でもない値を渡そうとすると、型チェッカーがエラーを出す。`string` と `number` はJavaScriptのプリミティブ型であり、一方が他方のサブタイプになることはない。だから `number` 型の変数に `string` 値を代入しようとすると、型チェッカーは厳密に弾く。

重要なのは「なぜTypeScriptが `string` と `number` を互換と見なさないのか」という根本だ。JavaScriptランタイムでは `"3" + 2` が `"32"` になるなど、暗黙の型変換が随所で起きる。TypeScriptはそういった危険なランタイム挙動を**コンパイル時に封じる**ために型システムを持っている。TS2322はその型システムが正しく機能している証拠であり、エラーは「意図しない暗黙変換を許すな」というコンパイラからの警告だ。

このエラーが起きる典型的な状況は以下のいずれかに集約される。

1. 外部から取得したデータ（APIレスポンス、URLパラメータ、フォーム値）が `string` として入ってくるのに、`number` を期待する関数や変数にそのまま渡している
2. オブジェクトのプロパティ型が `number` と宣言されているのに、リテラル `"0"` や テンプレートリテラルなど文字列を代入している
3. ユニオン型や型推論が複雑に絡み合って、想定外の型が推論された結果 `string` が混入している

## なぜ多くの解説では解決しないのか

よく見かける「解決策」はただひとつ — `Number()` か `parseInt()` で変換せよ、というものだ。確かにそれで型エラーは消える。しかし**エラーが消えることと、問題が解決することは別**だ。

### ケース1: `Number()` で変換しても `NaN` が紛れ込む

フォームや URLSearchParams から取った文字列をそのまま `Number()` に渡すと、ユーザーが空文字列や非数値を入力した場合に `NaN` が返る。`NaN` は `number` 型として通るため型エラーは出なくなるが、ランタイムで `NaN` を使った計算は全て `NaN` に伝播し、バグが静かに広がる。

```typescript
const raw = searchParams.get("page"); // string | null
const page = Number(raw);             // NaN の可能性を黙殺
fetchData(page);                      // バグ
```

### ケース2: `parseInt()` の基数省略

`parseInt("08")` はES5以前の環境で8進数として解釈されることがある。現代のエンジンでは問題ないが、`parseInt(str)` と書く習慣が残るプロジェクトでは第2引数 `10` の省略がコードレビューで指摘され続ける。型エラーは消えても、別の問題が顔を出す。

### ケース3: 型アサーションで強引に黙らせる

```typescript
const id = (response.id as unknown) as number;
```

これは型エラーを**コンパイラに見せなくするだけ**で、実際のランタイム値は依然 `string` のままだ。ダウンストリームで型を信用した比較や算術をすると、数値として期待した動作をしない。型アサーションは「私はコンパイラより事情をよく知っている」という宣言であり、その根拠がない場合は最悪の選択になる。

### ケース4: 入力型を変えずに出力型だけ変える

コンポーネントの `props` で `value: number` と宣言し、親から `value="42"` と文字列リテラルで渡している場合、HTMLのデフォルト動作に引きずられた設計になっている。この場合、変換コードを末端に置いても、真の問題は「インターフェース設計が現実のデータフローと一致していない」ことだ。

## 完全な再現と修正コード

### 最小再現コード

```typescript
// reproduce.ts
type Config = {
  retryCount: number;
  timeoutMs: number;
};

function initClient(config: Config): void {
  console.log(`retry: ${config.retryCount}, timeout: ${config.timeoutMs}`);
}

// 環境変数から取得した想定（全て string）
const env = {
  RETRY_COUNT: "3",
  TIMEOUT_MS: "5000",
};

// TS2322: Type 'string' is not assignable to type 'number'.
initClient({
  retryCount: env.RETRY_COUNT,
  timeoutMs: env.TIMEOUT_MS,
});
```

このコードをコンパイルすると `TS2322` が2箇所で出る。

### 修正コード（そのままコピペで動く）

```typescript
// fixed.ts
type Config = {
  retryCount: number;
  timeoutMs: number;
};

function initClient(config: Config): void {
  console.log(`retry: ${config.retryCount}, timeout: ${config.timeoutMs}`);
}

/**
 * 環境変数を number に安全に変換する。
 * 変換できない場合は defaultValue を返す。
 */
function envToNumber(value: string | undefined, defaultValue: number): number {
  if (value === undefined || value.trim() === "") return defaultValue;
  const parsed = Number(value);
  if (!Number.isFinite(parsed)) {
    throw new RangeError(`Invalid numeric env value: "${value}"`);
  }
  return parsed;
}

const env = {
  RETRY_COUNT: "3",
  TIMEOUT_MS: "5000",
};

initClient({
  retryCount: envToNumber(env.RETRY_COUNT, 3),
  timeoutMs: envToNumber(env.TIMEOUT_MS, 5000),
});
```

**ポイント:**

- `Number.isFinite()` で `NaN` と `Infinity` を両方排除している
- 変換ロジックを関数に隔離することで、型境界（外部データが内部型に入ってくる箇所）が1ヶ所に集まる
- デフォルト値を必ず要求することで、`undefined` の場合の挙動を明示的に定義している

### React コンポーネントでの典型例

```typescript
// 問題コード
type SliderProps = {
  value: number;
  onChange: (v: number) => void;
};

// input の onChange は string を返す
<input
  type="range"
  value={someNumberState}
  onChange={(e) => onChange(e.target.value)} // TS2322
/>;

// 修正コード
<input
  type="range"
  value={someNumberState}
  onChange={(e) => {
    const parsed = Number(e.target.value);
    if (Number.isFinite(parsed)) onChange(parsed);
  }}
/>;
```

## 本番投入前チェックリスト

- [ ] 変換箇所で `Number.isFinite()` または `Number.isInteger()` による検証を行い、`NaN` / `Infinity` をガードしているか
- [ ] 外部データ（APIレスポンス・URLパラメータ・フォーム値・環境変数）が `number` 型の内部表現に変換される境界が、コードベース内で1ヶ所に集約されているか（分散していると修正漏れが起きる）
- [ ] `as number` や `as unknown as number` 型アサーションを使っている場合、その根拠がコメントに明記されているか。また実際のランタイム値を `typeof` や `console.log` で確認したか
- [ ] `parseInt()` を使う場合、必ず第2引数に基数 `10` を渡しているか（`parseInt(str, 10)`）
- [ ] 変換後の値を使う全ての算術演算・比較演算で、`NaN` が紛れ込んだ場合の挙動をテストケースでカバーしているか（`NaN === NaN` は `false`、`NaN > 0` は `false` など、`NaN` は比較を静かに破壊する）
- [ ] zod・valibot などのランタイムバリデーションライブラリを使っている場合、スキーマで `.number()` と明示的に定義し、コンパイル時型とランタイム型が一致しているか確認したか
- [ ] TypeScriptの `strict` モードが有効か。`strict: false` 環境では `TS2322` が出ない設定もあり、型エラーが見えない状態で開発が進んでいる可能性がある

## 関連する2つのエラーへの応用

### TS2345: Argument of type 'string' is not assignable to parameter of type 'number'

`TS2322` が変数や型への**代入**で起きるのに対し、`TS2345` は**関数の引数**に型不一致の値を渡したときに起きる。見た目が違うだけで原因の本質は同じ — `string` と `number` の型境界が適切に処理されていない。

```typescript
function square(n: number): number {
  return n * n;
}

const input = "5"; // string
square(input); // TS2345
```

本記事の考え方をそのまま転用できる。`square` を呼ぶ前に `envToNumber` のような変換・検証関数を挟み、型境界を一箇所に集約する。`as number` で黙らせてはいけない点も同様だ。

### TS2769: No overload matches this call（型推論の失敗による関連エラー）

ライブラリの関数がオーバーロードを持っている場合、引数の型が全オーバーロードと一致しないと `TS2769` が出る。このエラーは一見して原因がわかりにくいが、展開されたエラーメッセージの中に `Type 'string' is not assignable to type 'number'` が含まれていることが多い。

```typescript
// 例: D3.jsのscaleLinear
import { scaleLinear } from "d3-scale";

const scale = scaleLinear().domain(["0", "100"]); // TS2769
// domain() は number[] を期待するが string[] を渡している
```

対処の考え方は本記事と同じ — エラーメッセージを丁寧に読んで「どこで string が混入したか」を追跡し、型境界で変換する。`TS2769` は表面上のエラーであり、実際の根本原因は `TS2322` や `TS2345` と同じく型境界の未処理だ。ライブラリの型定義をホバーで確認し、期待される型に対して変換関数を噛ませるという手順は共通して使える。
