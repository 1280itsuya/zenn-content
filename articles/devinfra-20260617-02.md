## 結論：月数百〜千円台のVPSで十分、選ぶ基準は「SSH接続の安定性」「時間課金の有無」「cron制限がないか」の3点

個人botをクラウドで常時稼働させる場合、AWSやGCPのような大手クラウドは設定の複雑さとコスト管理のオーバーヘッドが高い。ConoHa VPS、Xserver VPS、さくらのVPS、お名前.com VPSのような国内VPSは、月額固定かつroot権限付きで扱いやすく、個人開発の定常タスクには最適な選択肢になる。

---

## 選定基準の根拠

### 1. cron・プロセスの制限がないこと

共用レンタルサーバー（ロリポップ、さくらのレンタルサーバなど）はcronの実行間隔に制限がある場合があり、長時間プロセスを強制終了することもある。VPSであれば systemd timer や supervisor を使った常駐プロセスを自由に立てられる。botには必ずVPSか専用サーバーを使うこと。

### 2. 時間課金かどうか

開発・検証フェーズで「少し試して停止」したい場合、時間課金があると無駄が出ない。ConoHa VPSは時間単位の課金プランがあり、月額上限で打ち止めになる設計を取っている（料金は変動するため公式サイトで要確認）。固定費を最小化したい段階では重宝する。

### 3. データセンターの国内ロケーション

APIコールのレイテンシは国内サーバーなら東京・大阪のDCから数ms以内に収まることが多い。特に取引系・スクレイピング系のbotは応答時間が直接成功率に影響するため、ロケーション選択は重要だ。

## 各サービスの特徴（料金は変動するため必ず公式で最新を確認）

### ConoHa VPS
GMOグループの運営。時間課金と月額課金を切り替えられる点が特徴。コントロールパネルからOSイメージの再構築が素早くできるため、環境を使い捨てしたい自動化用途に向いている。最小構成プランで例として月額数百円台〜の帯域が設定されることが多いが、キャンペーン価格と通常価格の差が大きいため契約前に要確認。

### Xserver VPS
NVMe SSDを採用しているため、ディスクI/Oが多いアプリ（SQLiteローカルDB持ちのbot等）では体感が良い。Xserverブランドのサポートへの安心感を求めるユーザー向け。こちらも料金は公式キャンペーンで変動する。

### さくらのVPS
国内VPSの老舗。プランのラインナップが広く、長期利用で安定した実績がある。管理画面はやや古いが、SSH接続の安定性で問題が起きた報告はほとんど見かけない。長期契約割引が存在し、使い続けるほど割安になる構造。

### お名前.com VPS
[ドメイン](https://px.a8.net/svt/ejp?a8mat=4B3XB4+4VMW8I+50+2HU3GX)取得と同じアカウントで管理できるため、DNS設定とサーバー設定を1か所で完結させたい場合に便利。キャンペーン時の初期費用割引が大きいが、更新時の通常料金に注意。

## 実際の初期設定手順（共通）

```bash
# 1. SSHキー認証に切り替える（パスワード認証は無効化）
ssh-keygen -t ed25519 -C "bot-server"
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@<サーバーIP>

# 2. /etc/ssh/sshd_config を編集
PasswordAuthentication no
PermitRootLogin no

# 3. ufw でポートを絞る
ufw allow 22/tcp
ufw allow 443/tcp
ufw enable
```

botは外部APIをポーリングするだけの場合、インバウンドポートは22番（SSH）しか開ける必要がない。不要ポートを閉じておくだけでブルートフォース系の攻撃ログが激減する。

### systemd でbot常駐化

```ini
# /etc/systemd/system/mybot.service
[Unit]
Description=My Automation Bot
After=network.target

[Service]
User=botuser
WorkingDirectory=/home/botuser/app
ExecStart=/home/botuser/app/.venv/bin/python main.py
Restart=on-failure
RestartSec=30

[Install]
WantedBy=multi-user.target
```

```bash
systemctl enable mybot
systemctl start mybot
```

`Restart=on-failure` と `RestartSec=30` を入れておくと、APIレートリミットや一時的なネットワーク断でクラッシュしても自動復帰する。

## よくある失敗パターン

- **共用サーバーにbotを置く**：cron制限・強制終了・利用規約違反のトリプルリスク
- **rootで常時稼働させる**：万が一の侵害時の被害が最大化する
- **swap無し最小RAMプラン**：メモリリークしやすい[Python](https://www.amazon.co.jp/s?k=Python%20%E5%85%A5%E9%96%80%20%E6%9C%AC&tag=1280itsuya22-22)プロセスが深夜にOOMキルされる。最低でも512MB swap を `fallocate` で追加しておく

```bash
fallocate -l 512M /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile swap swap defaults 0 0' >> /etc/fstab
```
